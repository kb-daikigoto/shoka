AWS Lambda 入門
サーバーを持たずに、コードを動かす

Lambda は、サーバーの管理なしにコードを実行できるコンピューティングサービスです。イベントに反応して起動し、需要に応じて自動でスケールし、使った分だけ課金。あなたが責任を持つのは「コードだけ」で、サーバーや OS、容量設計、スケーリングは AWS が引き受けます。

種別 サーバーレス 起動 イベント駆動 最大実行 15 分 課金 リクエスト＋GB秒
01

一言でいうと

従来は「サーバーを用意し、OS を保守し、台数を見積もり、アクセス増減に合わせてスケールさせ、ログ基盤を整える」必要がありました。Lambda では、そうした計算リソースの管理を AWS がすべて肩代わりします。あなたは関数（コード）を置くだけ。呼ばれたら動き、終われば消え、待機中は料金がかかりません。

用途は広く、ファイル処理（S3 にアップされたら自動処理）、定期タスク（スケジュール実行）、ストリーム処理、Web/モバイルのバックエンド、IoT のバックエンドなどに使われます。

向いている形

「何かが起きたら、短い処理をさっと実行したい」というイベント駆動・断片的な処理に最適。常時起動の重いサーバーより、こうした“反応する小さな関数”の集まりとしてアプリを組み立てるのが Lambda 流です。

02

サーバーレスの考え方

「サーバーレス」はサーバーが無いという意味ではなく、あなたがサーバーを意識しなくてよいという意味です。高可用なインフラ上で AWS が実行し、サーバー・OS の保守、容量プロビジョニング、自動スケーリング、ロギングまで管理します。

Lambda はイベント駆動型なので、従来の「ずっと起動して待ち受ける Web アプリ」とは発想が違います。イベントが来て初めてコードが走り、処理して、終わる。この粒度の違いが、後で見るスケーリングや課金の仕組みにつながります。

03

プログラミングモデル（4 つの登場人物）

PART 01

関数 (Function)

Lambda アプリの基本単位。あなたのコードをこの単位で書き、まとめる。

PART 02

トリガー / イベント源

関数を起動する出来事。S3・API Gateway・SQS・EventBridge などが、JSON 形式のイベントデータを渡してくる。

PART 03

実行ロール (Execution Role)

関数がどの AWS サービスを触れるかを定める権限。最小権限で渡すのが原則。

PART 04

ランタイム

Node.js・Python など言語ごとの実行基盤。あなたのコードを実行環境内で動かす。

関数の中身は「イベント（JSON）を受け取り、処理して、結果を返す」だけ。たとえば Python ならこんな形です。

def handler(event, context):       # event = トリガが渡す JSON
    name = event.get("name", "world")
    return {
        "statusCode": 200,
        "body": f"Hello, {name}!",
    }
handler が入口

event にはトリガ元の情報が JSON で入り、context には実行に関するメタ情報（残り時間など）が入ります。あなたが書くのは「受け取って・処理して・返す」この関数本体だけです。

04

実行のしくみ（コールド／ウォーム）

イベントが届くと、Lambda サービスは実行環境を用意してあなたのコードを動かします。実行環境はランタイム・レイヤ・あなたの関数を内包した、隔離された使い捨ての箱です。

環境を新たに用意する初回は、その準備時間ぶん遅くなります。これがコールドスタート。一度使った環境はしばらく再利用され、2 回目以降は速い（ウォーム）。レイテンシに敏感な用途では、この差を意識します。

05

スケーリングと同時実行

同時に複数のイベントが来ると、Lambda は実行環境を必要なだけ複製して並行処理します。同時に動く数を「同時実行数（concurrency）」と呼び、ここを制御する仕組みがあります。

  • 予約済み同時実行（Reserved）：ある関数に同時実行枠を確保し、他関数に食いつぶされないようにする（上限の固定にも使える）。
  • プロビジョンド同時実行（Provisioned）：あらかじめ温めた環境を確保し、コールドスタートを避ける。
  • SnapStart：初期化済みのスナップショットから起動し、コールドスタートを大幅短縮（条件により 1 秒未満も）。多くの場合コード変更なしで効く。
スケールの直感

「1 リクエスト＝1 つの実行環境が処理」が基本。100 件同時に来れば最大 100 個の環境が並ぶイメージです。だからこそ、後段の DB などが同時接続に耐えられるか、という設計が重要になります。

06

ランタイムとパッケージング

Node.js・Python・Java・Go・Ruby・.NET などの言語ランタイムが用意され、独自ランタイムも作れます。コードの配り方・再利用の仕方にも選択肢があります。

仕組み何のため
レイヤ (Layers)共通ライブラリや依存を複数関数で共有し、コードを軽く保つ
コンテナイメージ複雑な依存をコンテナ流儀でパッケージ（大きめのイメージも可）
拡張機能 (Extensions)監視・セキュリティなどのツールを関数に外付け
環境変数 / バージョンコード変更なしに挙動を変える／安全に新機能を試す
07

制約とステートレス性（ここが設計の肝）

Lambda は短命・ステートレスが前提です。主な制約を押さえましょう。

  • 最大実行時間 15 分：これを超える処理は 1 回の実行では完了できません。
  • メモリ 128MB〜10,240MB：メモリを増やすと比例して CPU 性能も上がります。
  • 一時ストレージ /tmp は最大 10GB：ただし実行環境が消えれば中身も消える、揮発的な領域。
  • ステートレス：実行間で状態は保持されない前提。状態は外部（DynamoDB・S3 など）へ。

これらを補う機能も揃っています。長時間・多段のワークフローには Durable Functions（ステートフルに最大 1 年実行可）。外部から直接叩くなら Function URLs。社内リソースへ繋ぐなら VPC 接続、永続共有ストレージが要るなら EFS 連携。大きな応答を逐次返す レスポンスストリーミングもあります。

注意

同期呼び出しのペイロード上限など、細かな制限は他にもあり時期で変わります。長時間処理・大容量・常時接続が必要なら、Lambda 以外（ECS/Fargate 等）が適することもあります。要件に合わせて選びましょう。

08

料金の考え方

課金は主に 2 軸。リクエスト数と、実行時間 × 割り当てメモリ（GB-秒）です。長く動くほど・メモリを多く割り当てるほど高くなり、待機している間（呼ばれていない間）は無料。これがサーバー常時起動との大きな違いで、アクセスにムラがある処理ほどコスト効率が良くなります。

注意

単価・無料枠・メモリと CPU の関係は時期で変わります。見積もりは 公式の料金ページで最新を確認してください。

一言で

Lambda は「イベントが来たらコードが走り、勝手にスケールし、使った分だけ払う」サーバーレス計算。短い処理を、安く・伸縮自在に動かすのが得意です。
