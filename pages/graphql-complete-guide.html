GraphQL 完全学習ガイド

「API のためのクエリ言語」である GraphQL を、基礎の考え方から、スキーマ設計・実行のしくみ・ページネーション・型安全な設計といった中級トピックまで、これ一本で学べるようにまとめました。公式の学習内容に、実務で効くスキーマ設計の指針を加えています。

対象 初学〜中級 前提 HTTP / JSON の基礎 読了 約 60 分 範囲 クエリ〜設計指針

Part 1 — 基礎

01

GraphQL とは

GraphQL は API のためのクエリ言語であり、同時に、あなたが定義した型システムを使ってクエリを実行するサーバー側のランタイムでもあります。特定の DB やストレージに縛られず、あなたの既存のコードとデータに後ろから支えられて動きます。

GraphQL サービスは、型とそのフィールドを定義し、各型の各フィールドに対する関数（リゾルバ）を用意することで作ります。たとえば「ログイン中のユーザー（me）と、その名前」を返すサービスはこう書けます。

スキーマ（型の定義）

type Query {
  me: User
}

type User {
  id: ID
  name: String
}

各フィールドの関数（リゾルバ）

function Query_me(request) {
  return request.auth.user
}
function User_name(user) {
  return user.getName()
}

サービスが起動したら（多くは Web の 1 つの URL で待ち受けます）、GraphQL クエリを受け取って検証して実行します。まず「定義された型とフィールドだけを参照しているか」を検査し、問題なければ関数を呼んで結果を組み立てます。

クエリ

{
  me {
    name
  }
}

結果

{
  "data": {
    "me": {
      "name": "Luke Skywalker"
    }
  }
}
最重要の直感

レスポンスの形は、あなたが書いたクエリの形とそっくり同じになる。欲しいフィールドを書けば、その形のまま返ってくる。これが GraphQL の全ての出発点です。

02

REST との違い

REST では URL（エンドポイント）ごとに、サーバーが決めた形のデータが返ります。すると 2 つの困りごとが起きがちです。過剰取得（over-fetching）＝要らない項目までついてくる。過少取得（under-fetching）＝足りなくて何度も叩く（N 回の往復）。

GraphQL は単一のエンドポイントに対し、クライアントが「欲しい形」を 1 回のクエリで指定します。必要なものを、必要なだけ、1 往復で。

誤解しやすい点

GraphQL は REST の上位互換でも、万能の置き換えでもありません。関係が深いデータを 1 回で柔軟に取りたい用途で輝きます。単純なファイル配信やキャッシュ最優先の用途では REST が素直なこともあります。

03

クエリの基本 — フィールドとネスト

クエリは「欲しいフィールドを書き並べる」だけ。フィールドはネストでき、オブジェクトの中の項目をさらに掘れます。返ってくる JSON は、そのネスト構造をそのまま映します。

クエリ

{
  hero {
    name
    friends {
      name
    }
  }
}

結果

{
  "data": { "hero": {
    "name": "R2-D2",
    "friends": [
      { "name": "Luke" },
      { "name": "Leia" }
    ]
  }}
}

name は文字列を返すフィールド、friends はリストを返すフィールド。クエリの形＝レスポンスの形なので、何が返るか書いた時点で分かります。これが REST のドキュメントとの大きな違いです。

04

引数とエイリアス

各フィールドには引数を渡せます。REST のようにクエリ文字列を一本に詰め込むのではなく、フィールドごとに引数を持てるのが特徴です。

引数つきクエリ

{
  human(id: "1000") {
    name
    height(unit: FOOT)   # 入れ子のフィールドにも引数
  }
}

同じフィールドを違う引数で 2 回呼びたいときはエイリアスで名前を付け替えます。そのままだと JSON のキーが衝突するためです。

エイリアス

{
  empireHero: hero(episode: EMPIRE) { name }
  jediHero:   hero(episode: JEDI)   { name }
}

結果は empireHero / jediHero という別々のキーで返ります。

05

変数とディレクティブ

引数の値をクエリ本文に直書きすると再利用しづらいので、変数に切り出します。クエリに名前と変数定義を付け、値は別の JSON で渡します。

クエリ（変数つき）

query HeroName($episode: Episode) {
  hero(episode: $episode) {
    name
  }
}

変数

{
  "episode": "JEDI"
}

ディレクティブを使うと、変数の値に応じてフィールドを動的に出し入れできます。@include(if:) と @skip(if:) が基本です。

ディレクティブ

query Hero($withFriends: Boolean!) {
  hero {
    name
    friends @include(if: $withFriends) {  # true のときだけ取得
      name
    }
  }
}
06

フラグメント

同じフィールドの組を何度も書くならフラグメントで部品化します。「どの型に対するフィールド集合か」を on で示し、...名前 で展開します。

フラグメント定義と利用

fragment heroFields on Character {
  name
  appearsIn
}

{
  empireHero: hero(episode: EMPIRE) { ...heroFields }
  jediHero:   hero(episode: JEDI)   { ...heroFields }
}
中級メモ

抽象型（インターフェース／ユニオン）に対してはインラインフラグメント ... on Droid { ... } で「この具体型のときだけ取りたいフィールド」を書きます。後述の抽象型と合わせて使います。

07

ミューテーション（書き込み）

データの取得は query、変更（作成・更新・削除）は mutation で行います。書き方はクエリと同じで、頭に mutation を付けるだけ。変更後の状態をそのまま問い合わせて返せるのが便利な点です。

ミューテーション

mutation CreateReview($ep: Episode!, $review: ReviewInput!) {
  createReview(episode: $ep, review: $review) {
    stars        # 変更結果をそのまま取得できる
    commentary
  }
}
クエリとの違い

クエリのトップレベルのフィールドは並行に解決され得ますが、ミューテーションのトップレベルは記述順に直列で実行されます。「在庫を減らす→注文を作る」のような順序依存の操作を 1 リクエストにまとめても安全です。

08

サブスクリプション（概要）

3 つ目のオペレーションが subscription。サーバー側の変化をリアルタイムで push し続けてもらう仕組みで、多くは WebSocket 上で動きます。チャットの新着、通知、ライブな数値などに使います。

サブスクリプション

subscription OnNewMessage($roomId: ID!) {
  messageAdded(roomId: $roomId) {
    id
    body
    author { name }
  }
}

クエリ・ミューテーションが「1 リクエスト＝1 レスポンス」なのに対し、サブスクリプションは「1 リクエスト＝イベントが起きるたびに繰り返しレスポンス」。まずは「そういう 3 つ目がある」と押さえれば十分です。

Part 2 — スキーマと型システム

09

スキーマと型システム

GraphQL の心臓はスキーマです。SDL（Schema Definition Language）で型を宣言します。土台になるのが スカラー型（これ以上分解できない値）。組み込みは Int / Float / String / Boolean / ID の 5 つです。

オブジェクト型はフィールドの集まり。各フィールドは型を持ち、引数やデフォルト値も持てます。

オブジェクト型と引数・デフォルト値

type Starship {
  id: ID!
  name: String!
  length(unit: LengthUnit = METER): Float
}

リストと Non-Null

2 つの修飾子が型の意味を決めます。! は Non-Null（null を返さない＝必ず値がある）、[ ] は リスト。組み合わせで意味が変わります。

記法意味
String文字列、または null
String!必ず文字列（null 不可）
[String!]リスト自体は null 可。要素は null 不可
[String!]!リストも要素も null 不可
設計のコツ

「絶対にある」と言い切れるものは ! を付けると、クライアントが null チェックを省け、契約が強くなります。逆に将来 null になり得るなら ! を外しておくと、後で互換を壊さずに済みます。

10

Enum・インターフェース・ユニオン・入力型

Enum（列挙型）

取り得る値が決まっているなら String ではなく enum に。タイプミスを型で防げます。

enum Episode { NEWHOPE EMPIRE JEDI }

インターフェース（共通フィールドの契約）

複数の型が共通フィールドを持つなら interface を定義し、各型が implements します。「これらは同じ顔（共通フィールド）を持つ」という抽象型です。

interface Character {
  id: ID!
  name: String!
  friends: [Character]
}

type Human implements Character {
  id: ID!
  name: String!
  friends: [Character]
  starships: [Starship]   # Human 固有のフィールド
}

ユニオン（無関係な型の束）

共通フィールドが無くても「どれかが返る」を表すのが union。検索結果などに。

union SearchResult = Human | Droid | Starship

抽象型を取得するときは、具体型ごとのフィールドをインラインフラグメントで指定します。

{
  search(text: "an") {
    ... on Human    { name height }
    ... on Starship { name length }
  }
}

入力型（引数用のオブジェクト）

ミューテーションなどに複雑な引数を渡すときは input を使います。出力用のオブジェクト型とは別物として扱うのが GraphQL の流儀です。

input ReviewInput {
  stars: Int!
  commentary: String
}

Part 3 — 動作のしくみ

11

リゾルバと実行

スキーマは「形」、リゾルバは「中身をどう取ってくるか」。フィールド 1 つにつき 1 つの関数が対応し、その値を計算して返します。多くの実装でリゾルバは 4 つの引数を受け取ります。

リゾルバの形

const resolvers = {
  Query: {
    // parent, args, context, info
    human(parent, args, context) {
      return context.db.getHuman(args.id)
    }
  },
  Human: {
    starships(human, args, context) {
      return context.db.getStarships(human.starshipIds)
    }
  }
}
  • parent：親フィールドのリゾルバが返した値（Human.starships なら、その human）。
  • args：そのフィールドに渡された引数。
  • context：リクエスト全体で共有する箱（DB 接続、ログインユーザーなど）。
  • info：実行中のクエリに関するメタ情報。

実行はクエリを上から下へたどり、フィールドごとにリゾルバを呼ぶだけ。親が返した値が子の parent に渡り、葉まで降りて値が揃うと、クエリと同じ形の JSON に組み上がります。フィールドが構造体のプロパティをそのまま返すだけのときは、書かなくても動くデフォルトリゾルバが使われます。

12

検証とイントロスペクション

GraphQL は実行前にクエリをスキーマと突き合わせて検証します。存在しないフィールド、型の合わない引数、必須変数の欠落などは、実行されずにエラーになります。型システムのおかげで「壊れたクエリ」を走らせずに弾けるわけです。

さらに GraphQL はイントロスペクション＝「スキーマ自身に問い合わせる」機能を標準で持ちます。これが GraphiQL や各種クライアントの自動補完・ドキュメント生成を支えています。

スキーマに問い合わせる

{
  __type(name: "Droid") {
    name
    fields {
      name
      type { name kind }
    }
  }
}
中級メモ

本番ではイントロスペクションを無効化して攻撃面を減らす運用もあります（スキーマを外部に晒したくない場合）。開発では有効、公開 API では要検討、と覚えておきましょう。

Part 4 — 中級・設計と実践

13

グラフで考える

GraphQL の「Graph」は、データをグラフ（ノードとエッジ）として捉える発想から来ています。User が posts を持ち、Post が author を持つ……というように、型同士がフィールドで結ばれてグラフを成します。

設計のコツは、エンドポイントではなく「ビジネス上のオブジェクトと、その関係」からスキーマを描くこと。「画面 A 用の API」ではなく「ドメインの型と関係」を定義すれば、新しい画面が来てもクエリを変えるだけで対応でき、スキーマを作り直さずに済みます。

考え方の転換

REST は「どんな URL を生やすか」を考えますが、GraphQL は「どんな型が、どう繋がっているか」を考えます。グラフを正しく描ければ、クライアントは欲しい経路を自由にたどれます。

14

ページネーションと Relay 接続

一覧を返すとき、全件返すのは非現実的。ページネーションが要ります。素朴には offset / limit ですが、件数が動くとズレる弱点があります。そこで業界標準になっているのが、Meta の Relay 仕様の「接続（Connection）」パターンです。

「カーソルベース」で、edges（各要素＝node＋その位置を示す cursor）と、ページ状態を表す pageInfo を返します。

Relay スタイルの接続

type Query {
  users(first: Int, after: String): UserConnection!
}
type UserConnection {
  edges: [UserEdge!]!
  pageInfo: PageInfo!
}
type UserEdge {
  cursor: String!
  node: User!
}
type PageInfo {
  hasNextPage: Boolean!
  endCursor: String
}

「次のページは after: endCursor で first: 20」と辿ります。さらに Relay 仕様ではグローバルに一意な ID（Global Object Identification）と、ID から再取得する node フィールドを定義します。

グローバル ID と node 再取得

interface Node { id: ID! }

type User implements Node {
  id: ID!        # 全型で一意な ID
  name: String!
}

type Query {
  node(id: ID!): Node   # どの型でも ID から引ける
}
なぜ Relay 仕様に従うのか

Relay 接続は主要な GraphQL ライブラリがサポートする事実上の標準で、クライアントや CDN のキャッシュ管理がこの形を前提にしています。独自のページネーションを発明する前に、まずこの仕様に則るのが安全です。グローバル ID は、後述のクライアントキャッシュの要にもなります。

15

型安全なスキーマ設計

中級の核心。原則は「曖昧な型を避け、型で意味を表す」こと。String を多用せず、決まった値は enum、特別な値は scalar、種類で構造が変わるものは interface で型自体を分ける。

「種別 enum ＋ 空フィールド」を避け、型を分ける

たとえばチャットルームに「マイチャット・ダイレクト・グループ」があり、種類で取れるデータが違うとします。1 つの型に type: enum を持たせ、使わないフィールドを null にするのは悪手。代わりに interface を実装して型を分けます。

✗ 避けたい設計

type ChatRoom {
  id: ID!
  type: RoomType!
  members: [User!]   # Group以外ではnull?
  peer: User          # Direct以外ではnull?
}

○ 型で分ける

interface ChatRoom {
  id: ID!
  name: String!
}
type GroupChat implements ChatRoom {
  id: ID! name: String!
  members: [User!]!
}
type DirectChat implements ChatRoom {
  id: ID! name: String!
  peer: User!
}

こうすると「Group には必ず members、Direct には必ず peer」を型で保証でき、クライアントは ... on GroupChat で安全に取得できます。

ミューテーションには専用のエラー型を

失敗を「null ＋ 後述の errors 配列」で表すより、結果をユニオンで型として表現すると、クライアントが成功・失敗を型で分岐できます。

type CreateTodoSuccess { todo: Todo! }
type ValidationError   { field: String! message: String! }

union CreateTodoResult = CreateTodoSuccess | ValidationError

type Mutation {
  createTodo(input: CreateTodoInput!): CreateTodoResult!
}
実務の指針

迷ったらGitHub の公開スキーマを参照するのが近道です。あなたがやりたい“独自の工夫”は、たいてい GitHub の優れたエンジニアが検討の末に採用しなかった方法かもしれません。やるなら、なぜ採用されていないのかを推察し、利点と欠点を評価した上で決めましょう。

16

運用の勘所 — エラー・N+1・キャッシュ・認可

エラー

GraphQL の応答は data と errors が同居できます。一部のフィールドが失敗しても、成功した部分は data に入り、失敗は errors 配列に積まれます。

{
  "data": { "hero": { "name": null } },
  "errors": [{ "message": "...", "path": ["hero", "name"] }]
}

「業務上のエラー（バリデーション失敗など）」は、前章のようにスキーマの型として表す方がクライアントに優しいことが多いです。

N+1 問題と DataLoader

リストの各要素で関連データをリゾルバが取りに行くと、一覧 1 回＋要素ごとに N 回のクエリが飛ぶ「N+1 問題」が起きます。対策は DataLoader：同じ実行サイクル内の取得要求を溜め、まとめて 1 回で解決します（DB 側は WHERE id IN (...) でバッチ取得）。

キャッシュ

単一エンドポイント・POST が基本なので、HTTP レベルの URL キャッシュは効きにくい。代わりに クライアント側の正規化キャッシュ（Apollo / Relay）が主役です。各オブジェクトをグローバルに一意な ID でキャッシュに格納するため、前章のグローバル ID 設計がここで効きます。

認可（Authorization）

「誰が何を見てよいか」は、GraphQL 層ではなくその下のビジネスロジック層で判断するのが定石です。リゾルバから呼ぶサービス側で権限を確認し、GraphQL はあくまで「形」を扱う、と役割を分けます。

17

スキーマの進化とベストプラクティス

GraphQL はバージョニング（/v1, /v2）を避けるのが思想です。型システムと「クライアントが欲しいフィールドだけ取る」性質のおかげで、フィールドを足しても既存クライアントは壊れません。古いフィールドは消さずに @deprecated で非推奨にし、移行を促します。

type User {
  fullName: String
  name: String @deprecated(reason: "Use fullName")
}

まとめ：設計の 4 指針

  1. GitHub のスキーマに則る：迷ったら実績ある公開スキーマを手本にする。
  2. Relay 仕様に則る：接続（ページネーション）とグローバル ID。クライアント／CDN のキャッシュ前提に合う。
  3. 型安全に扱えるようにする：String 乱用を避け、enum / scalar / interface / 専用エラー型で意味を型に込める。
  4. Relay と Apollo の 2 クライアントを確認する：この 2 大クライアントに対応できれば、他のクライアントでも問題なく扱える。挙動に迷ったら両者のドキュメントと実挙動を確認。
なぜこの 4 つか

GraphQL は仕様の自由度が高いぶん、「我流」で設計するとクライアントのキャッシュやツールが噛み合わなくなりがち。実績ある型（GitHub・Relay）と 2 大クライアントに合わせることが、結局いちばん遠回りしない道になります。

初級：クエリ／引数／変数／フラグメント／ミューテーションで「欲しい形を取る・変える」。型：スカラー・オブジェクト・enum・interface・union・input でスキーマを書く。しくみ：検証→リゾルバ実行→同形 JSON。中級：グラフで設計し、Relay 接続とグローバル ID、型安全な設計、N+1・キャッシュ・認可・スキーマ進化まで。ここまで来れば、実務のスキーマ設計とレビューに入っていけます。

本記事は GraphQL 公式「Introduction to GraphQL」の学習内容と、スキーマ設計の実践指針（GitHub スキーマ準拠・Relay 仕様・型安全・Relay/Apollo クライアント確認）をもとに、初級〜中級を一冊にまとめたものです。仕様や各ライブラリの作法は更新され得るため、実装時は Relay / Apollo の公式ドキュメントも参照してください。
