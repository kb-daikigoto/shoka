実用 Go言語
現場で効く「Goらしい」書き方

文法を覚えた次に待っているのは、「で、現場ではどう書くのが正解？」という壁。本書は、定義型・小さなインタフェース・値としてのエラー・通信による並行処理といったGoらしいイディオムを、システム開発の実務目線で束ねた一冊です。シンプルさを保ったまま、壊れにくく読みやすいコードに辿り着くための型を、初級の先〜中級向けに自分の言葉で再構成します。

01

一言でいうと

Go の魅力はシンプルさです。言語機能が少なく、書き方の「正解」が収束しやすい。だからこそ、本書のテーマは「言語の流儀（イディオム）に逆らわないこと」に尽きます。他言語の癖を持ち込んで複雑な抽象を作るより、Go が用意した素直な道具を素直に使うほうが、結局いちばん強い。

この記事では、その流儀を3本の柱で捉えます。型で意味を表す・小さく合成する・明示的に扱う。エラーも並行処理も後始末も、すべてこの3つの延長線上にあります。

この記事のゴール

「動くGo」から「レビューを通るGo」へ。なぜその書き方が好まれるのかを言葉で説明できるようになり、現場のコードベースに馴染ませることを目指します。

定義型とゼロ値を活かす

最初の一歩は、生の string や int を、意味を持った型に昇格させること。UserID と Email がどちらも string だと、引数の取り違えにコンパイラは気づけません。定義型（defined type）で包めば、型システムが間違いを止めてくれます。

// 単なる string ではなく「意味」を型にする
type UserID string
type Email  string

// 取り違えるとコンパイルエラー。ドキュメントにもなる
func SendInvite(id UserID, to Email) error { /* ... */ }

もう一つの要が ゼロ値（zero value）です。Go は変数を必ず初期化し、その既定値がそのまま使える状態になるよう型を設計するのが作法。var buf bytes.Buffer は初期化なしで書き込めるし、sync.Mutex もゼロ値でロックできます。「使う前に初期化メソッドを呼ばせる」設計は、できるだけ避けます。

var buf bytes.Buffer   // new も Init も不要、ゼロ値で即使える
buf.WriteString("hello")

var mu sync.Mutex      // ゼロ値がそのまま「未ロック」の有効な状態
mu.Lock()
設計のコツ

新しい型を作るとき「ゼロ値は意味を持つか？」を自問する。持つなら最高（利用者が楽になる）。持てないなら、New...() コンストラクタを用意し、不正な状態を作れないようにする。

インタフェースは小さく

Go のインタフェースは暗黙的です。「implements」と宣言する必要はなく、メソッドが揃っていれば自動的に満たす。この特性が活きるのは、インタフェースを「使う側」が、必要なメソッドだけを小さく定義するときです。標準ライブラリの io.Reader・io.Writer がメソッド1つなのは象徴的。

// 使う側が、欲しい振る舞いだけを宣言する（小さく！）
type Storer interface {
    Save(ctx context.Context, u User) error
}

// 「引数はインタフェース、戻り値は具体型」が基本形
func NewService(s Storer) *Service {
    return &Service{store: s}
}

これにより、テストでは Storer を満たすモックを差し込むだけで差し替えられます。実装側（DB アクセスの具体型）はインタフェースを知らなくてよい——依存の向きが「呼ぶ側」に閉じるのが Go 流です。継承の代わりには 埋め込み（embedding）で機能を合成します。

やりがちな失敗

「将来のために」と巨大なインタフェースを実装側に先回りで置くこと。Go では逆。インタフェースは消費される場所で、実際に必要になってから小さく作る。メソッドが3つを超えたら、分割を疑いましょう。

エラーは値として扱う

Go に例外はありません。エラーは error というただの値で、関数の戻り値として返ってくる。だから if err != nil が頻出します。冗長に見えますが、これは「失敗しうる場所が、コード上で常に見える」という安全装置です。握りつぶしも、伝播も、判断も、すべて明示的になります。

実務の肝は ラップ（wrap）。fmt.Errorf の %w で文脈を足しながら原因を保持し、呼び出し側で errors.Is / errors.As で中身を判定します。

// 文脈を足して上へ返す。原因は %w で保持される
if err := repo.Find(ctx, id); err != nil {
    return fmt.Errorf("find user %s: %w", id, err)
}

// 番兵エラーとの一致を判定（ラップを貫いて見てくれる）
var ErrNotFound = errors.New("not found")
if errors.Is(err, ErrNotFound) { /* 404 を返す等 */ }

// 具体的なエラー型を取り出す
var perr *fs.PathError
if errors.As(err, &perr) { log.Println(perr.Path) }
アンチパターン

if err != nil { return err } で素のまま投げ続けると、最後に「どこで失敗したか」が分からないログだけが残ります。境界を越えるときは文脈を足す。逆に、同じエラーを何度もラップしてログにも出すと二重報告になる。報告は一箇所、と決めておきます。

並行処理 — 通信で共有する

Go の代名詞、goroutine と channel。goroutine は go f() と書くだけで動く軽量な並行単位です。設計の合言葉は 「メモリを共有して通信するな。通信してメモリを共有せよ」。ロックで守り合うより、データを channel で受け渡すほうが、Go では素直で壊れにくくなります。

// 各仕事を goroutine で走らせ、結果は channel で集める
results := make(chan Result)
for _, job := range jobs {
    go func(j Job) {        // ループ変数は引数で渡す
        results <- process(j)
    }(job)
}
for range jobs {
    r := <-results               // 必要な数だけ受け取る
}

とはいえ、単純なカウンタ保護などは sync.Mutex のほうが素直なこともあります。複数 goroutine の完了待ちは sync.WaitGroup、エラーを集めて最初の失敗で打ち切るなら golang.org/x/sync/errgroup が定番。道具は使い分け、が実務の答えです。

goroutine リーク注意

起動した goroutine が終わる道筋を必ず用意する。受け手のいない channel 送信や、止め方のない無限ループは静かに溜まり続けます。停止の合図は次節の context で渡すのが定石です。

context で寿命を渡す

context.Context は、処理の「寿命」と「中断の合図」を関数からその先へ運ぶための仕組みです。リクエストがキャンセルされた、制限時間を超えた——そうした事情を、呼び出しの連鎖の末端まで伝えます。約束事はシンプルで、ブロックしうる関数は ctx を第1引数で受け取る。

// 3秒の締め切りを付ける。cancel は必ず defer で呼ぶ
ctx, cancel := context.WithTimeout(ctx, 3*time.Second)
defer cancel()

// I/O や DB、HTTP には ctx を必ず渡す（先頭で受ける）
func Fetch(ctx context.Context, url string) ([]byte, error) {
    req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
    // ctx がキャンセルされれば、この通信も即座に打ち切られる
}

キャンセルは下流へ伝播します。親 context を畳めば、そこからぶら下がる子の処理がまとめて止まる。これが goroutine リークを防ぎ、無駄な処理を切り上げる土台になります。context.Value で値も運べますが、濫用は禁物——本来は引数で渡すべきものをこっそり潜ませる隠れ依存になりがちです。リクエストスコープのID等に限定します。

3つの約束

① ctx は第1引数。② 構造体のフィールドに溜め込まない（その場で渡す）。③ cancel は defer で必ず呼ぶ。この3つを守るだけで、中断とリソース解放がきれいに揃います。

io と後始末（defer / Close）

Go の I/O は io.Reader / io.Writer という2つの小さなインタフェースに集約されています。ファイルもネットワークもバッファも、すべて同じ「読み出し口・書き込み口」として扱える。だから io.Copy(dst, src) 一つで、出どころを問わずデータを流せます。

f, err := os.Open(name)
if err != nil { return err }
defer f.Close()              // 後始末は defer で“その場”に書く

// Reader → Writer をそのまま流す（中身を全部メモリに載せない）
if _, err := io.Copy(dst, f); err != nil {
    return fmt.Errorf("copy %s: %w", name, err)
}

defer は、確保したリソースの解放を確保の直後に書ける仕組みです。関数を抜けるときに逆順で実行されるので、Close や Unlock の呼び忘れを防げます。「開いたらすぐ defer で閉じる」をセットで身体に入れるのが王道です。

defer の落とし穴

ループの中で defer すると、解放が関数の最後まで遅れる（＝ループ中ずっと開きっぱなし）。大量に回すなら、本体を小さな関数に切り出してその関数末尾で閉じる。また、書き込み側の Close のエラーは握りつぶさず確認します（フラッシュ漏れを検知するため）。

database/sql と net/http

標準ライブラリだけで、実用的な Web サービスは十分組めます。DB アクセスは database/sql。*sql.DB はコネクションプールそのものなので、アプリで1つ作って使い回すのが鉄則です（リクエストごとに開かない）。クエリには必ず ...Context 版を使い、プレースホルダで値を渡してSQLインジェクションを防ぎます。

// ctx 付き・プレースホルダ。文字列連結は厳禁
rows, err := db.QueryContext(ctx,
    "SELECT id, name FROM users WHERE team_id = $1", teamID)
if err != nil { return nil, err }
defer rows.Close()
for rows.Next() {
    var u User
    if err := rows.Scan(&u.ID, &u.Name); err != nil { return nil, err }
}
return users, rows.Err()      // ループ後の Err() 確認を忘れない

HTTPサーバは net/http。ハンドラは http.Handler インタフェースで合成でき、ミドルウェアは「ハンドラを受け取ってハンドラを返す関数」で素直に書けます。本番ではタイムアウトを明示した http.Server を組み立てるのが必須作法です。

srv := &http.Server{
    Addr:        ":8080",
    Handler:     mux,
    ReadTimeout:  5 * time.Second,   // 既定は無制限。必ず付ける
    WriteTimeout: 10 * time.Second,
}
if err := srv.ListenAndServe(); err != nil { /* ... */ }
要点

*sql.DB は使い回す・クエリはctx＋プレースホルダ・サーバはタイムアウト必須。この3点を外さなければ、標準ライブラリだけで堅牢な土台になります。

テーブル駆動テスト

Go の testing は標準装備で、外部フレームワークなしで十分戦えます。定番の型が テーブル駆動テスト。入力と期待値の表（スライス）を作り、ループで回す。ケースの追加が1行で済み、t.Run でサブテストにするとどのケースで落ちたかが一目で分かります。

func TestAdd(t *testing.T) {
    tests := []struct {
        name       string
        a, b, want int
    }{
        {"zero", 0, 0, 0},
        {"pos",  1, 2, 3},
        {"neg",  -2, 1, -1},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            if got := Add(tt.a, tt.b); got != tt.want {
                t.Errorf("Add(%d,%d)=%d, want %d", tt.a, tt.b, got, tt.want)
            }
        })
    }
}

差し替えたい依存は §03 の小さなインタフェースでモックに置き換えます。期待される大きな出力はゴールデンファイル（期待値をファイルに保存し -update フラグで更新）にすると、テストが読みやすくなります。go test -race での競合検出も習慣にしましょう。

テストを書きやすい設計

「テストしづらい」と感じたら、それは設計の声。副作用（時刻・乱数・I/O）をインタフェース越しに注入できる形へ寄せると、テスト容易性と本体の見通しが同時に良くなります。テストは品質の検査であり、設計のフィードバックでもあります。

パッケージとモジュール設計

依存は Goモジュール（go.mod）で管理します。go mod tidy が使う依存を整理し、go.sum がハッシュで再現性を担保。ここは迷うところが少なく、素直に従えば問題ありません。

設計で効くのはパッケージの切り方です。Go のパッケージは「公開する名前（大文字始まり）が、その境界のAPI」。だから、パッケージ名は中身を表す名詞にし、利用側で user.New() のように読めるようにします。utils や common のような意味のない受け皿は、何でも入って肥大化するので避けます。

観点避けたいGoらしい
パッケージ名utils / common / models役割を表す名詞（billing, auth）
命名user.UserService（重複）user.Service（パッケージ名と重ねない）
公開範囲とりあえず全部 export必要な物だけ大文字に
依存の向き相互参照・循環一方向。境界はインタフェースで
なぜ名前が大事か

Go では名前そのものがドキュメント。context.Context、io.Reader のように「パッケージ名＋型名」で意味が完成するよう設計すると、利用側のコードが英文のように読めます。小さく、正直に名付ける——それがいちばん効く設計判断です。

運用へつなぐ

最後に、書いた Go を届けて動かす側面。Go は単一バイナリにコンパイルされるのが最大の運用上の強みです。ランタイムを同梱しないので、配布もコンテナ化も軽い。クロスコンパイルも GOOS / GOARCH を指定するだけです。

# Linux 向けに、依存ゼロの静的バイナリを1つ吐く
$ CGO_ENABLED=0 GOOS=linux go build -o app ./cmd/server

# コンテナは「ビルド用」と「実行用」を分ける（軽く・安全に）
# 実行イメージは scratch / distroless にバイナリ1枚だけ置ける
  • 設定：環境変数やフラグで外から注入する（flag / 環境変数）。秘密はコードに埋めない。
  • ログ：標準の log/slog で構造化ログ。検索・集計しやすい形で出す。
  • graceful shutdown：シグナルを受けたら srv.Shutdown(ctx) で処理中のリクエストを捌き切ってから止める。
つながった

ここまでの全要素が運用で合流します。context が graceful shutdown を支え、小さなインタフェースが設定やログの差し替えを可能にし、エラーのラップが構造化ログで効く。Goらしさは、そのまま運用しやすさになります。

一言で

Goらしさとは、型で意味を表し・小さく合成し・明示的に扱うこと。言語のシンプルさに逆らわず、標準ライブラリと素直なイディオムに乗る。それがいちばん速く、いちばん壊れにくい道です。

用語集

定義型既存型に新しい名前と意味を与えた型（type UserID string）。取り違えを型で防ぐ。
ゼロ値変数の既定の初期値。これがそのまま使える状態になるよう設計するのが作法。
暗黙的インタフェース宣言なしで、メソッドが揃えば自動的に満たす仕組み。使う側で小さく定義する。
%w / errors.Is / Asエラーに文脈を足して包む（%w）／包みを貫いて原因を照合・取り出す。
goroutine / channel軽量な並行単位と、その間でデータを受け渡す通路。「通信で共有」する道具。
context.Context処理の寿命とキャンセルの合図を下流へ運ぶ型。ブロックする関数の第1引数。
defer関数を抜けるときに実行する後処理の予約。確保の直後に解放を書ける。
テーブル駆動テスト入力と期待値の表をループで回すテストの型。t.Run でケースを分離。
