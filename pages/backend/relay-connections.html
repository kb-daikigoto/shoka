Relay Connections と
カーソルページネーション

「一覧の続きを読む」を堅牢に表現するための GraphQL の作法 ― Connection。edges・node・cursor・pageInfo という4つの部品が、なぜその形をしているのかを理屈から噛み砕き、サーバ実装と Relay クライアントの両側まで一気に押さえます。

01

なぜ「次のページ」が難しいのか

一言でいうと

件数で位置を指す offset 方式は、データが動くとズレる・飛ぶ・重複する。Connection は「位置」を件数ではなくカーソル（しおり）で持つことで、これを避ける。

一覧をすべて一度に返すのは現実的ではありません。フォロワーが10万人いるユーザーの friends を全部返したらクライアントもサーバも壊れます。だから「最初の N 件 → 続きの N 件 …」と区切って取りに行く＝ページネーションが要ります。

素朴なやり方は offset / limit（SQL の LIMIT 20 OFFSET 40）です。しかしこれは「先頭から何件目」で位置を指すため、ページを読んでいる最中に前方の要素が増減すると破綻します。

要素が飛ぶ

1ページ目を読んだ後に先頭へ1件挿入されると、2ページ目の先頭が1件ぶん後ろにずれ、1件読み飛ばす。

要素が重複

逆に先頭が1件削除されると、すでに見た要素が2ページ目にもう一度出てくる。

深いページが遅い

OFFSET 100000 はDBが10万行を数えて捨てる。後ろのページほど線形に遅くなる。

◑ なぜカーソルなのか

offset は「順序の中の序数（何番目）」で位置を指す。序数は他の要素の増減で変わってしまう。カーソルは「その要素自身を指すしおり」なので、前後で何が起きても「この要素の次から」を安定して再開できる。

02

Connection の全体像 ― 4つの部品

一言でいうと

Connection は「一覧」を表す箱。中身は edges（縁）の配列と pageInfo（ページの状態）。各 edge は cursor（しおり）と node（実体）を持つ。

GraphQL の慣習では、コレクションをただの配列 [User] ではなく、決まった形の入れ子で返します。これが Relay Cursor Connections 仕様です。形を絵にするとこうなります。

スキーマで書くと次のとおり。friends が裸の [User] ではなく FriendsConnection を返している点に注目してください。

schema.graphql
type User {
  id: ID!
  name: String
  # 一覧は Connection を返す。引数で「窓」を指定する
  friends(first: Int, after: String, last: Int, before: String): FriendsConnection
}

type FriendsConnection {
  edges: [FriendsEdge]
  pageInfo: PageInfo!
}

type FriendsEdge {
  cursor: String!     # この要素を指すしおり
  node: User          # 実体（中身のオブジェクト）
}

type PageInfo {
  hasNextPage: Boolean!
  hasPreviousPage: Boolean!
  startCursor: String
  endCursor: String
}
● 押さえどころ

箱が二重（Connection → Edge → node）なのは冗長に見えますが、これは 「縁（edge）に情報を載せる余地」を作るための設計です（→ 05）。node が実体、edge は「その実体がこの一覧に属していること」を表す関係そのもの。

03

カーソルの正体 ― 不透明なしおり

一言でいうと

cursor は「ここまで読んだ」を表す不透明（opaque）な文字列。クライアントはそのまま受け取り、そのまま送り返すだけ。中身を解釈してはいけない。

cursor は仕様上 String です。中身は実装の自由で、よくあるのは「ソートキー＋IDを base64 エンコードしたもの」。たとえば "id:1024" を符号化して "Y3Vyc29yOjEwMjQ=" のように見えます。

◑ なぜ「不透明」にするのか

クライアントが cursor の中身（例: offset=40）に依存し始めると、サーバはページング方式を二度と変えられなくなる。base64 で「読めなく」しておくのは、中身に依存させないための心理的な壁。クライアントは「黒い箱」として往復させるだけにする。

カーソルの良いところは、各要素が自分専用のしおりを持つこと。一覧に変化が起きても「この cursor の次から」という指定は意味が変わりません。これが 01 で見た offset の弱点を消します。

◆ 実装のコツ

cursor にはソートに使ったキーを必ず含める。ID だけだと「created_at 降順」のような並びで再開位置を一意に決められない。例: (created_at, id) の複合キーを符号化しておくと、同時刻の要素があっても安定する。

04

引数 first / after / last / before

一言でいうと

first + after で前方向（古→新でなく、先頭から続きへ）、last + before で後ろ方向にページングする。「何件」＋「どのしおりの隣から」の組。

引数意味向き
first: Int先頭から N 件取る前方向（forward）
after: Stringこの cursor の次から
last: Int末尾から N 件取る後ろ方向（backward）
before: Stringこの cursor の手前まで

1ページ目 ― first だけ

query — 最初の3件
query {
  user {
    friends(first: 3) {
      edges {
        cursor
        node { id name }
      }
      pageInfo {
        endCursor
        hasNextPage
      }
    }
  }
}

レスポンスはこんな形。末尾の endCursor が「ここまで読んだ」のしおりです。

response.json
{
  "data": { "user": { "friends": {
    "edges": [
      { "cursor": "YzE=", "node": { "id": "1", "name": "Alice" } },
      { "cursor": "YzI=", "node": { "id": "2", "name": "Bob" } },
      { "cursor": "YzM=", "node": { "id": "3", "name": "Carol" } }
    ],
    "pageInfo": { "endCursor": "YzM=", "hasNextPage": true }
  } } }
}

2ページ目 ― endCursor を after に渡す

hasNextPage が true なら、もらった endCursor をそのまま after に詰めて再リクエスト。これがページネーションの心臓部です。

query — 続きの3件
friends(first: 3, after: "YzM=") {   # ← 前回の endCursor
  edges { cursor node { id name } }
  pageInfo { endCursor hasNextPage }
}
▲ 注意

first と last を同時に指定するのは仕様上推奨されない（曖昧になる）。前方向と後ろ方向はどちらか一方に決める。多くのUIは first/after の前方向だけで足りる。

05

なぜ二重の箱？ ― edge にメタを載せる

一言でいうと

「いつ友達になったか」のような関係そのものの情報は node（User自身）ではなく edge に置く。これが Connection が node を直接並べず edge を挟む理由。

もし friends: [User] だったら、「Alice といつ友達になったか」を置く場所がありません。それは Alice の属性ではなく、「私とAliceの関係」の属性だからです。edge があると、その縁の上に情報を載せられます。

schema.graphql — edge にフィールドを足す
type FriendsEdge {
  cursor: String!
  node: User
  friendedAt: DateTime    # ★ 関係の属性。node ではなく edge に置く
}
node に置くもの

その実体そのものの属性。id / name / avatar ― 誰から見ても同じ。

edge に置くもの

「この一覧に属していること」に固有の属性。friendedAt / 並び順スコア / ロール。

● 押さえどころ

cursor が edge 側にあるのも同じ理屈。cursor は「この一覧の中での位置」であって User 自身の属性ではない。同じ User が別の一覧では別の cursor を持つ。

06

pageInfo ― 端と「続きの有無」

一言でいうと

pageInfo はページの状態を1か所にまとめたもの。hasNextPage でボタンを出すか決め、endCursor で次の窓を指定する。

フィールド意味使いどころ
hasNextPage前方向にまだ要素があるか「もっと見る」を出すか
hasPreviousPage後ろ方向にまだ要素があるか「前へ」を出すか
startCursor今のページ先頭の cursorbefore に渡して前へ戻る
endCursor今のページ末尾の cursorafter に渡して次へ進む
◆ 実装のコツ

hasNextPage を正確に返すには 「N+1件取って、N+1件目があれば true」のテクニックが定番（→ 08）。総件数を数える必要はない。totalCount は仕様の必須ではなく、要るときだけ Connection に足す拡張。

▲ 注意

前方向ページング（first/after）でも hasPreviousPage は返せるが、「after が指定されていれば true」程度の素朴実装になりがち。仕様も「前方向のとき hasPreviousPage の厳密さは要求しない」とゆるく定めている。UIで本当に要るかで実装コストを決める。

07

Relay クライアント側 ― @connection と usePaginationFragment

一言でいうと

Relay は @connection ディレクティブで「これはページングする一覧」と印を付ける。あとは usePaginationFragment が cursor の往復・キャッシュ追記・追加読み込みを全部やってくれる。

ここからは「サーバが Connection 仕様に従っている」前提で、JS クライアントの Relay がどう楽をさせてくれるかです。フラグメントで一覧を宣言し、@connection を付けます。

FriendsList.tsx — フラグメント定義
const FriendsFragment = graphql`
  fragment FriendsList_user on User
  @argumentDefinitions(
    count: { type: "Int", defaultValue: 3 }
    cursor: { type: "String" }
  )
  @refetchable(queryName: "FriendsListPaginationQuery") {
    friends(first: $count, after: $cursor)
      @connection(key: "FriendsList_friends") {
      edges {
        node { id name }
      }
    }
  }
`;
  • @connection(key: ...) ― 「ここはページングする一覧。新しいページを同じリストに追記して」と Relay に伝える印。key はストア内でその一覧を識別する名前。
  • @refetchable ― このフラグメントを起点に追加クエリを自動生成させる。
  • @argumentDefinitions ― count / cursor をフラグメント引数として宣言。Relay が after に cursor を流し込む。
FriendsList.tsx — フックで描画＆追加読み込み
function FriendsList({ user }) {
  const { data, loadNext, hasNext, isLoadingNext } =
    usePaginationFragment(FriendsFragment, user);

  return (
    <div>
      {data.friends.edges.map((e) => (
        <Friend key={e.node.id} name={e.node.name} />
      ))}

      {hasNext && (
        <button
          disabled={isLoadingNext}
          onClick={() => loadNext(3)}>   // ← 次の3件
          もっと見る
        </button>
      )}
    </div>
  );
}
● 押さえどころ

loadNext(3) を呼ぶだけで Relay は「endCursor を after に詰めて再クエリ → 返ってきた edges を既存リストに追記 → 再描画」を自動でやる。cursor を自分で持ち回るコードは書かない。これが Connection 規約をクライアントが守る最大の見返り。

◆ 実装のコツ

フックは data（描画用）/ loadNext / hasNext（≒ pageInfo.hasNextPage）/ isLoadingNext（ローディング表示用）を返す。後ろ方向には loadPrevious / hasPrevious が対になっている。

08

サーバ実装の勘どころ

一言でいうと

「N+1件取得」で hasNextPage を判定し、各行のソートキーを符号化して cursor にする。Relay でなくても通用する普遍的なレシピ。

言語非依存の擬似コードで、first/after の前方向リゾルバを書くと次のようになります。

resolver — first / after の実装（擬似コード）
function friends(parent, { first, after }) {
  // 1) after があれば復号して「どこから」を決める
  const afterKey = after ? decodeCursor(after) : null;

  // 2) ★ first+1 件取る。「次があるか」を1件多く取って判定する
  const rows = db.query(`
    SELECT * FROM friends
    WHERE user_id = ?
      AND (?::cursor IS NULL OR (created_at, id) < ?)  -- after の隣から
    ORDER BY created_at DESC, id DESC
    LIMIT ?`, [parent.id, afterKey, afterKey, first + 1]);

  // 3) +1件目が取れたら「次のページあり」
  const hasNextPage = rows.length > first;
  const page = rows.slice(0, first);

  // 4) 各行を edge に。cursor はソートキーを符号化
  const edges = page.map((r) => ({
    cursor: encodeCursor({ createdAt: r.created_at, id: r.id }),
    node: r,
  }));

  return {
    edges,
    pageInfo: {
      hasNextPage,
      hasPreviousPage: afterKey != null,
      startCursor: edges[0]?.cursor ?? null,
      endCursor: edges[edges.length - 1]?.cursor ?? null,
    },
  };
}
  • N+1件取得：総件数を COUNT(*) するより安く hasNextPage がわかる。
  • キーセット方式：WHERE (created_at, id) < afterKey はインデックスが効くので、深いページでも速度が落ちない（offset の弱点を解消）。
  • cursor にソートキーを内包：ORDER BY に使った列をそのまま符号化する。並び順を変えたら cursor の中身も変える。
■ 落とし穴

ソート順が一意でないと再開位置がブレる。created_at だけで並べると同時刻の要素で順序が不定になり、ページ境界で取りこぼし／重複が起きる。必ず (created_at, id) のように最後にユニークなキー（id）を足して全順序にする。

09

よくある落とし穴

cursor を offset にする

「不透明」とうたいつつ中身が offset=40 だと、offset の弱点（飛び・重複）をそのまま引き継ぐ。キーセットで作る。

cursor を跨いで並び順を変える

1ページ目を created_at 順、2ページ目を name 順で取ると cursor の意味が崩壊。並び順は一覧の往復で固定する。

totalCount を必須だと思う

仕様の必須は edges と pageInfo だけ。総件数が要るUIのときだけ Connection に足す拡張。

node を nullable にし忘れ／しすぎ

権限などで個別 node を隠す設計なら nullable が活きる。要件に合わせて決める。

クライアントが cursor を解釈

cursor をパースして自前ロジックを組むと、サーバが方式を変えられなくなる。黒い箱として往復だけ。

@connection の key 衝突

Relay で同じ key を別フラグメントに使うとストアが混ざる。一覧ごとに一意な key を付ける。

10

チートシート

Connection の型

  • Connection = { edges: [Edge], pageInfo: PageInfo! } ― 一覧の箱
  • Edge = { cursor: String!, node: T }（＋関係の属性）
  • PageInfo = hasNextPage / hasPreviousPage / startCursor / endCursor

ページングの引数

  • 前方向：first: N, after: 前回のendCursor
  • 後ろ方向：last: N, before: 前回のstartCursor
  • 進める条件：pageInfo.hasNextPage === true

実装レシピ

  • サーバ：first+1件取得で hasNextPage 判定、(sortKey, id) をキーセットで where、cursor にソートキーを符号化
  • cursor：不透明（base64）・全順序・並び順を内包
  • Relay：@connection(key) ＋ usePaginationFragment → loadNext(N)
11

用語集

Connection
コレクションを表す型。edges と pageInfo を持つ「一覧の箱」。
Edge（縁）
一覧の1要素を包む型。cursor と node、そして「関係そのものの属性」を持つ。
node（節点）
edge が指す実体オブジェクト。User や Post など本体。
cursor（しおり）
一覧内の位置を指す不透明な文字列。after/before に渡して再開位置を示す。
pageInfo
ページの状態。続きの有無（hasNextPage 等）と端の cursor（start/endCursor）。
キーセットページネーション
WHERE (sortKey, id) < lastSeen で次を取る方式。offset と違いインデックスが効き安定する。
@connection
Relay クライアントで「ページングする一覧」を示すディレクティブ。新ページを既存リストへ追記させる。
usePaginationFragment
cursor の往復・キャッシュ追記・追加読み込みを担う Relay の React フック。
