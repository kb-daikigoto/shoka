Designing APIs people love to use

Web API: The Good Parts
使いやすく、変えやすく、壊れにくいAPI設計

APIは一度公開すると、使う人がコードを書き、簡単には変えられなくなります。だからこそ最初の設計が効く。本書のテーマは「美しいAPIは、使いやすく・変更に強く・堅牢」であること。エンドポイント設計、HTTPの作法、レスポンス構造、バージョニング、セキュリティまで。この記事では、その「良い設計の勘どころ」を、設計の理由とともに自分の言葉で再構成します。

01

一言でいうと

本書の主張はシンプルです。「APIは、人が使うUIである」。叩くのが人間か他のプログラムかが違うだけで、使いやすさが価値を決めるのは同じ。良いAPIは、ドキュメントを隅々まで読まなくても直感で叩けて、想像どおりに動きます。

「良さ」は3つの軸で測れます。これは本書の構成そのものでもあります。

軸1

使いやすい

URIやレスポンスが素直で一貫。最小の驚き（least surprise）で、推測どおりに動く。

軸2

変えやすい

公開後も後方互換を保って進化できる。バージョニングと互換性の設計。

軸3

壊れにくい

悪意や過負荷に耐える。HTTPS・レートリミット・既知の攻撃への備え。

そして全体を貫く哲学が「HTTPに逆らわない」。Web APIはHTTPの上に立つのだから、メソッド・ステータスコード・キャッシュといったHTTPが既に持つ仕組みを再発明せず、素直に活かす。これが、薄くて強い設計への近道です。

この記事のゴール

「とりあえず動くAPI」から「他人が気持ちよく使えるAPI」へ。URI1本・ステータスコード1つの選択に、理由を持って判断できるようになることを目指します。

02
Part I · 土台

良いAPIの原則

奇抜さは要りません。良いAPIはたいてい「ありきたり」です。多くのAPIが採る作法に素直に乗ることが、利用者にとっての学習コストを最小にします。本書が繰り返す原則をまとめると、こうなります。

  • 仕様・規約に逆らわない：HTTPやRESTの慣習どおりに。独自ルールは利用者の負担。
  • 一貫性：命名・構造・エラー形式をAPI全体で統一する。例外を作らない。
  • 最小の驚き：名前から動きが推測でき、推測が裏切られない。
  • self-descriptive（自己記述的）：レスポンスやステータスを見れば、何が起きたか分かる。

土台になる考え方が REST です。難しく考えず、「世界をリソース（名詞）として捉え、それをURIで指し、HTTPメソッド（動詞）で操作する」と掴めば十分。「ユーザーを取得」ではなく「ユーザーというリソースに GET する」。この視点の転換が、設計を一気に整えます。

03
Part II · 設計の実際

エンドポイントの設計

エンドポイント（URI）は、APIの顔です。短く・覚えやすく・規則的に。本書の指針は明快です。リソースは名詞・複数形、階層は / で表し、操作はURIに書かずHTTPメソッドに任せる。

メソッド意味例性質
GET取得GET /users/123安全・冪等
POST新規作成POST /users冪等でない
PUT更新（全体置換）PUT /users/123冪等
PATCH更新（部分）PATCH /users/123—
DELETE削除DELETE /users/123冪等

良し悪しは具体例で一目瞭然です。下の対比のように、動詞をURIに埋め込んだり、略語や大文字を混ぜたりすると、とたんに使いにくくなります。

迷ったときの基準

「このURIを初見の人に見せたら、何のリソースか言い当てられるか？」を自問する。当てられるなら良いURI。一貫した複数形（/users・/articles）でリソースを並べ、関連は階層（/users/123/articles）で表すと、全体に統一感が出ます。

04
Part II · 設計の実際

リクエストの設計

同じ「取得」でも、パスパラメータとクエリパラメータは役割が違います。本書の使い分けは明快です。リソースを一意に特定するものはパス、絞り込み・並べ替え・任意指定はクエリ。

# パス＝リソースの特定／クエリ＝絞り込み・任意指定
GET /users/123                 # 123 番のユーザー（特定）
GET /users?role=admin&sort=name # 条件で絞る・並べ替える（任意）

一覧APIで欠かせないのがページネーション。件数が増えても安全に返せるよう、ページ単位で区切ります。素朴な page/per_page は分かりやすい一方、データが更新されると件数ズレが起きやすい。安定性が要るなら「最後に見たIDの続きから」返す方式（カーソル/キーセット）が堅牢、と本書も触れています。

方式例長所 / 短所
オフセット型?page=3&per_page=20直感的 / 更新中に重複・抜けが起きやすい
カーソル型?limit=20&since_id=456大規模・更新中でも安定 / 任意ページへ飛びにくい

そして認証。一般公開でない操作には、リクエストが「誰のものか」を示す必要があります。広く使われるのが OAuth 2.0（第三者アプリにパスワードを渡さず権限委譲する仕組み）と、サーバ間で手軽な APIキー。資格情報は必ずヘッダ（Authorization）で渡し、URLに載せない——URLはログやブラウザ履歴に残るためです。

やりがちな失敗

APIキーやトークンを ?api_key=xxxx のようにクエリに置くこと。アクセスログ・プロキシ・リファラに漏れます。秘密は Authorization ヘッダへ。あわせて通信は必ず HTTPS（§09）で、平文の盗聴を防ぎます。

05
Part III · 返し方

レスポンスデータの設計

返すデータの形式は、いまやJSONが標準。本書がレスポンス設計で説く要点は、どれも「使う側が楽になるか」を基準にしています。

  • 余計なエンベロープで包まない：{"data":{...}} のような共通ラッパは多くの場合不要。HTTPが既にメタ情報（ステータス・ヘッダ）を持つので、本体は素直に返す。
  • むやみにネストを深くしない：階層が深いと使う側のアクセスが煩雑。フラットめな構造を心がける。
  • 名前は一貫させる：camelCase か snake_case をAPI全体で統一。混在は最悪。
  • 日付は標準形式で：独自フォーマットを避け、RFC 3339 / ISO 8601（例 2024-01-15T09:30:00Z）で。タイムゾーンも明示。
// ○ 素直でフラット。日付はRFC3339、命名は統一
{
  "id": 123,
  "name": "Sato Hanako",
  "role": "admin",
  "created_at": "2024-01-15T09:30:00Z"
}
なぜエンベロープを避けるのか

「成功/失敗」や「件数」を本体に詰め込みたくなりますが、それらはHTTPのステータスコードやヘッダの役目。本体で重複させると、使う側は「HTTPステータスと本体のどちらを信じるか」で混乱します。HTTPに任せられることはHTTPに任せ、本体はリソースそのものに集中させる——これが self-descriptive な設計です。

06
Part III · 返し方

エラーの設計

エラーレスポンスは、APIの親切さが最も出るところ。鉄則は2つ。① 適切なHTTPステータスコードを返す ② 本体に人間（と機械）が読める詳細を載せる。成功時は 200 なのにエラーでも 200 を返し、本文に {"error":true} と書く——これは最悪のアンチパターンです。

# ステータスでまず種別を伝え、本体で具体を伝える
HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "code": "invalid_parameter",
  "message": "email の形式が正しくありません",
  "details": [{ "field": "email", "reason": "format" }]
}
系統意味代表例
2xx成功200 OK / 201 Created / 204 No Content
3xxリダイレクト・未変更301 Moved / 304 Not Modified
4xxクライアント側の誤り400 / 401 認証 / 403 権限 / 404 / 429 過多
5xxサーバ側の誤り500 / 503 Service Unavailable
アンチパターン

何でも 200 で返し、本体でエラーを示す。これだと利用者は毎回本体を解析せねばならず、HTTPの仕組み（リトライ・監視・キャッシュ）も効かなくなります。4xx は「あなたの誤り」、5xx は「こちらの不調」を正しく使い分けるだけで、使う側の対応が劇的に楽になります。

07
Part IV · HTTPを活かす

HTTPを味方にする

Web APIの土台はHTTP。本書が強く勧めるのは「HTTPが既に持つ機能を、自前で再発明しない」こと。中でも効果が大きいのがキャッシュです。適切にキャッシュ制御すれば、サーバ負荷も通信量も応答時間も同時に改善します。

鍵は条件付きリクエスト。レスポンスに「このデータの版」を表す ETag を付けておくと、次回クライアントは「この版から変わった？」と尋ねられます。変わっていなければサーバは本体を返さず 304 Not Modified だけ返す——中身を再送しないので速く軽い。

明示的な期限は Cache-Control で指定します（例 max-age=3600 で1時間有効）。さらに、レスポンスのメディアタイプを Content-Type: application/json と正しく宣言する、gzip圧縮を効かせるなど、HTTPの作法を守るだけでAPIは速く・親切になります。

同時更新の衝突も防げる

ETag は更新時にも効きます。If-Match: "v7" 付きで PUT すれば、自分が見た版から他人が変更していたら拒否できる（楽観的ロック）。「後勝ちで上書きして気づかない」事故を、これまたHTTP標準で防げます。

Part V · 育てる

変化に強くする

APIは公開した瞬間から、誰かの本番コードが依存し始めます。だから「壊さずに進化させる」設計が要る。第一歩は後方互換性を保つこと。フィールドの追加は安全、削除・改名・意味変更は破壊的——この感覚を常に持ちます。

変更互換性方針
フィールドを追加する安全（後方互換）既存利用者は無視できる。やってよい
フィールドを削除/改名する破壊的新バージョンで。旧版はしばらく残す
必須パラメータを増やす破壊的任意＋デフォルトで導入する
レスポンスの意味を変える破壊的避ける。どうしても要るなら別エンドポイント

破壊的変更が避けられないときはバージョニング。本書はいくつかの方式を比較しますが、実務で広く使われるのはURIにメジャー版を埋める方式（/v1/users）。シンプルで、ブラウザでも確認しやすいのが利点です。

# メジャー版をURIに（最も分かりやすい）
GET /v1/users/123
GET /v2/users/123      # 破壊的変更は新バージョンで提供

# 旧版を畳むときは予告する（Sunsetヘッダ等で廃止日を通知）
HTTP/1.1 200 OK
Sunset: Sat, 31 May 2025 23:59:59 GMT
バージョンは増やしすぎない

安易に /v2、/v3 と分けると、維持コストが倍々になります。まずは後方互換な変更（追加中心）で乗り切り、メジャー版を上げるのは本当に必要なときだけ。上げたら旧版の廃止時期を必ず予告し、利用者に移行の猶予を与えます。

09
Part V · 育てる

堅牢にする

公開APIは、善意のユーザーだけが叩くとは限りません。本書の最終テーマは「安全に、落ちずに」。土台は常時HTTPSです。平文HTTPは盗聴・改ざんし放題なので、認証情報を扱うAPIでは必須。これは議論の余地がありません。

そのうえで、Web API特有のリスクへ備えます。本書が挙げる代表的なものを押さえておきましょう。

過負荷対策

レートリミット

一定時間あたりの呼び出し回数を制限。超過は 429 Too Many Requests を返し、残量を X-RateLimit-* ヘッダで通知する。

なりすまし対策

CSRF / Origin確認

ブラウザ経由の不正リクエストを防ぐ。トークン検証や Origin/Referer の確認で、意図しない実行を弾く。

情報漏れ対策

JSONハイジャック対策

JSONを直接スクリプトとして読まれないよう、必ずオブジェクトを最上位に（配列を直接返さない）等の防御を施す。

入力検証

バリデーション

受け取った値は信用しない。型・範囲・形式を検証し、SQLインジェクション等を入口で止める。

加えて、攻撃の手がかりを与えない配慮も大切です。エラーメッセージで内部実装（スタックトレースやDB構造）を漏らさない、不要なサーバ情報ヘッダを出さない。「親切な詳細」と「漏らしてはいけない情報」の線引きを意識します。

堅牢さの土台

HTTPS・レートリミット・入力検証——この3つは、ほぼすべての公開APIに共通の最低ライン。派手な対策の前に、まずここを固める。セキュリティは「使いやすさ」と対立しがちですが、標準的な仕組みに乗れば両立できるのが、HTTPを活かす設計の強みです。

一言で

Web APIは人が使うUI。HTTPに逆らわず、リソースを名詞のURIで表し、メソッドとステータスコードに語らせる。素直なレスポンス、適切なエラー、互換性への配慮、標準的な防御——どれも派手さはないが、積み重ねれば「使い続けたくなるAPI」になります。

用語集

RESTリソース(名詞)をURIで表し、HTTPメソッド(動詞)で操作する設計スタイル。
エンドポイント / URIリソースを指す住所。短く・名詞・複数形・規則的に。
冪等(べきとう)同じ操作を何度繰り返しても結果が変わらない性質。GET/PUT/DELETEが該当。
パス / クエリパラメータリソースの特定はパス、絞り込み・任意指定はクエリで渡す。
ページネーション一覧を分割して返す仕組み。大規模・更新中はカーソル型が安定。
ステータスコード結果の種別を表す3桁。2xx成功/4xxクライアント誤り/5xxサーバ誤り。
ETag / 条件付きリクエストデータの版を示す印。変わっていなければ304で本体を省き、衝突検知にも使える。
後方互換性既存利用者を壊さない性質。フィールド追加は安全、削除・改名は破壊的。
バージョニング破壊的変更を分離する仕組み。/v1/ のようにURIへ埋めるのが分かりやすい。
レートリミット呼び出し回数の制限。超過は429で返し、残量をヘッダで通知する。
