gqlgen コード解剖
書くコードと生成コードを全部見る

同じスキーマから、あなたが書くコードと gqlgen が生成するコードがどう生まれるのか。そして「自動バインドされたフィールド」と「リゾルバになったフィールド」が、生成コードのどの一行で分岐するのかを、実物のコードで開いて見せます。

02

入力 — あなたが書くスキーマ

出発点はこれ一枚だけ。todo アプリのスキーマです。注目してほしいのは Todo の中の user フィールド。後で「これだけリゾルバにする」操作をします。

✍ あなたが書くgraph/schema.graphqls
type Todo {
  id: ID!
  text: String!
  done: Boolean!
  user: User!
}

type User {
  id: ID!
  name: String!
}

type Query {
  todos: [Todo!]!
}

type Mutation {
  createTodo(input: NewTodo!): Todo!
}

input NewTodo {
  text: String!
  userId: String!
}

あなたが書いたのは、ここまでです。以降に出てくるコードのうち色が変わるものは、すべてここから generate で生えてきます。

03

生成① — モデル構造体

まず、スキーマの各 type / input に対応する Go の構造体が生成されます。String! は string、Boolean! は bool へ、というように型がマッピングされています。

⚙ 生成（自動バインド）graph/model/models_gen.go
package model

type Todo struct {
	ID   string `json:"id"`
	Text string `json:"text"`
	Done bool   `json:"done"`
	User *User  `json:"user"`
}

type User struct {
	ID   string `json:"id"`
	Name string `json:"name"`
}

type NewTodo struct {
	Text   string `json:"text"`
	UserID string `json:"userId"`
}
理屈

この構造体があることが、後の「バインドできるかどうか」の判定材料になります。例えば Todo.text は、この構造体に Text string があるので「読むだけでよい＝自動バインド」と判断されます。スキーマのフィールド名と構造体のフィールド名が対応するか——これが分岐の鍵です。

04

生成② — リゾルバの「契約」（インターフェース）

次に、生成エンジンの中に「あなたが実装すべき関数の一覧」がインターフェースとして定義されます。これは契約書です。「この形の関数を用意してね」という宣言だけがあり、中身はありません。

⚙ 生成（リゾルバの契約）graph/generated/generated.go（抜粋）
// 全リゾルバの入口
type ResolverRoot interface {
	Mutation() MutationResolver
	Query() QueryResolver
	Todo() TodoResolver        // ← Todo に「穴」がある時だけ生える
}

type QueryResolver interface {
	Todos(ctx context.Context) ([]*model.Todo, error)
}

type MutationResolver interface {
	CreateTodo(ctx context.Context, input model.NewTodo) (*model.Todo, error)
}

type TodoResolver interface {
	User(ctx context.Context, obj *model.Todo) (*model.User, error)
}
読み方のコツ

QueryResolver や MutationResolver は最初から生えます（Query/Mutation は必ずリゾルバだから）。一方 TodoResolver の User(...) は、Todo.user をリゾルバにする設定をしたときだけ現れます。obj *model.Todo という引数に注目 ―― 「どの todo の user か」が渡ってくるのがフィールドリゾルバの特徴です。

05

生成③ — 実行ラッパー（このページの核心）

ここが一番見せたかった所です。生成エンジンは、フィールド 1 つにつき 1 個、値を取り出す関数（ラッパー）を作ります。Todo.text(バインド) と Todo.user(リゾルバ) の 2 つを並べます。外側の定型処理はほぼ同一で、違うのは中央のたった一行だけです。

⚙ バインド_Todo_text
func (ec *executionContext) _Todo_text(
  ctx context.Context, field …, obj *model.Todo,
) (ret graphql.Marshaler) {

  // フィールド情報の準備・
  // panic 復帰などの定型処理（省略）

  resTmp, err := ec.ResolverMiddleware(ctx,
    func(ctx context.Context) (any, error) {
      return obj.Text, nil   // ★ 構造体から直接読む
    },
  )

  // null チェック・型アサート（省略）
  res := resTmp.(string)
  return ec.marshalNString(ctx, …, res)
}
⚙ リゾルバ_Todo_user
func (ec *executionContext) _Todo_user(
  ctx context.Context, field …, obj *model.Todo,
) (ret graphql.Marshaler) {

  // フィールド情報の準備・
  // panic 復帰などの定型処理（省略）

  resTmp, err := ec.ResolverMiddleware(ctx,
    func(ctx context.Context) (any, error) {
      return ec.resolvers.Todo().User(ctx, obj) // ★ あなたのリゾルバを呼ぶ
    },
  )

  // null チェック・型アサート（省略）
  res := resTmp.(*model.User)
  return ec.marshalNUser(ctx, …, res)
}
ここだけ覚えればいい

2 つの関数の違いは中央の一行だけ:

return obj.Text, nil ＝ 構造体のフィールドをそのまま読む（だからあなたは何も書かない）。
return ec.resolvers.Todo().User(ctx, obj) ＝ あなたが書いたリゾルバを呼ぶ（だから穴を埋める必要がある）。

第 4 節の契約（インターフェース）は、まさにこの ec.resolvers.Todo().User(...) を呼ぶための型でした。生成コードはこうしてあなたのコードへバトンを渡しているのです。

余談 — なぜ自分で書きたくないか

実際の marshalNUser… は marshalNUser2ᚖexampleᚗcomᚋgqlgenᚑtodosᚋgraphᚋmodelᚐUser のような、記号でエンコードされた長い名前になります。型ごとに変換関数を用意するための機械的な命名です。こういうコードを人間が手書きしないで済むのが、生成の価値です。

06

生成④ — リゾルバの「穴」

第 5 節の右側が呼んでいた ec.resolvers.Todo().User(...)。その実体を入れる場所が、この schema.resolvers.go に生成されます。生成直後は中身が panic の穴です。

⚙ 生成（穴 / 未実装）graph/schema.resolvers.go（生成直後）
// ↓↓↓ あなたが埋めるべき穴（panic のまま）↓↓↓

func (r *queryResolver) Todos(ctx context.Context) ([]*model.Todo, error) {
	panic(fmt.Errorf("not implemented: Todos - todos"))
}

func (r *mutationResolver) CreateTodo(ctx context.Context, input model.NewTodo) (*model.Todo, error) {
	panic(fmt.Errorf("not implemented: CreateTodo - createTodo"))
}

func (r *todoResolver) User(ctx context.Context, obj *model.Todo) (*model.User, error) {
	panic(fmt.Errorf("not implemented: User - user"))
}

// ↓↓↓ ここから下も生成される「配線」。基本さわらない ↓↓↓

func (r *Resolver) Mutation() MutationResolver { return &mutationResolver{r} }
func (r *Resolver) Query() QueryResolver       { return &queryResolver{r} }
func (r *Resolver) Todo() TodoResolver         { return &todoResolver{r} }

type mutationResolver struct{ *Resolver }
type queryResolver   struct{ *Resolver }
type todoResolver    struct{ *Resolver }
理屈 — 配線の意味

下半分の func (r *Resolver) Todo() TodoResolver が、第 4 節の契約を満たす「実装」です。これにより生成エンジンの ec.resolvers.Todo().User(...) が、最終的にあなたの todoResolver.User にたどり着きます。穴（panic の関数）＝あなたの担当、配線＝gqlgen の担当、と分かれているわけです。

07

あなたが書く — 穴を埋める

あとは panic を消して、本物の処理に書き換えるだけ。これが gqlgen であなたが書くコードのほぼすべてです。resolver.go に状態（ここではメモリ上の todos）を持たせ、各穴を埋めます。

✍ あなたが書くgraph/resolver.go
package graph

type Resolver struct {
	todos []*model.Todo   // アプリの状態（本番なら *db.Queries など）
}
✍ あなたが書く（穴を埋めた後）graph/schema.resolvers.go
// 穴① 一覧を返す
func (r *queryResolver) Todos(ctx context.Context) ([]*model.Todo, error) {
	return r.todos, nil
}

// 穴② todo を作って返す（user 本体は作らず UserID だけ持たせる）
func (r *mutationResolver) CreateTodo(ctx context.Context, input model.NewTodo) (*model.Todo, error) {
	n, _ := rand.Int(rand.Reader, big.NewInt(100000))
	todo := &model.Todo{
		ID:     fmt.Sprintf("T%d", n),
		Text:   input.Text,
		UserID: input.UserID,
	}
	r.todos = append(r.todos, todo)
	return todo, nil
}

// 穴③ user が要求された時だけ呼ばれる（obj = その todo）
func (r *todoResolver) User(ctx context.Context, obj *model.Todo) (*model.User, error) {
	return &model.User{ID: obj.UserID, Name: "user " + obj.UserID}, nil
}

ハイライト部分（ピンク）が、あなたが純粋に書き足したロジックです。関数のシグネチャ（名前・引数・戻り値）は生成された契約に合わせるだけで、考えるのは中身だけ。これが「穴を埋める」の正体です。

08

バインドを「穴」に変えるスイッチ

ここまでで、Todo.user がリゾルバになっている前提で進めました。ではどうやって「バインド」から「リゾルバ」へ切り替えたのか? あなたが書く設定はこの数行です。

✍ あなたが書くgqlgen.yml
models:
  Todo:
    fields:
      user:
        resolver: true   # ← この一行が分岐を右（穴）へ曲げる
何が変わるのか

resolver: true を書く前と後で、生成コードはこう変わります。

  • 書く前（バインド）: 第 5 節の _Todo_user は return obj.User, nil を生成。TodoResolver インターフェースも穴も生成されない。あなたは User を実装しない。
  • 書いた後（リゾルバ）: 同じ _Todo_user が return ec.resolvers.Todo().User(ctx, obj) を生成。TodoResolver インターフェースと穴が生える。あなたは User を実装する。

つまり gqlgen.yml のたった一行が、第 5 節で見た「中央の一行」をどちらにするかを決めていた、というわけです。

09

全体の対応表

最後に、ここまで見たコードを一望できる表にまとめます。色（誰が書くか）と、それがスキーマのどこから来たかが分かります。

コード由来誰が書く
schema.graphqls—（出発点）あなた
model.Todo / User / NewTodoスキーマの type / input生成（バインド）
QueryResolver など interfaceQuery/Mutation＋穴のあるtype生成（契約）
_Todo_text（obj.Text を読む）バインドされたフィールド生成（バインド）
_Todo_user（リゾルバを呼ぶ）resolver:true のフィールド生成（リゾルバ）
Todos / CreateTodo / User の穴リゾルバの契約生成（穴）
穴の中身（ロジック）—あなた
resolver.go の状態—あなた
gqlgen.yml の resolver:true—あなた
一言でいうと

あなたが書くのは 4 種類だけ: スキーマ、穴の中身、状態(resolver.go)、設定(gqlgen.yml)。残りは全部 gqlgen が生成し、生成コードは ec.resolvers.Todo().User(...) という一行であなたのコードへバトンを渡す。「バインド＝obj から読む／リゾルバ＝あなたを呼ぶ」、生成コードの違いはこの中央一行だけ——これが gqlgen のすべてです。

生成コードは読みやすさのため定型処理（panic 復帰・null チェック・型アサート・マーシャル名のエンコード等）を一部省略しています。実際の出力は gqlgen のバージョンにより細部が異なります。正確な出力は手元で go tool gqlgen generate 後の graph/generated/ を参照してください。公式: gqlgen.com
