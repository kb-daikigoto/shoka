Unit Testing · Principles
Tests that enable sustainable growth

単体テストの考え方/使い方
持続可能な成長を支えるテスト設計

テストは「数を増やせば安心」ではありません。質の低いテストは、むしろ開発を遅らせる。本書の核心は「単体テストの目的は、プロジェクトの持続可能な成長を支えること」。何を・どう・どこまでテストすれば価値が出るのか。良いテストと悪いテストを見分ける物差しを、設計の理由とともに自分の言葉で再構成します。

01

一言でいうと

本書がまず壊すのは「テストは多いほど良い」という思い込みです。テストにも保守コストがある。少しの変更で大量のテストが赤くなる、何を守っているか分からない——そんなテストは負債です。だから問うべきは「カバレッジ何%か」ではなく、「このテストは価値があるか」。

テストの目的は、コードベースが大きくなっても変更し続けられること。プロジェクトは時間とともに必ずエントロピーが増し、変更が怖くなる。良いテストはその劣化に抗い、リファクタリングを安全にし、開発速度を長期にわたって保ちます。これが「持続可能な成長」の意味です。

この記事のゴール

自分のテストを見て「これは価値があるか／ただの足かせか」を判断できるようになること。テストを書く技術と同じくらい、テスト“しやすい”コードを設計する視点を持ち帰ります。

02
Part I · 物差し

良い単体テストの4本柱

本書の背骨が、良いテストを測る4つの柱です。テストの価値は、この4つの掛け算で決まります（どれかが0なら価値も0）。

柱1

退行(リグレッション)に対する保護

バグ（=機能の退行）をどれだけ捕まえられるか。多くのコードを通り、重要なロジックを検査するほど高い。

柱2

リファクタリングへの耐性

振る舞いを変えずに内部を直したとき、誤って赤くならないか。偽陽性の少なさ。

柱3

迅速なフィードバック

速く実行できるか。遅いテストは回されなくなり、保護の価値ごと失われる。

柱4

保守のしやすさ

テスト自体が読みやすく・壊れにくいか。理解と修正のコスト。

残酷な事実として、柱1〜3を同時に最大化することはできません。3つのうち2つを選ぶトレードオフです。たとえば「保護」と「迅速さ」を極めると、テストが実装に密着して「リファクタ耐性」を失いがち。だから本書は優先順位を示します——リファクタリングへの耐性は、削ってはいけない。ここを犠牲にしたテストは、すぐに信頼を失うからです。

なぜ耐性が最優先か

テストが偽陽性（振る舞いは正しいのに赤くなる）を連発すると、人は赤信号を信じなくなります。「また誤検知だろう」とテストを無視し始め、やがて本物のバグも見逃す。テストへの信頼こそが全ての土台であり、それを壊すのが偽陽性。だから耐性は譲れません。

03
Part I · 物差し

2つの学派と定義

「単体（unit）」とは何かで、流派が分かれます。ロンドン学派（mockist）は「1クラス＝1単位」とし、依存をすべてモックに置き換えて隔離します。一方古典学派（Detroit/classical）は「1つの振る舞い＝1単位」とし、実物の依存を使い、テストどうしを隔離（共有状態を持たせない）します。

観点古典学派（本書が推す）ロンドン学派
隔離するものテストどうしテスト対象（依存を全モック）
単位1つの振る舞い（複数クラス可）1クラス
依存の扱い実物を使う（共有依存だけ差替）原則すべてモック
リファクタ耐性高い（実装に縛られにくい）低くなりがち

本書は古典学派を支持します。理由は柱2——ロンドン学派のように依存を全部モックにすると、テストがクラスの内部構造（誰を呼ぶか）に縛られ、リファクタリングで簡単に壊れるから。モックは「使ってよい場面」を絞るべき、という主張につながります（→§06）。

用語を1つ整理

「単体テスト」とは本書では① 1つの振る舞いを検証し、② 実行が速く、③ 他のテストから隔離されているテスト。この3条件を満たさない（例: 実DBや外部APIを触る）ものは統合テストとして区別します（→§08）。

04
Part II · 何を検証するか

何をテストするか

柱2（リファクタ耐性）を決める最重要の判断が、これです。テストは「観察可能な振る舞い」だけを検証し、「実装の詳細」には触れない。観察可能な振る舞いとは、クライアントが意味を持って使う公開API——入力に対する出力や、状態の変化です。実装の詳細とは、それを実現する内部の手続き。

判定法はシンプル。「このメソッドは、クライアントが目的を達成するために直接呼ぶか？」がYesなら振る舞い、Noなら実装の詳細です。private メソッドを無理にテストしたくなったら、それは設計のにおい——隠れた責務が公開クラスとして独立すべきサインかもしれません。

実装の詳細に縛られた例

「内部で calculateTax() が呼ばれたこと」を検証するテストは、税計算のやり方を変えただけで壊れます。本当に守りたいのは「正しい税額が結果に反映されること」。How（手順）ではなく What（結果）を検証する——これが耐性の源です。

05
Part II · 何を検証するか

構造とAAAパターン

読みやすいテストには型があります。AAA（Arrange-Act-Assert）——準備し、実行し、確認する。3ブロックに分けるだけで、テストが一目で読めるようになります。

test("残高を超える出金は失敗する", () => {
  // Arrange : 前提を整える
  const account = new Account({ balance: 1000 });

  // Act : 検証したい操作を1つだけ実行
  const result = account.withdraw(1500);

  // Assert : 結果を確認
  expect(result.isSuccess()).toBe(false);
  expect(account.balance).toBe(1000);   // 残高は不変
});

良いテストの作法は、本書ではこう整理されます。1テストにつき Act は1つ（複数の振る舞いを混ぜない）、条件分岐を書かない（テストにifがあるのは、テストが複雑すぎる証拠）、そしてテスト名は「何を検証するか」を日本語の文で表す。test_withdraw_1 ではなく「残高を超える出金は失敗する」と。

名前は仕様書になる

良いテスト名は、ドメインを知る非開発者が読んでも分かる振る舞いの説明です。メソッド名を機械的に並べた名前（withdraw_returnsFalse）ではなく、ビジネス上の意味を書く。テスト一覧が、そのまま「このシステムは何を約束するか」の目録になります。

06
Part III · 依存とモック

モックの正しい使い方

モックは強力ですが、使いすぎると柱2を壊します。本書の指針は明快——モックを使ってよいのは「システム境界を越えるプロセス外依存」かつ「アプリの外から観察できる相互作用」だけ。まずテストダブルを2種に分けて捉えます。

スタブとの相互作用を検証してはいけません。値を取得するだけのクエリは、何回呼ぼうが結果に影響しない。「getConfig() が呼ばれたこと」を verify するのは、実装の詳細の検証＝過剰検証です。一方、メール送信のような外部から観察できる副作用（コマンド）は、「送ったこと」自体が振る舞いなのでモックで検証してよい。

// ○ コマンド(外部への副作用)はモックで検証してよい
emailGateway = Mock(IEmailGateway);
service.confirm(user);
verify(emailGateway).send(user.email);   // 送信は観察可能な振る舞い

// ✕ クエリ(値取得)の呼び出し回数を検証するのは過剰
verify(configStore).get("rate");     // ← 実装の詳細に縛られる
プロセス内依存はモックにしない

同じプロセス内の依存（自分のドメインオブジェクト等）をモックすると、テストが内部構造に密着します。モックの対象はDB・メール・外部APIなど“外”との、しかも外から見える相互作用に限る。それ以外は実物を使うのが古典学派の流儀です。

07
Part III · 依存とモック

コードの4象限

「テストしにくい」は、たいていコードの問題です。本書はコードを2軸——複雑さ/重要度と協力者（依存）の数——で4象限に分けます。テストの費用対効果は、どの象限かで決まります。

厄介な「過度に複雑なコード」（複雑かつ依存だらけ）を消す鍵が Humble Object パターン。テストしにくい部分（外部依存とのやり取り）から、テストしたいロジックを引きはがす。複雑なロジックは依存のないドメインモデルへ、外部とのつなぎは薄いコントローラへ。前者は単体テストで手厚く、後者は統合テストで少数、と役割が分かれます。

テスト容易性＝良い設計

「ロジックと副作用の分離」は、関数型でいう関数的核（functional core）と命令的な殻（imperative shell）でもあります。テストしやすい形に寄せると、結果としてコードの見通しも良くなる。テストのしにくさは、設計からのフィードバック——本書が繰り返す最も重要なメッセージの一つです。

08
Part IV · 広げる

統合テスト

単体テストは「左上」を守りますが、部品をつないだ時に壊れることは捕まえられません。そこを担うのが統合テスト。実物のDBや外部システムを通して、コントローラ（つなぎ）が正しく協調するかを検証します。本書は「最も長いハッピーパスを統合テストで1本通し、細かな分岐や端のケースは単体テストで」という配分を勧めます。

依存の種類例テストでの扱い
管理下にあるプロセス外依存自分のDB（他から見えない）実物を使う（状態で検証）
管理外のプロセス外依存外部API・メール・他社サービスモックにする（相互作用で検証）
プロセス内依存自分のドメインオブジェクト実物を使う（モックにしない）

勘どころは「管理下のDBはモックにしない」こと。自分のDBは外から観察されない実装の詳細なので、実物に対して最終的な状態を検証する方が、振る舞いを正しく捉えます。逆に、他社が見ている外部APIは「どんな要求を送ったか」が観察可能な契約なので、モックで相互作用を検証します。

テストピラミッド

速くて多数の単体テストを土台に、少数の統合テスト、ごく僅かのE2Eを頂点に積む。ただしドメインが単純なCRUD中心なら、ピラミッドは平たくなる（単体で守る価値が薄く、統合テスト寄りに）。形は教条ではなく、コードの性質で決まります。

09
Part IV · 広げる

アンチパターンと罠

最後に、価値を下げる代表的な落とし穴を。どれも「柱を壊している」という一点で共通します。

  • privateメソッドを直接テストする：実装の詳細の検証＝耐性を失う。公開された振る舞い経由で間接的に検証する。
  • テストにドメイン知識を漏らす：本番ロジックをテスト側でも再計算すると、同じ間違いを両方でしてすり抜ける。期待値はべた書きする。
  • 現在時刻・乱数への依存：実行のたびに結果が変わる不安定テスト。時刻や乱数は注入して固定する。
  • テスト間で状態を共有：実行順で結果が変わる。各テストは独立・自己完結に。

そして最大の誤解がコードカバレッジ。本書はカバレッジを「目標」にすることを強く戒めます。カバレッジは「コードが実行されたか」を測るだけで、「正しく検証されたか」は測れない。アサーションが無くても数字は上がる。100%でもバグだらけにできるし、低くても良いテストはあり得ます。

カバレッジは目標にするな

「カバレッジ80%必達」をルールにすると、人は数字を満たすためだけの空虚なテストを書き始めます。アサーションの薄い、ただ実行するだけのテスト——これは負債そのもの。カバレッジは低い箇所を見つける道具であって、品質の証明ではありません。良い指標は、結局のところ「変更が怖くないか」です。

一言で

良い単体テストとは、観察可能な振る舞いを・速く・壊れにくく検証するもの。リファクタリングへの耐性を最優先に、モックは外部との相互作用だけに絞り、テストしにくさは設計のサインとして受け止める。テストとプロダクトコードは地続きの設計問題です。
