00

データ指向とは何か

処理速度(CPU)よりも、データの量・複雑さ・変化の速さが主な難しさになるアプリ ― それが「データ指向」アプリである。

かつてアプリの限界はCPU能力でした（計算指向）。今は違います。多くのアプリの限界はデータ側にあります ― 膨大なデータ量、複雑に絡む関係、そして絶えず変わる要求。こうしたアプリをデータ指向（data-intensive）と呼びます。

そして現代のアプリは、ゼロから作るのではなく、標準化されたデータシステムの部品を組み合わせて作ります。

store

データベース

データを保存し、後で取り出す。

speed

キャッシュ

重い処理の結果を覚えておく。

search

検索インデックス

キーワードやフィルタで素早く探す。

flow

ストリーム / バッチ処理

メッセージを流す／大量データを定期処理する。

本書が一貫して問うこと

これらの部品をどう組み合わせれば、信頼でき・伸ばせて・保ち続けられるシステムになるか。そして、どの選択にも必ず代償（トレードオフ）があること。「銀の弾丸はない。あるのはトレードオフの理解だけ」 ― これが全巻を貫く態度です。まずはその評価軸となる三つの関心事から。

PART A

設計の三つの関心事

あらゆる設計判断を測る物差し ― 信頼性(Reliability)・拡張性(Scalability)・保守性(Maintainability)。バズワードではなく、具体的な意味を持たせて使えるようにします。

01

信頼性 ― 壊れても動き続ける

信頼性とは「障害(fault)が起きても、システム全体の停止(failure)に至らせない」こと＝フォールトトレラント。

まず2つの言葉を区別します。fault（フォールト）は部品レベルの不具合（ディスク1台の故障など）、failure（フェイラー）はシステムが利用者への約束を果たせなくなる全体停止です。faultを完全にゼロにはできないので、設計のゴールは「faultがfailureに発展しないようにする」こと。これがフォールトトレランスです。

障害の種類性質主な対処
ハードウェア個々はランダムに故障冗長化（部品を多重に持つ）
ソフトウェア条件が揃うと一斉に多発しやすいテスト・監視・分離・段階的展開
人的ミス最も多い障害原因戻せる仕組み・サンドボックス・自動化
tip ― あえて壊して鍛える

「いつか壊れる」を前提に、平時から意図的に障害を注入してフォールトトレランスを検証する考え方があります（カオスエンジニアリング）。障害を例外扱いせず、日常として設計に組み込む発想が、信頼性の出発点です。

02

拡張性 ― 負荷が増えても保てる

「スケーラブル」とは言わない。負荷(load)を数値で定義し、それが増えたとき性能をどう保つかを語る。

拡張性を曖昧な形容詞で語ってはいけません。本書は2段階で具体化します ― まず負荷を負荷パラメータ（数値）で表す、次に負荷が増えたときの性能を測る。

負荷パラメータと、設計が変わる例

負荷パラメータはアプリ次第です（毎秒リクエスト数、読み書き比、同時接続数など）。重要なのは、同じ機能でも、どのパラメータが大きいかで最適な設計が変わること。たとえばSNSのタイムラインは、「投稿時に各フォロワーの受信箱へ配る（書き込み時に展開）」のか「閲覧時に集めて作る（読み込み時に展開）」のかで、まったく違う設計になります。有名人のフォロワーが膨大な場合、前者は書き込みが爆発します。

性能は平均でなくパーセンタイルで見る

性能にはスループット（単位時間の処理量）と応答時間（1リクエストの所要時間）があります。そして応答時間は平均で見てはいけません。本当に効くのは、遅い側の裾（テール）です。

図: 応答時間は右に裾を引く。平均では見えない p95/p99（テールレイテンシ）こそ体験を左右する
なぜテールが重要か

① 最も遅い体験をする利用者は、しばしばデータを多く抱える優良顧客。② 1つの画面表示が内部で何十回もの呼び出しに展開されると、その中で最も遅い1つに全体が引きずられる（テールレイテンシ増幅）。だから「平均は速い」では不十分で、p95やp99を目標値に据えます。スケール手段としては、1台を強化するスケールアップ（垂直）と台数を増やすスケールアウト（水平）があり、万能の自動構成はなく、負荷の形に合わせて設計します。

03

保守性 ― 未来の自分とチームのために

ソフトのコストの大半は開発後の運用・改修。保守性は「運用性・単純性・進化性」の3つで考える。

operability

運用性

運用チームが楽に保てること。良い監視、見通しのよい挙動、定型作業の自動化。

simplicity

単純性

本質的でない偶発的複雑性を、適切な抽象化で取り除く。複雑さは事故とコストの源。

evolvability

進化性

要求変化に合わせて変えやすいこと。前作で扱った疎結合・テスト・リファクタリングが効く。

要点 ― 前作とのつながり

「進化性」は、姉妹編『ちょうぜつソフトウェア設計入門』の主題そのものです。結合を弱め、抽象に依存し、テストで守る ― あの設計原則は、システム規模になっても同じく効きます。本書はそれを「データシステム全体」のスケールへ拡張したもの、と捉えると見通しが効きます。

PART B

データモデルとクエリ言語

データを「どんな形」で持つか。リレーショナル・ドキュメント・グラフの3モデルと、その問い合わせ方を、優劣ではなく「データの関係の形」で選びます。

04

3つのデータモデル

関係の形で選ぶ ― 入れ子ならドキュメント、多対多ならリレーショナル、複雑に絡む網ならグラフ。

アプリのオブジェクトと、保存形式の間にはしばしば溝があります（インピーダンスミスマッチ）。それを埋める代表的な3モデルを比べます。

図: データの「関係の形」が、適したモデルを決める
モデル得意向くケース
ドキュメント1対多の入れ子・データの局所性・柔軟なスキーマ1件で完結する木構造（1つの記事＋コメント等）
リレーショナル多対一・多対多・重複排除（正規化）と結合多くの実体が相互参照する業務データ
グラフ多対多が高度に絡み合う関係・経路探索SNSの交友、推薦、経路、知識グラフ
要点

「どれが優れているか」という問いは筋が悪い。データ同士の関係の形に、モデルを合わせるのが正解です。多対多が増えるほどドキュメントは苦しくなり、関係が網状ならグラフが輝きます。前作『DB設計』の正規化は、まさにこの多対多をリレーショナルで扱う技術でした。

05

クエリ言語とスキーマの扱い

「何が欲しいか」だけ書く宣言的クエリは、最適化と並列化をエンジンに任せられる。

宣言的 と 命令的

SQLのような宣言的クエリは「欲しい結果」だけを書き、取り出し手順はエンジンに委ねます。手順を逐一書く命令的コードと違い、エンジンが裏でインデックスを選び、順序を変え、並列実行できます。これは姉妹編『達人に学ぶSQL』の「集合で考える」と同じ思想。宣言的だからこそ、機械が賢く最適化できるのです。

スキーマ・オン・ライト と スキーマ・オン・リード

write

スキーマオンライト

書き込み時に構造を検証（RDBの明示スキーマ）。一貫性が高い。「静的型付け」に似る。

read

スキーマオンリード

読み込み時に構造を解釈（ドキュメントの暗黙スキーマ）。多様で変化するデータに柔軟。「動的型付け」に似る。

tip

データの形が揃っていて長く安定 → 書き込み時検証が安全。形が多様・頻繁に変わる・外部由来 → 読み込み時解釈が柔軟。どちらが上ではなく、データの性質で選びます。

PART C

ストレージエンジンの中身

DBは内部でどうデータを保存し、探しているのか。LSMツリーとBツリーという二大方式の違いが、書き込み重視か読み込み重視かを決めます。

06

LSMツリー と Bツリー

読みを速くするインデックスは、書きを遅くする。この綱引きへの2つの回答がLSMツリーとBツリーである。

最も単純なDBは「追記専用のログ」です。書き込みは末尾に足すだけで爆速ですが、読み出しは全件走査で激遅。そこでインデックスを足します。ただしインデックスは読みを速くする代わり、書き込みのたびに更新が必要で書きを遅くします。この本質的トレードオフ（姉妹編『DB設計』のインデックス章と同じ綱引き）への、設計の異なる2つの答えがこれです。

図: LSMは「ためて順次書き出し→後で統合」、Bツリーは「決まった場所を直接更新」
観点LSMツリーBツリー
書き込み◎ シーケンシャルで高スループット○ ページ単位の更新
読み込み○ 複数階層を確認（Bloomフィルタで緩和）◎ 予測しやすく安定
向くワークロード書き込みが多い読み中心・トランザクション
採用例の傾向新世代のKVS/分散DB系多くの伝統的RDB
warn ― 単純化しています

上は本質を掴むための簡略図です。実際にはLSMのコンパクション戦略、Bツリーの先行書き込みログ(WAL)など多くの工夫があり、製品ごとに挙動は異なります。どちらが速いかはワークロード次第で、最後は実測が原則です。

07

OLTP と OLAP ― 行指向と列指向

少数行をキーで触る業務処理(OLTP)と、大量行を集計する分析(OLAP)では、最適な保存の向きが正反対。

OLTP（業務トランザクション）OLAP（分析）
アクセス少数の行をキーで読み書き大量行を走査して集計
例注文の登録・残高更新「先月の地域別売上」
保存の向き行指向（1行をまとめて）列指向（1列をまとめて）
置き場所本番DBデータウェアハウス
なぜ分析は列指向なのか

分析は「全行の特定の数列だけ」を集計しがち。列ごとにまとめて保存すれば、必要な列だけ読めて無駄がなく、同種の値が並ぶので圧縮も猛烈に効きます。逆にOLTPは1行をまるごと扱うので行指向が有利。同じデータでも、用途で保存の向きを変える ― これが分析基盤を別に建てる理由です。

PART D

エンコーディングと進化

データをバイト列に変換して保存・送信する方法と、コードが変わってもデータが壊れない「互換性」の作り方。第2巻（分散）への橋渡しです。

08

互換性とスキーマ進化

大規模システムでは新旧のコードとデータが必ず共存する。だから前方・後方の両互換が要る。

メモリ上のオブジェクトを、保存・送信できるバイト列に変換するのがエンコード（シリアライズ）、戻すのがデコードです。形式はテキスト系（JSON/XML：汎用だが冗長）とバイナリ系（Protocol Buffers等：スキーマ前提でコンパクト）があります。ここで本当に重要なのは形式選びより互換性です。

図: 後方互換（新→旧データ）と前方互換（旧→新データ）。両方そろって安全に更新できる
なぜ両方いるのか ― ローリングアップグレード

大規模システムは全サーバーを一瞬で更新できません。新旧バージョンが同時に動く期間が必ず生じます。新コードが古いデータを読め（後方互換）、かつ古いコードが新データを読み飛ばせる（前方互換）からこそ、止めずに少しずつ更新できます。スキーマ進化の設計（フィールド追加は任意に、削除や型変更は慎重に）が、この共存を可能にします。

09

データフローの3つの経路

データはDB経由・サービス呼び出し・非同期メッセージの3経路で流れる。どれも互換性が生命線。

via database

データベース経由

書き手と読み手が時間をまたいで結びつく。今日書いたデータを来年の（別バージョンの）コードが読むことも。

via service

サービス経由（REST/RPC）

同期的に呼び合う。クライアントとサーバーのバージョンがずれても通じるよう互換性が必要。

via message

非同期メッセージ経由

メッセージブローカを挟み、送受信を時間的に切り離す（疎結合）。送り手は受け手の都合を待たない。

第2巻への橋

どの経路でも、送り手と受け手は別々に進化します。だから互換性は分散システムの前提条件。ここまでが「1台（あるいは論理的に1つ）のデータシステム」の土台でした。次の第2巻からは、データを複数のマシンに複製・分割したときに何が起きるか ― 分散データの世界へ進みます。

分散データ：複製と分割

データを複数マシンに「複製（レプリケーション）」し「分割（パーティショニング）」すると、可用性とスケールが上がる一方、一貫性という新たな難問が現れます。第2巻でその入口へ。

このガイドについて： 書籍『データ指向アプリケーションデザイン ― 信頼性、拡張性、保守性の高い分散システム設計の原理』（Martin Kleppmann 著、原題 Designing Data-Intensive Applications）の主要テーマを、コードを少し書ける初学者〜中級者向けに自分の言葉で再構成した独立教材です。原書の逐語引用は含みません。本巻は全4巻シリーズの第1巻（原書 第1部・1〜4章に対応）。姉妹編「達人に学ぶDB設計／SQL」「ちょうぜつソフトウェア設計入門」と接続する構成です。
