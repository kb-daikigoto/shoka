Git 操作 完全ガイド

分散バージョン管理 Git を、最初のセットアップから、コミット・ブランチ・マージ・リベース・リモート連携、そして「やらかした！」ときの救出術まで、これ一本で実務に入れるレベルにまとめました。コマンドは全部コピーして試せます。

対象 初学〜実務 前提 ターミナルの基本 形式 コマンド主体 読了 約 50 分

Part 1 — 基礎

01

Git とは

Git は分散型のバージョン管理システムです。ファイルの変更履歴を記録し、いつでも過去の状態に戻れ、複数人の作業を安全に統合できます。「分散型」とは、各自の手元に履歴の完全なコピーを持つという意味で、ネットに繋がっていなくてもコミットや履歴閲覧ができます。

もう一つの肝は、Git が変更を差分ではなくスナップショットとして扱うこと。コミットのたびに「その時点のプロジェクト全体の写真」を撮るイメージです（変わっていないファイルは前の写真への参照で軽量に保ちます）。

なぜ使うのか

「あの動いていた状態に戻したい」「誰がいつ何を変えた？」「複数人で同じファイルを触りたい」——これらを安全に解決するのが Git。履歴は資産で、丁寧に刻むほど後で自分を助けます。

02

セットアップ

まず名前とメールを設定します（コミットに記録される署名）。最初の一度だけでOK。

$ git --version                          # 入っているか確認
$ git config --global user.name  "Your Name"
$ git config --global user.email "you@example.com"
$ git config --global init.defaultBranch main   # 既定ブランチ名を main に

プロジェクトを Git 管理下に置くには 2 通り。新規なら init、既存のリモートからなら clone。

$ git init                 # 今のフォルダを新規リポジトリに
$ git clone [URL]          # 既存リポジトリを丸ごと複製
03

Git の 3 つのエリアと、ファイルの状態

Git を理解する鍵は3 つの場所です。ファイルはこの間を行き来します。ここが腹に落ちれば、後のコマンドはほぼ「どこからどこへ動かすか」の話になります。

ファイルの状態は大きく 4 つ。untracked（まだ Git が見ていない新規）/ modified（変更済み・未ステージ）/ staged（次のコミットに含める）/ committed（履歴に確定）。git add は「この変更を次のコミットに入れる」という選択の操作だと捉えると腑に落ちます。

04

基本サイクル — status / add / commit / log

日々の 9 割はこの 4 つ。状態を見て → 選んで → 確定して → 振り返る。

$ git status                 # 今どのファイルがどの状態か
$ git add file.txt           # 特定ファイルをステージ
$ git add .                  # 変更を全部ステージ
$ git commit -m "説明を書く"   # ステージ済みを履歴に確定
$ git log --oneline --graph  # 履歴を1行＋グラフで
便利技

git commit -am "..." は「追跡済みファイルの add ＋ commit」をまとめて実行（新規 untracked ファイルは対象外）。git log --oneline は履歴をざっと眺めるのに必須です。

05

変更を見る — diff / show

git diff は「何を変えたか」を見せます。どこと比べるかで意味が変わります。

$ git diff             # 作業ツリー vs ステージ（未ステージの変更）
$ git diff --staged    # ステージ vs 直近コミット（次に commit される内容）
$ git diff main        # 今 vs main ブランチ
$ git show [commit]    # そのコミットの中身と差分

コミット前に git diff --staged で「これから確定する内容」を確認する習慣をつけると、余計なファイルの混入を防げます。

06

取り消し・やり直し（最重要・要注意）

初心者が最も混乱し、最も事故るゾーン。「どこを巻き戻すか」で道具が違います。落ち着いて選びましょう。

まだコミットしていない変更

$ git restore file.txt          # 作業ツリーの変更を破棄（編集を捨てる）
$ git restore --staged file.txt # ステージから降ろす（変更自体は残る）

直前のコミットを直す

$ git commit --amend -m "新しいメッセージ"  # 直前コミットを書き換え
$ git add 追加ファイル
$ git commit --amend --no-edit            # 内容だけ足してメッセージ据え置き

コミットを巻き戻す — reset の 3 モード

git reset は HEAD（今いる位置）を過去のコミットへ動かします。変更をどこまで巻き戻すかがモードで決まります。

コマンドHEADステージ作業ツリー
reset --soft HEAD~1戻す保持（staged のまま）保持
reset --mixed HEAD~1（既定）戻す戻す（未ステージに）保持
reset --hard HEAD~1戻す戻す破棄
危険

git reset --hard は作業中の変更を消します。実行前に git status で確認を。万一消しても、コミット済みのものなら後述の git reflog で救えることが多いです。

公開済みのコミットを打ち消す — revert

すでに push して他人と共有したコミットは、履歴を書き換える reset ではなく revert を使います。「打ち消すための新しいコミット」を積むので、履歴が壊れません。

$ git revert [commit]   # そのコミットを取り消す新コミットを作成

Part 2 — ブランチと統合

07

ブランチ

ブランチは履歴上の軽い「しおり（ポインタ）」。本線（main）に影響を与えずに作業を分け、後で統合します。HEAD は「今どのブランチ／コミットにいるか」を指す印です。

$ git branch                  # ブランチ一覧（* が現在地）
$ git switch -c feature/login # 新規ブランチを作って移動
$ git switch main             # main へ移動
$ git branch -d feature/login # ブランチ削除（統合済みのみ）
新旧コマンド

昔は git checkout が移動も作成も兼ねていましたが、今は役割を分けた git switch（ブランチ移動）と git restore（ファイル復元）が分かりやすく推奨です。古い記事の checkout もそのまま動きます。

08

マージとコンフリクト解決

ブランチの成果を統合するのが merge。本線に feature を取り込むには、取り込み先（main）に移動してから merge します。

$ git switch main
$ git merge feature/login

コンフリクト（衝突）の解決

同じ箇所を両ブランチで変えていると、Git は自動統合できずコンフリクトになります。該当ファイルに次のマーカーが入ります。

<<<<<<< HEAD
こちらは今のブランチ（main）の内容
=======
こちらは取り込もうとした feature の内容
>>>>>>> feature/login

解決手順はシンプル。① マーカーを手で編集し、正しい最終形に直す（マーカー行は全部消す）。② git add で「解決した」と印を付ける。③ git commit で確定。

$ # ファイルを編集してマーカーを消す
$ git add 解決したファイル
$ git commit                 # マージを完了
$ git merge --abort          # やっぱりマージをやめて元に戻す
09

リベース

rebase は、自分のコミットを別の起点の上に置き直す操作。マージが「合流点（マージコミット）を作る」のに対し、リベースは履歴を一直線に整えます。

マージ後（合流が残る）

      A---B---C feature
     /         \
D---E---F---G---M main

リベース後（一直線）

              A'--B'--C' feature
             /
D---E---F---G main
$ git switch feature
$ git rebase main            # main の先端に自分のコミットを乗せ直す
$ git rebase -i HEAD~3       # 直近3コミットを対話的に整理

対話的リベース（-i）では、各コミットに対し squash（まとめる）/ reword（メッセージ修正）/ drop（削除）/ 並べ替え、などができ、push 前にコミット履歴を綺麗に整えるのに使います。

黄金律

すでに push して共有したコミットは rebase しない。リベースは履歴（コミットの ID）を作り直すため、共有済みの履歴を書き換えると他人の手元と食い違い、大混乱します。リベースは「自分だけの・未 push の履歴」に限定するのが鉄則です。

merge か rebase か

合流の事実を残したい・共有ブランチ＝merge。履歴を直線で読みやすくしたい・手元の整理＝rebase。チームの方針に合わせるのが一番です。

Part 3 — リモートと共同作業

10

リモート — fetch / pull / push

GitHub などサーバー上のリポジトリがリモート。clone 元は慣習的に origin という名前が付きます。手元とリモートのやり取りは 3 つの動詞で覚えます。

コマンド意味
git fetchリモートの更新を取得だけ（手元の作業には未反映）
git pullfetch ＋ マージ（取得して今のブランチに統合）
git push手元のコミットをリモートへ送る
 
$ git remote -v                       # 登録済みリモートを確認
$ git fetch origin                    # 取得だけ（安全に差分確認できる）
$ git pull                            # 取得＋統合
$ git push -u origin feature/login    # 初回: 上流を設定して push
$ git push                            # 2回目以降はこれだけ

origin/main のようなリモート追跡ブランチは「リモートが最後にこうだった」という手元の記憶です。git fetch で更新され、git pull はこれを手元のブランチに統合します。迷ったら、まず fetch してから差分を見て統合するのが安全です。

11

共同作業の流れ（典型）

チーム開発の基本リズム。main は直接触らず、ブランチで作業して PR（プルリクエスト）でレビュー・統合します。

  1. git switch -c feature/xxx で作業ブランチを作る。
  2. コミットを重ねる（小さく意味のある単位で）。
  3. git push -u origin feature/xxx でリモートへ。
  4. GitHub 等でプルリクエストを作成 → レビュー → main にマージ。
  5. ローカルの main を最新化（git switch main → git pull）。
push が弾かれたら

「リモートが先に進んでいる」拒否は、先に取り込めば解決。git pull --rebase でリモートの更新の上に自分のコミットを乗せ直すと、履歴が直線的に保てて綺麗です。コンフリクトが出たら前述の手順で解決。

12

一時退避 — stash

「作業途中だけど、急いで別ブランチに移りたい」。そんなとき、変更を一時的に棚に上げて作業ツリーを綺麗にするのが stash。

$ git stash             # 変更を退避して綺麗な状態に
$ git stash list        # 退避したものの一覧
$ git stash pop         # 最新の退避を戻して棚から削除
$ git stash apply       # 戻すが棚には残す
13

タグ — リリースの目印

特定コミットに「v1.0.0」のような不変の名札を付けるのが tag。リリース地点の記録に使います。メッセージ付きの注釈タグが推奨です。

$ git tag -a v1.0.0 -m "最初の正式版"   # 注釈タグを作成
$ git tag                              # タグ一覧
$ git push origin v1.0.0               # タグをリモートへ（個別）
$ git push origin --tags               # 全タグをまとめて

Part 4 — 調査と救出

14

履歴を探す・救出する

原因調査と「やらかし」からの復活に効く道具たち。特に reflog はお守りです。

$ git log --oneline --graph --all   # 全ブランチを俯瞰
$ git blame file.txt                # 各行を「誰がいつ」変えたか
$ git bisect start                  # 二分探索でバグ混入コミットを特定
$ git reflog                        # HEAD の移動履歴（消えた地点も残る）
reflog は救命具

reset --hard やブランチ削除で「コミットを失った！」ときも、git reflog に直前までの HEAD 位置が残っています。そこの ID を見つけて git reset --hard [ID] や git switch -c 復活 [ID] でかなりの確率で救出できます。まず慌てず reflog。

15

.gitignore — 追跡しないものを宣言

ビルド成果物・依存フォルダ・秘密情報など、履歴に入れたくないものは .gitignore に書きます。プロジェクト最初に置くのが定石です。

# .gitignore の例
node_modules/
dist/
*.log
.env            # 秘密情報は絶対に追跡しない
.DS_Store
注意

すでにコミットしてしまったファイルは .gitignore に書いても追跡され続けます。その場合は git rm --cached [file] で追跡から外してからコミットします。秘密情報を一度 push したら、無効化・ローテーションが必要（履歴に残るため）。

16

困ったとき逆引き

「こうしたい」から引ける早見表。迷ったらここ。

やりたいことコマンド
直前のコミットメッセージを直すgit commit --amend
直前コミットに変更を足すgit add . ; git commit --amend --no-edit
ステージを取り消すgit restore --staged [file]
作業中の変更を捨てるgit restore [file]
直前コミットを取消（変更は残す）git reset --soft HEAD~1
公開済みコミットを打ち消すgit revert [commit]
作業を一時退避git stash → git stash pop
マージをやめるgit merge --abort
消したコミット/ブランチを救出git reflog → git reset/switch [ID]
間違って追跡したファイルを外すgit rm --cached [file]
17

ベストプラクティス

  • 小さく、意味のある単位でコミット。1 コミット＝1 つの論理的な変更。後で読みやすく、戻しやすい。
  • コミットメッセージは「何を・なぜ」。1 行目は命令形で簡潔に（例: Fix login redirect）、必要なら空行を挟んで本文で理由を。
  • ブランチ名は用途が分かる形に（feature/..., fix/...）。
  • main を直接触らない。作業はブランチ＋PR で。
  • 共有済み履歴は rebase しない（黄金律）。push 前の整理に留める。
  • push 前に git pull --rebase で最新を取り込み、履歴を直線に保つ。
  • 秘密情報はコミットしない。.env 等は最初に .gitignore へ。
  • 困ったらまず git status と git reflog。慌てて --hard しない。

最後に — Git の地図

Git の操作はほぼ全部、3 つのエリア（作業ツリー / ステージ / リポジトリ）とブランチ・リモートの間で、コミットという「点」をどう動かすかに還元できます。add/commit で刻み、branch/merge で分けて統合し、push/pull で共有し、困ったら reflog で救う。この地図さえ頭にあれば、未知のコマンドも「どこを動かすのか」で読み解けます。

本記事は Git の標準的な操作・概念を、初級から実務レベルまで一冊にまとめたものです。コマンドの細かな挙動は Git のバージョンで差があるため、手元では git help [コマンド] や 公式ドキュメント（git-scm.com）も併せて確認してください。
