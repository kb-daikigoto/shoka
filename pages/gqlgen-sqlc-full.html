gqlgen × sqlc 完全版
作る流れと動く流れを一気通貫で

GraphQL の API 層（gqlgen）と、PostgreSQL のデータ層（sqlc）を 1 本の todo アプリに統合します。どう組み立てるか（構築フロー）と、1 リクエストがどう走るか（実行フロー）を、コードとファイルの粒度で最後まで通します。

API gqlgen DB sqlc ＋ PostgreSQL 題材 todo アプリ module example.com/gqlgen-todos
01

これで作る完成形

クライアントが todos { text user { name } } を投げると、gqlgen がそれを受けてリゾルバを動かし、リゾルバの中で sqlc が PostgreSQL に SQL を投げ、行データが GraphQL の形に整って返る ―― そんな 端から端まで型安全な todo API を作ります。

あなたが書く
gqlgen 生成（API 層）
sqlc 生成（データ層）

このページではコードを必ずこの 3 色でラベル付けします。ピンク＝gqlgen 側、ティール＝sqlc 側、オレンジ＝あなたが書く部分です。

02

全体像 — 2 つの生成と、2 段の層

鍵は「2 つの独立したコード生成」と「2 段に重なる層」です。gqlgen は GraphQL スキーマから API 層を、sqlc は SQL からデータ層を、それぞれ別々に生成します。そして API 層が、その下のデータ層を呼びます。

なぜ噛み合うのか

どちらも「宣言を書く → 生成 → 使う」という同じ性格の道具で、ORM のような隠蔽がありません。担当する層が隣り合い、間にできる「リゾルバの中身」を sqlc のメソッド呼び出しで埋められる。だから DB のカラム型 → sqlc → 詰め替え → GraphQL モデル → gqlgen → API と、型が一直線につながります。

構築 ①gqlgen 側 — API 層を生成する
04

構築① gqlgen 側

1

プロジェクトと GraphQL スキーマ

$ mkdir gqlgen-todos && cd gqlgen-todos
$ go mod init example.com/gqlgen-todos
$ go get -tool github.com/99designs/gqlgen
$ go tool gqlgen init

graph/schema.graphqls を todo 用に書き換えます。

✍ あなたgraph/schema.graphqls
type Todo { id: ID!  text: String!  done: Boolean!  user: User! }
type User { id: ID!  name: String! }

type Query    { todos: [Todo!]! }
type Mutation { createTodo(input: NewTodo!): Todo! }
input NewTodo { text: String!  userId: String! }
2

user を「フィールドリゾルバ」にする

連携の主役は Todo.user。これをリゾルバにしておくと、「user が要求されたときだけ DB を引く」遅延ロードができます。gqlgen.yml を編集し、自作モデルを置きます。

✍ あなたgqlgen.yml（追記）
autobind:
  - "example.com/gqlgen-todos/graph/model"

models:
  Todo:
    fields:
      user:
        resolver: true   # ← user を穴（リゾルバ）にする
✍ あなたgraph/model/todo.go（新規）
package model

type Todo struct {
	ID     string `json:"id"`
	Text   string `json:"text"`
	Done   bool   `json:"done"`
	UserID string `json:"userId"`  // ← user 本体ではなく ID だけ持つ
	User   *User  `json:"user"`
}
$ go tool gqlgen generate   # Todos / CreateTodo / Todo.User の「穴」が生える
この時点で

graph/schema.resolvers.go に 3 つの穴（Todos / CreateTodo / User）が panic 付きで生成されています。中身は④で sqlc を使って埋めます。

構築 ②sqlc 側 — データ層を生成する
05

構築② sqlc 側

1

テーブルとクエリを SQL で書く

✍ あなたdb/schema.sql
CREATE TABLE users (
  id   text PRIMARY KEY,
  name text NOT NULL
);
CREATE TABLE todos (
  id      text    PRIMARY KEY,
  text    text    NOT NULL,
  done    boolean NOT NULL DEFAULT false,
  user_id text    NOT NULL REFERENCES users(id)
);
✍ あなたdb/query.sql
-- name: ListTodos :many
SELECT * FROM todos ORDER BY id;

-- name: CreateTodo :one
INSERT INTO todos (id, text, user_id)
VALUES ($1, $2, $3) RETURNING *;

-- name: GetUser :one
SELECT * FROM users WHERE id = $1;
2

sqlc.yaml を置いて生成

出力先を graph/db にして、gqlgen と同じツリーに同居させます。

✍ あなたsqlc.yaml
version: "2"
sql:
  - engine: "postgresql"
    queries: "db/query.sql"
    schema: "db/schema.sql"
    gen:
      go:
        package: "db"
        out: "graph/db"
        sql_package: "pgx/v5"
$ sqlc generate
$ go get github.com/jackc/pgx/v5
生えたもの

graph/db/ に db.Queries(メソッド集)、db.Todo / db.User(行の型)、db.CreateTodoParams が生成されました。全カラム NOT NULL なので、db.Todo は ID string / Text string / Done bool / UserID string と素直な型になります。

構築 ③配線 — 2 層をつなぐ
06

構築③ 配線

gqlgen の Resolver に、sqlc の入口 *db.Queries を持たせます。これが「上の層が下の層を呼ぶ」ための一本のパイプです。

✍ あなたgraph/resolver.go
package graph

import "example.com/gqlgen-todos/graph/db"

type Resolver struct {
	Queries *db.Queries   // ← sqlc の入口を保持
}

起動時に PostgreSQL への接続プールを作り、それを db.New(...) に渡して Resolver に注入します。生成済み server.go の &graph.Resolver{} を置き換えます。

✍ あなたserver.go（要点）
// import: "os", "github.com/jackc/pgx/v5/pgxpool",
//         "example.com/gqlgen-todos/graph/db"

pool, err := pgxpool.New(context.Background(), os.Getenv("DATABASE_URL"))
if err != nil { log.Fatal(err) }
defer pool.Close()

resolver := &graph.Resolver{Queries: db.New(pool)}  // ← 注入
// 以降は生成コードのまま: ...Config{Resolvers: resolver}...
構築 ④リゾルバ実装 — 穴の中で sqlc を呼ぶ
07

構築④ リゾルバ実装

連携の心臓部です。各リゾルバ（gqlgen の穴）の中で sqlc のメソッドを呼び、返ってきた sqlc の行（db.Todo）を GraphQL モデル（model.Todo）へ詰め替えます。

✍ あなたgraph/schema.resolvers.go
// 一覧: sqlc で取り、model に詰め替える
func (r *queryResolver) Todos(ctx context.Context) ([]*model.Todo, error) {
	rows, err := r.Queries.ListTodos(ctx)        // ← sqlc
	if err != nil { return nil, err }
	todos := make([]*model.Todo, len(rows))
	for i, row := range rows {
		todos[i] = &model.Todo{ID: row.ID, Text: row.Text, Done: row.Done, UserID: row.UserID}
	}
	return todos, nil
}

// 作成: sqlc の CreateTodo を呼ぶ
func (r *mutationResolver) CreateTodo(ctx context.Context, input model.NewTodo) (*model.Todo, error) {
	n, _ := rand.Int(rand.Reader, big.NewInt(100000))
	row, err := r.Queries.CreateTodo(ctx, db.CreateTodoParams{
		ID: fmt.Sprintf("T%d", n), Text: input.Text, UserID: input.UserID,
	})
	if err != nil { return nil, err }
	return &model.Todo{ID: row.ID, Text: row.Text, Done: row.Done, UserID: row.UserID}, nil
}

// user フィールド: 要求されたときだけ呼ばれる（obj = その todo）
func (r *todoResolver) User(ctx context.Context, obj *model.Todo) (*model.User, error) {
	row, err := r.Queries.GetUser(ctx, obj.UserID)   // ← sqlc（ここが後で N+1 に）
	if err != nil { return nil, err }
	return &model.User{ID: row.ID, Name: row.Name}, nil
}
なぜ詰め替えるのか

db.Todo(sqlc が DB から生成)と model.Todo(gqlgen が GraphQL から生成)は別の型。だから小さなマッピングを挟みます。NULL 可カラムは sqlc 側が pgtype.Text 等になるので、その場合は *string への変換を足します（今回は全 NOT NULL なので不要）。

構築⑤ 起動

実 DB に 2 テーブルを作り（db/schema.sql を流す。golang-migrate などが定番）、接続情報を環境変数に入れて起動します。

$ export DATABASE_URL="postgres://user:pass@localhost:5432/todos"
$ go run server.go
# http://localhost:8080 の Playground でクエリを実行
▶ Playground投げてみるクエリ
mutation { createTodo(input: { text: "牛乳を買う", userId: "1" }) { id text } }

query { todos { text done user { name } } }

見た目は前回までと同じですが、裏では gqlgen のリゾルバが sqlc 経由で PostgreSQL を読み書きしています。次の章で、その「裏」を 1 ステップずつ追います。

09

実行フロー — 1 リクエストを一気通貫で追う

いよいよ本題。query { todos { text user { name } } } を 1 本投げたとき、5 つの持ち場（クライアント / gqlgen / あなたの Resolver / sqlc / PostgreSQL）をデータがどう往復するか。ピンク＝gqlgen があなたを呼ぶ瞬間、ティール＝あなたが sqlc を呼ぶ瞬間です。

ステップ解説

  • ① クライアントの POST /query を gqlgen の handler が受け、クエリをパース・検証する。
  • ② 生成エンジンが _Query_todos 経由で、あなたの queryResolver.Todos を呼ぶ。
  • ③〜⑥ あなたの Todos が sqlc の ListTodos を呼び、sqlc が SELECT * FROM todos を実行。行（[]db.Todo）が戻る。
  • ⑦ あなたが db.Todo を model.Todo に詰め替えて gqlgen に返す。
  • ⑧ gqlgen が各 todo を展開し、user が要求されているので あなたの todoResolver.User(obj) を呼ぶ。
  • ⑨〜⑫ あなたの User が sqlc の GetUser(obj.UserID) を呼び、user 行を取り、model.User へ詰め替える。
  • ⑬⑭ 全フィールドが揃ったら gqlgen が JSON に整え、handler が HTTP で返す。
連携の要点はこの 2 種類の矢印

ピンク（②⑧）= gqlgen が「あなたのリゾルバ」を呼ぶ。ティール（③⑨）= あなたのリゾルバが「sqlc」を呼ぶ。この 2 段のバトンが連携の全てです。あなたのコードは、その中継地点で sqlc を呼んで詰め替えるだけ。

10

N+1 と dataloader（連携での解決）

実行フローの図のピンク帯に気づいたはず。user の解決（⑧〜⑫）は todo の件数ぶん繰り返されます。100 件なら ListTodos 1 回 ＋ GetUser 100 回 ＝ 101 回の SQL。これが N+1 問題です。

解決は、gqlgen の dataloader と sqlc の まとめ取りクエリの合わせ技。まず sqlc に「複数 ID を一括取得」する SQL を 1 本足します。

✍ あなたdb/query.sql（追記）
-- name: ListUsersByIDs :many
SELECT * FROM users WHERE id = ANY($1);   -- 複数 ID をまとめて 1 回で

そして dataloader が「同じ実行サイクルで溜まった ID」をまとめ、この 1 本を呼ぶようにします（概念コード）。

✍ あなたgraph/loaders.go（概念）
// dataloader の「バッチ関数」: 溜まった userIDs を 1 回の SQL で解決
func (l *Loaders) batchUsers(ctx context.Context, userIDs []string) []*model.User {
	rows, _ := l.Queries.ListUsersByIDs(ctx, userIDs)  // ← ANY($1) で一括
	// rows を userIDs の順に並べ替えて返す（id→User の対応付け）
	...
}

// リゾルバ側は GetUser 直呼びをやめ、loader 経由にする
func (r *todoResolver) User(ctx context.Context, obj *model.Todo) (*model.User, error) {
	return loaders.For(ctx).UserByID.Load(ctx, obj.UserID)  // ← 溜めてまとめる
}
きれいな分業

gqlgen 側の dataloader が「いつ束ねるか」を、sqlc 側の ANY($1) クエリが「束ねた SQL」を担当します。結果、SQL は 101 回から 2 回（ListTodos ＋ ListUsersByIDs）に激減。これが連携の実戦的な到達点です。

補足

dataloader の実装ライブラリは複数あり（gqlgen 公式の例、dataloaden、dataloadgen など）バージョンで作法が変わります。上は仕組みを示す概念コードなので、実装時は gqlgen 公式の dataloaders ガイドに合わせてください。

一望すると

1 リクエストの旅は handler → gqlgen → あなたの Todos → sqlc ListTodos → Postgres → 詰替 → gqlgen → あなたの User → sqlc GetUser → Postgres → 詰替 → marshal → handler。

あなたが書くのは「2 つのスキーマ＋2 つの設定＋リゾルバの中身（sqlc を呼んで詰め替える）」だけ。残りは gqlgen と sqlc が生成し、その境目はいつも 「gqlgen があなたを呼ぶ」→「あなたが sqlc を呼ぶ」の 2 段バトン。これが連携版の全体像です。
