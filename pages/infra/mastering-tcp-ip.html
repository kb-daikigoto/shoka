マスタリング TCP/IP 入門編
階層で読み解くネットワークの地図

「インターネットでデータが届く」——当たり前に見えて、その裏では膨大な約束事が連携しています。本書の核心は「役割を階層に分けて考える」こと。Ethernet から IP、TCP、DNS、HTTP まで、各層が何を担い、どう積み重なって1つの通信になるのか。この記事では、その層の地図と、層をまたぐ1本の流れを、初学者目線で自分の言葉に置き換えて再構成します。

著者 竹下隆史 ほか 出版 オーム社（第6版） 分野 ネットワーク / TCP-IP 対象 ネットワーク初学者〜中級
01

一言でいうと

ネットワークの全体像は複雑です。それを正気で扱うための最大の発明が「階層化（レイヤリング）」。通信という大仕事を、役割ごとの層に分割し、各層は「すぐ下の層」だけに頼る。物理的な信号の心配はデータリンク層に任せ、IPは「どの機械へ届けるか」だけを考える。TCPは「確実に届いたか」だけを見る。関心の分離そのものです。

本書を貫く問いは2つだけ。① 各層は何を約束するのか、② 層と層はどう受け渡すのか。この2つを軸に読むと、無数のプロトコル名が「地図上の役割」として整理されていきます。

この記事のゴール

「ping が通らない」「名前解決できない」を、どの層の問題かで切り分けられるようになること。プロトコルの暗記ではなく、層の地図を持ち帰ります。

02
Part I · 地図を持つ

階層という考え方

歴史的な参照モデルが OSI 7階層、実際に動いているのが TCP/IP 4階層。試験では7層、現場では4層で語られることが多い、と割り切って対応を押さえれば十分です。大事なのは層の数より、「各層がヘッダを付け足しながらデータを包んでいく」という動き——カプセル化です。

カプセル化のおかげで、各層は中身を気にしません。IPは「TCPのセグメントだろうがUDPだろうが、ただのデータとして運ぶ」。だから新しいアプリ層プロトコルが増えても、下の層は無改修。層の独立性こそ、インターネットが拡張し続けられた理由です。各層のデータの呼び名（セグメント/パケット/フレーム）も、この「どの層で包まれた状態か」を表しています。

なぜ層に分けるのか

もし全部を1枚岩で作ったら、無線に変えるたび、アプリを足すたびに全体を作り直すことになります。層という「契約された境界」を引けば、各層は内部を自由に進化させられる。Wi-Fi も 5G も、IP より下を差し替えただけ——上の層は何も知らずに動き続けます。

03
Part II · 下から積む

データリンク層 — Ethernet

一番下、同じネットワーク内の「隣の機器」へ信号を届けるのがデータリンク層。主役は Ethernet で、機器を識別するのが MACアドレス——ネットワークカードに焼き付けられた、世界で重複しない48ビットの番号です。この層のデータの単位をフレームと呼びます。

同じネットワーク内の橋渡しをするのが スイッチ。スイッチは「どのポートの先に、どのMACがいるか」を学習し、宛先MAC宛のフレームを必要なポートだけに送ります（賢い仕分け）。昔のハブが全ポートにばらまいていたのとは対照的です。

$ ip link              # NICとMACアドレスを確認
2: eth0: <BROADCAST,MULTICAST,UP> ...
    link/ether 02:42:ac:11:00:02   # ← これがMACアドレス

$ ip neigh             # 近隣(同一LAN)のIP↔MAC対応表（ARPキャッシュ）
192.168.1.1 dev eth0 lladdr aa:bb:cc:dd:ee:ff REACHABLE
MAC と IP の役割分担

MACは「同じネットワーク内での宛名」、IPは「世界中どこへでもの宛名」。手紙でいえば、IPが最終的な住所、MACが「次に手渡す相手」。実際の配送は、各区間で MAC を次々に書き換えながら、IP の住所を目指して進みます（§05のARPがこの橋渡し役）。

04
Part II · 下から積む

ネットワーク層 — IPアドレス

ネットワークをまたいで「世界中の目的の機械まで運ぶ」のがIP。住所が IPアドレスです。IPv4 は 192.168.1.10 のように32ビットを8ビットずつ4つに区切って表記します（枯渇対策に128ビットの IPv6 もある）。この層のデータの単位がパケット。

肝は、IPアドレスが「ネットワーク部」と「ホスト部」に分かれていること。郵便番号（どの地域か）と番地（その中の誰か）に似ています。どこまでがネットワーク部かを示すのが サブネットマスクで、/24 のような CIDR表記で「上位24ビットがネットワーク部」と表します。

この区切りが、通信判断の分岐点になります。送信元は宛先IPを見て、「相手は自分と同じネットワークか？」を計算する。同じなら直接（データリンク層で）渡し、違えば「外への出口」＝デフォルトゲートウェイ（ルータ）へ託します。

$ ip addr              # 自分のIPアドレスとサブネット
    inet 192.168.1.10/24 brd 192.168.1.255 scope global eth0
$ ip route             # 経路表。default が外への出口
default via 192.168.1.1 dev eth0        # ← デフォルトゲートウェイ
192.168.1.0/24 dev eth0 proto kernel scope link
05
Part II · 下から積む

経路を決める — ルーティング

相手が別のネットワークにいるとき、パケットはルータからルータへバケツリレーで運ばれます。各ルータは自分のルーティングテーブル（経路表）を見て、「この宛先なら次はあのルータへ」と次の1ホップだけを決めて転送する。全体経路を誰かが把握しているわけではなく、各ルータの局所判断の積み重ねで世界中へ届きます。

この「次のホップ」の連なりは traceroute で実際に覗けます。TTL（生存時間）——パケットがルータを通るたびに1ずつ減る値——を巧みに使い、経路上の各ルータを1つずつ炙り出す仕掛けです。TTLが0になったパケットは破棄され、ループでパケットが永遠に彷徨うのを防ぎます。

$ traceroute example.com   # 宛先までに通るルータを順に表示
 1  192.168.1.1      0.5 ms     # 最初の1ホップ＝自宅ルータ
 2  10.0.0.1         3.2 ms     # プロバイダ
 3  ...              12.1 ms    # さらに上流へ
06
Part III · 縁の下

IPを支える働き者

IPだけでは通信は完結しません。表に出ないが不可欠な補助プロトコルたちがいます。本書が丁寧に扱うこの4人を押さえると、ネットワークが「実際に動く」像が結びます。

名前解決(L2)

ARP

「このIPの相手のMACは？」を同一LANに問い合わせる。IPからMACへの橋渡し。これが無いとフレームを作れない。

通知・診断

ICMP

到達不能やTTL切れなどを知らせる連絡係。ping や traceroute はこれを利用している。

自動設定

DHCP

つないだだけでIP・ゲートウェイ・DNSを自動で配る。手動設定の手間を消す。

アドレス変換

NAT

家庭内のプライベートIPを1つのグローバルIPに変換。IPv4枯渇をしのぐ立役者。

たとえば ping を1回打つだけで、複数の層が連携します。宛先が同一LANなら ARP でMACを引き、ICMP のechoを IP に載せ、Ethernet フレームで送出する。「ping が通る／通らない」は、この連鎖のどこが切れているかを示す最初の手がかりです。

$ ping -c 3 192.168.1.1     # ICMPで疎通確認
64 bytes from 192.168.1.1: icmp_seq=1 ttl=64 time=0.4 ms
$ ping example.com          # 名前で打つと、まずDNS解決→IPへping
切り分けの順番

つながらない時は下の層から。① ip addr（自分にIPはあるか）→ ② ping ゲートウェイ（LAN内は届くか）→ ③ ping 8.8.8.8（外へIPで届くか）→ ④ ping 名前（DNSは効くか）。④だけ失敗なら、ネットワークではなく名前解決の問題と切り分けられます。

Part IV · 確実に届ける

トランスポート層 — TCP/UDP

IPは「届けようと努力する」だけで、確実さは保証しません（途中で消えても知らんぷり＝ベストエフォート）。その上で「アプリ同士の会話」を成立させるのがトランスポート層。性格の異なる2つがあります。

観点TCPUDP
信頼性あり（到達確認・再送・順序保証）なし（投げっぱなし）
接続コネクション型（事前に握手）コネクションレス
速さ・軽さその分オーバーヘッドあり軽量・低遅延
向く用途Web・メール・ファイル転送動画/音声・DNS・ゲーム

もう一つの主役がポート番号。1台の機械では複数のアプリが同時に通信します。IPが「どの機械か」を決め、ポートが「その中のどのアプリか」を決める。Webは80/443、DNSは53…といったウェルノウンポートで、サービスの入口が決まっています。「IP:ポート」の組が、通信相手を一意に指す宛名です。

$ ss -tlnp           # 待ち受け中(LISTEN)のTCPポート一覧
State   Local Address:Port
LISTEN  0.0.0.0:22          # SSH
LISTEN  0.0.0.0:443         # HTTPS
なぜ2種類あるのか

「正確さ」と「速さ」はしばしば両立しません。1フレームの欠けも許せないファイル転送はTCP。多少の欠落より遅延ゼロが大事な音声・動画はUDP。用途に応じて代償を選べるよう、あえて両方を用意しているのです。最近のHTTP/3が（TCPでなく）UDPの上に信頼性を再実装したのも、この選択の延長です。

Part IV · 確実に届ける

TCPの信頼性

TCPは「信頼できないIP」の上に、信頼できる通信を組み立て直します。その出発点が 3ウェイハンドシェイク——通信開始前に、互いの準備が整ったことを3往復（厳密には3メッセージ）で確認する儀式です。

接続後の信頼性は、いくつかの仕組みの合わせ技で支えられます。それぞれが「IPの不確かさ」のどこを埋めるのかを見ると、設計の意図が見えてきます。

仕組み埋める穴やること
確認応答(ACK)と再送パケットが消える届いたらACKを返す。一定時間来なければ再送
シーケンス番号順序が入れ替わる各バイトに通し番号。受信側で正しい順に並べ直す
フロー制御受信側があふれる「今これだけ受け取れる」(ウィンドウ)を相手に伝える
輻輳(ふくそう)制御網が混んで詰まる送信ペースを徐々に上げ、損失を感じたら落とす

とりわけ輻輳制御は、ネットワーク全体への思いやりです。皆が遠慮なく全力で送れば回線はパンクする。だからTCPは控えめに送り始め、うまく届く間は少しずつ増やし、損失（混雑のサイン）を感じたら一気に控える。この自制が、共有された網を破綻させずに保っています。

「遅い」の原因は層をまたぐ

体感の遅さは、帯域だけでなく往復遅延(RTT)に強く効きます。3ウェイハンドシェイクも、ACKを待つ仕組みも、1往復の時間に縛られるから。遠いサーバが遅いのは「細い」のではなく「遠い（往復に時間がかかる）」ことが多い。CDNでサーバを近くに置く意義は、ここにあります。

Part V · 人の世界へ

アプリケーション層 — DNS/HTTP

最上層は、人やアプリが直接使う言葉。土台が DNS と HTTP です。

DNSは、覚えにくいIPアドレスの代わりにexample.com のような名前を使えるようにする「電話帳」。実際は1台の巨大な帳簿ではなく、世界中のサーバが役割を分担する階層的な分散システムです。「.com を管理する所」「example.com を管理する所」と順にたどって答えにたどり着きます。

$ dig example.com +short    # 名前 → IPアドレスを引く
93.184.216.34
$ dig example.com           # 詳細（どのサーバが答えたか等）

HTTPは、Webの会話を担うリクエスト／レスポンス型のプロトコル。クライアントが「これをくれ（GET）」と要求し、サーバが状態コード（200=OK、404=見つからない…）とともに返す。現在は通信を暗号化する TLS を被せた HTTPS が標準です。中身が盗み見・改ざんされないよう、TCP接続の確立後にTLSの鍵交換を行います。

$ curl -v https://example.com   # 通信の中身を冗長表示
* Connected to example.com (93.184.216.34) port 443   # TCP確立
* TLS handshake ...                                    # TLS鍵交換
> GET / HTTP/2                                         # リクエスト
< HTTP/2 200                                           # レスポンス(200=OK)
10
Part V · 人の世界へ

通して見る — URLを開くと

最後に、ここまでの全層を1本の流れで貫きます。ブラウザに https://example.com と打ってEnter——その瞬間に、各層がバケツリレーで動き出します。

この流れを知っていると、トラブルの切り分けが一気に速くなります。「ページが出ない」とき、dig で①、ping/traceroute で②、curl -v で③④⑤——と段階を当てれば、どの層で止まっているかがたいてい特定できます。ネットワークは魔法ではなく、層の積み重ねです。

つながった

DNS（名前）→ IP（住所）→ ルーティング（道）→ ARP（次の手渡し先）→ TCP（確実な配送）→ TLS（安全）→ HTTP（会話）。本記事の全要素が、URL1つで合流します。各層の役割を覚えていれば、未知の不具合も「どの層か」で攻められる——それがマスタリングの第一歩です。

一言で

通信とは、カプセル化で包んで下ろし、宛先で解いて上げること。IPが住所で運び、TCPが確実さを足し、DNSが名前を翻訳し、HTTPが会話する。役割を層で分けたからこそ、インターネットはここまで拡張できた。層の地図を手に、数字とログから原因へ最短でたどり着けます。
