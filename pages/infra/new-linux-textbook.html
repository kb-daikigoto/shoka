Linux · Command Line
Getting comfortable at the prompt

新しい Linuxの教科書
コマンドラインに親しむ最初の一冊

GUIに慣れた手を、いったんキーボードに戻す。本書は「コマンドラインを怖がらずに使えるようになる」ことだけを丁寧に目指した入門書です。シェルの仕組み、ファイルとパス、パイプ、パーミッション、vim、そしてシェルスクリプトまで。この記事では、「なぜそのコマンドでそう動くのか」を初学者目線で自分の言葉に置き換えて再構成します。

01

一言でいうと

本書の姿勢は「暗記より、仕組みと手触り」。コマンドを丸暗記するのではなく、シェルがどう入力を解釈し、どう実行するかの感覚をつかめば、知らないコマンドにも応用が効きます。土台になる考え方は、たった3つです。

考え方1

すべてはコマンド

「コマンド＋オプション＋引数」という同じ形の繰り返し。形を覚えれば、新しいコマンドも怖くない。

考え方2

小さな道具を組み合わせる

1つのことだけ上手にやる小道具を、パイプでつないで大きな仕事にする。これがUNIX流。

考え方3

すべてはファイル

設定も、デバイスも、入出力も「ファイル」として同じ作法で扱える。だから道具が使い回せる。

この3つを意識しながら読むと、個々のコマンドが「ばらばらの呪文」ではなく、一貫した設計の現れに見えてきます。それが、コマンドラインに親しむ最短ルートです。

この記事のゴール

ターミナルを開いて固まらないこと。ls や cd から始めて、パイプでコマンドをつなぎ、簡単なスクリプトを書くところまで、地続きで到達できる地図を描きます。

02
Part I · 入り口

シェルとターミナル

まず用語を1つだけ整理します。ターミナルは文字を打って表示する「窓」、シェルはその窓の奥で入力を読み取り、解釈し、プログラムを起動する「通訳」です。Linuxで広く使われるシェルが bash（最近は zsh も）。プロンプト（$）が出ていれば、シェルが入力を待っている合図です。

コマンドの形はいつも同じ。コマンド名・オプション・引数の3要素です。

困ったら頼れる相棒が マニュアル。man ls でそのコマンドの全オプションが読めますし、多くのコマンドは --help で要約を出します。「分からなければ man を引く」を習慣にすると、暗記の負担がぐっと減ります。

$ ls --help        # 使い方の要約
$ man ls            # 詳しいマニュアル（q で終了）
$ type cd           # そのコマンドの正体を表示
最初に効くコツ

Tabキーで補完、↑キーで履歴。この2つを使うだけで、タイプ量と打ち間違いが激減します。長いファイル名は途中まで打って Tab。先人のコマンドは ↑ で呼び戻す。手に馴染ませる価値が一番高い操作です。

03
Part I · 入り口

ファイルシステムとパス

Linuxのファイルは、一番上の ルート（/）から枝分かれする1本の木になっています。Windowsのような C: ドライブの区切りはなく、すべてが / の下に連なる。場所を表す住所がパスです。

パスには2種類あります。絶対パスは / から始まる完全な住所、相対パスは「今いるディレクトリ」を起点にした道順です。今いる場所を表す .、1つ上を表す ..、自分のホームを表す ~ を覚えると、移動が一気に楽になります。

$ pwd                 # 今いる場所を表示（現在地の確認）
/home/user
$ cd /etc             # 絶対パスで移動
$ cd ..               # 1つ上のディレクトリへ
$ cd ~                # ホームディレクトリへ一発で戻る
$ ls -la              # 隠しファイル(.で始まる)も含めて詳細表示
迷子になったら

pwd で現在地、cd（引数なし）でホームへ即帰宅。相対パスで操作する前に「自分が今どこにいるか」を確認する癖をつけると、思わぬ場所のファイルを触る事故を防げます。

Part II · 基本操作

ファイル操作の基本

作る・写す・動かす・消す。日々の操作はこの数個に集約されます。どれも「コマンド＋対象」の形に従うだけです。

やりたいことコマンドひとこと
ディレクトリを作るmkdir work-p で途中の階層もまとめて作成
空ファイルを作るtouch memo.txt既存なら更新日時だけ変わる
コピーcp a.txt b.txtディレクトリごとは -r
移動・改名mv a.txt sub/同じ場所へ別名で＝リネーム
削除rm a.txtディレクトリは -r。ゴミ箱なし

強力なのがワイルドカード。* は「任意の文字列」、? は「任意の1文字」にマッチし、シェルが実行前にファイル名へ展開してくれます。「2024で始まる .log を全部」といった一括操作が一行で書けます。

$ ls *.log              # 拡張子 .log のファイルすべて
$ cp 2024-*.txt backup/  # 2024- で始まる .txt をまとめてコピー
$ mkdir -p a/b/c        # 途中の a, b も自動で作る
rm は取り消せない

Linuxの rm にゴミ箱はありません。特に rm -rf（再帰・強制）はディレクトリを丸ごと一瞬で消します。実行前に「対象が合っているか」を ls で確かめる、ワイルドカード削除は -i（確認付き）で慎重に。rm -rf / のような事故は一生ものの後悔になります。

Part II · 基本操作

標準入出力・パイプ・リダイレクト

ここがコマンドラインの心臓部です。多くのコマンドは、決まった3つの口を持っています。データが入ってくる標準入力（stdin）、結果を出す標準出力（stdout）、エラーを出す標準エラー出力（stderr）。この口をつなぎ替えたり、ファイルへ向けたりできるのが、UNIXの強さの源泉です。

パイプ | は、前のコマンドの出力を次のコマンドの入力へ直結します。リダイレクトは出力先をファイルへ向ける道具で、> は上書き、>> は追記。入力をファイルから取るなら < です。

$ ls -l > list.txt        # 結果をファイルに保存（上書き）
$ echo "done" >> log.txt    # 末尾に追記
$ cat access.log | grep 404 | wc -l   # 404を含む行数を数える
$ command 2> error.log     # エラー(stderr)だけ別ファイルへ
なぜこの設計が強いのか

各コマンドが「1つのことだけ上手にやる」小道具に徹し、出力をただのテキストで渡すから、自由に組み合わせられます。専用の巨大ツールを作る代わりに、小道具をつないで目的を達成する——これがUNIX哲学。覚える対象が減り、応用が無限に広がります。

Part II · 基本操作

テキスト処理と grep

Linuxの設定もログも、多くはテキストです。だからテキストを「見る・探す・切り出す」道具が日々効きます。中心にいるのが grep——指定したパターンを含む行だけを抜き出すフィルタです。

道具役割例
cat / less中身を表示（less はスクロール閲覧）less big.log
head / tail先頭/末尾の数行。tail -f で追従tail -f app.log
grepパターンを含む行を抽出grep -i error log
sort / uniq並べ替え / 重複の集約sort f | uniq -c
wc行数・単語数・文字数を数えるwc -l f

grep は正規表現でパターンを表現できます。最初は ^（行頭）、$（行末）、.（任意の1文字）、*（直前の繰り返し）だけ知っていれば十分。-i（大小無視）・-r（再帰検索）・-n（行番号）が日常の三種の神器です。

$ grep -rn "TODO" src/      # src以下を再帰検索し行番号付きで表示
$ grep "^2024" access.log    # 行頭が 2024 の行だけ
$ tail -f app.log | grep -i error   # 流れるログからerrorだけ追う
組み合わせて使う

「アクセスの多いIP上位5件」のような集計も、パイプの定石で書けます：cut で列を切り出し → sort → uniq -c で集計 → sort -rn で多い順 → head -5。一つひとつは単純な道具なのに、つなぐだけで分析になります。

Part III · 編集と権限

vim 入門

サーバ上ではGUIエディタが使えないことも多く、vim が最後の頼みになります。最初の関門は「モードがある」こと。文字を打つノーマルモード（操作・移動）とインサートモード（入力）が分かれていて、これを知らないと「文字が打てない」「抜けられない」と詰みます。

操作キー意味
入力開始iカーソル位置から挿入（インサートへ）
戻るEscノーマルモードへ。迷ったらまず Esc
保存して終了:wqwrite & quit
保存せず終了:q!変更を捨てて強制終了
取り消し / やり直しu / Ctrl-rundo / redo
「抜けられない」を脱出

何が起きているか分からなくなったら、まず Esc を数回押してノーマルモードに戻る。それから保存せず逃げたいなら :q!、保存して出たいなら :wq。この2つの呪文を覚えておけば、どんな状態からでも脱出できます。

Part III · 編集と権限

パーミッションと sudo

Linuxは複数人で1台を共有する前提で生まれました。だから各ファイルに「誰が・何をできるか」の権限が付きます。対象は所有者(u)・グループ(g)・その他(o)の3者、操作は読み(r)・書き(w)・実行(x)の3種。ls -l の左端の文字列が、まさにこの権限表です。

権限を変えるのが chmod、所有者を変えるのが chown。スクリプトを実行可能にする chmod +x は頻出です。そして、システム全体に関わる操作には管理者(root)権限が要る。普段は一般ユーザーで作業し、必要なときだけ sudo で一時的に管理者として実行するのが鉄則です。

$ chmod +x deploy.sh        # 実行権限を付ける
$ chmod 644 config.txt      # 所有者は読み書き、他は読みのみ
$ sudo systemctl restart nginx   # 管理者権限が要る操作だけ sudo
sudo は慎重に

sudo はカーネルの安全装置を一時的に飛び越える行為です。意味の分からないコマンドを sudo 付きで実行しない、chmod 777（誰でも何でも可）を安易に使わない。「動いたから」で権限を緩めると、セキュリティの穴になります。最小権限が原則です。

Part IV · 動かす・束ねる

プロセスとジョブ管理

実行中のプログラムはプロセスとして動き、それぞれに番号（PID）が振られます。今何が動いているかを見るのが ps や top、止めるのが kill。kill は「殺す」というより「シグナル（合図）を送る」コマンドで、丁寧に終了を促す TERM と、強制終了の KILL（-9） があります。

$ ps aux | grep nginx     # nginx のプロセスを探す
$ top                      # CPU/メモリ使用量をリアルタイム表示(q で終了)
$ kill 1234                # PID 1234 に終了シグナル(TERM)を送る
$ kill -9 1234             # 効かない時の最終手段(強制終了)

もう一つ便利なのがジョブ制御。コマンドの末尾に & を付けるとバックグラウンドで実行され、プロンプトがすぐ戻ります。実行中のコマンドは Ctrl-z で一時停止、bg で背面へ、fg で前面に戻せます。

ログアウトしても止めたくない

SSH接続が切れると、その上で動いていたプロセスは巻き添えで止まりがち。長時間の処理は nohup や、本書も触れる tmux／screen といった端末を切り離せる道具の上で動かすと、接続が切れても走り続けます。

Part IV · 動かす・束ねる

シェルをカスタマイズする

毎回同じ長いコマンドを打つのは退屈です。シェルは自分好みに育てられます。鍵になるのが環境変数とエイリアス。とりわけ大事なのが PATH——コマンドを名前で打てるのは、シェルが PATH に並ぶディレクトリを順に探して実行ファイルを見つけてくれるからです。

$ echo $PATH                    # コマンドを探す場所の一覧（: 区切り）
/usr/local/bin:/usr/bin:/bin
$ which ls                      # ls の実体がどこにあるか
/usr/bin/ls

# よく使う操作に短い別名を付ける
$ alias ll='ls -la'
# 自分のスクリプト置き場を PATH に足す
$ export PATH="$HOME/bin:$PATH"

これらを毎回有効にするには、設定ファイル（bashなら ~/.bashrc、zshなら ~/.zshrc）に書いておきます。シェルの起動時に読み込まれるので、次回から自動で効きます。history で過去のコマンドを振り返り、Ctrl-r で履歴をインクリメンタル検索できるのも、覚えると手放せません。

仕組み役割例
環境変数シェルやプログラムへ渡す設定値export EDITOR=vim
PATHコマンドを探すディレクトリの並びecho $PATH
エイリアスコマンドの別名・短縮alias g=git
設定ファイル起動時に自動で読み込む~/.bashrc
なぜ PATH を知ると世界が広がるか

「command not found」の正体は、たいていPATHに実行ファイルの場所が入っていないだけ。インストールしたツールが使えない時、PATHを疑えれば自力で解決できます。コマンドが「名前で呼べる」魔法の裏に、この素朴な仕組みがあると知るのが第一歩です。

Part IV · 動かす・束ねる

シェルスクリプト入門

同じコマンド列を繰り返すなら、ファイルにまとめて1つの命令にする——それがシェルスクリプトです。特別な言語ではなく、ターミナルで打つコマンドを、そのまま並べたもの。そこに変数・条件分岐・繰り返しを足せば、立派な自動化になります。

#!/bin/bash              # shebang：このスクリプトを動かす実行系
set -e                   # エラーが出たら途中で止める（安全策）

dir="backup"            # 変数（= の前後にスペースを入れない）
mkdir -p "$dir"

# 引数で受け取ったファイルを順にコピー（$1, $2... が引数）
for f in "$@"; do
  if [ -f "$f" ]; then      # ファイルが存在すれば
    cp "$f" "$dir/"
    echo "copied: $f"
  fi
done

動かすには実行権限を付けて呼び出します（§08の chmod +x が効いてきます）。引数は $1 $2 …、全部まとめては $@ で受け取れます。

$ chmod +x backup.sh
$ ./backup.sh a.txt b.txt    # a.txt, b.txt が $@ に入る
copied: a.txt
copied: b.txt
スクリプトの落とし穴

変数は必ず "$var" とダブルクォートで囲む。囲まないと、空白を含むファイル名やスペースで思わぬ分割が起き、事故になります。冒頭に set -e（エラーで停止）を置くのも定番。「動いた」だけで満足せず、変な入力でも壊れないかを一度試すのがコツです。
