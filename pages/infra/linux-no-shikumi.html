01

一言でいうと

本書のメッセージは一つ、「OSの挙動は、暗記するものではなく測って腑に落とすもの」。CPUは限られ、メモリは有限、ディスクは遅い——この3つの制約を、カーネルがいかに賢く隠して「無限にあるかのように」見せているか。それを実験で確かめていく構成です。

地図として、登場人物を3層で捉えると見通しが良くなります。アプリ（プロセス）が要求を出し、カーネルがハードウェアを抽象化して仲介し、その下にCPU・メモリ・ストレージという物理がある。本書の各章は、この層のどこかを拡大して「実際どう動くか」を見せるものです。

この記事のゴール

top や sar の数字が「なぜその値になるのか」を説明できるようになること。性能の不調を、勘ではなくしくみから推理できる土台を作ります。

02
Part I · 土台

カーネルとシステムコール

CPUには2つの動作モードがあります。アプリが動くユーザモードと、ハードウェアを直接触れるカーネルモード。アプリはハードウェアに勝手に触れません。ファイルを読むのもメモリを増やすのも、必ずシステムコールという「正式な窓口」を通してカーネルに依頼します。これが、1つのアプリの暴走が全体を巻き込まないための仕切りです。

この「窓口の出入り」は strace で覗けます。プログラムが裏で何を頼んでいるかが、そのまま見えます。

$ strace -f ls /tmp
openat(AT_FDCWD, "/tmp", O_RDONLY|O_DIRECTORY) = 3   # ディレクトリを開く
getdents64(3, ...)                            = 512   # 中身を読む
write(1, "file1  file2\n", 13)                = 13    # 画面へ出力
close(3)                                      = 0

ユーザモードとカーネルモードの行き来にはコストがあります。だから「システムコールを呼んでいる時間」と「アプリ自身が計算している時間」は分けて見たい。time コマンドの user / sys がまさにそれで、sys が大きければ「カーネルへのお願いが多い」サインです。

$ time ./myprog
real    0m2.13s   # 実時間（壁の時計）
user    0m1.80s   # ユーザモードでの計算時間
sys     0m0.30s   # カーネルモード（システムコール）の時間
なぜモードを分けるのか

もし全アプリがハードウェアを直接触れたら、1つのバグが他のプロセスのメモリやディスクを壊し放題になります。権限の壁を1枚挟むことで、カーネルが「正しい依頼か」を検査でき、システム全体の安定と公平が保てます。利便と安全のトレードを、この仕切りが引き受けています。

03
Part I · 土台

プロセスを生む — fork と exec

Linuxで新しいプログラムを動かす手順は、独特の2段構えです。fork() で自分の分身を作り、その分身がexecve() で別のプログラムに化ける。シェルがコマンドを起動するのも、この組み合わせです。

ここで効くのが コピーオンライト（CoW）。fork() の瞬間に親のメモリを全部複製したら重すぎます。そこでカーネルは「最初は親子で同じ物理メモリを共有させ、どちらかが書き換えた瞬間に、そのページだけ実コピーする」。読むだけなら一切コピーが起きません。賢い先延ばしです。

プロセスの状態は ps で見えます。CPUを使っている R（実行/実行可能）、I/O待ちなどで眠る S/D（スリープ）、終了したのに親が回収していない Z（ゾンビ）。ゾンビは終了コードを親が wait() で受け取れば消えます。

試してみる

ps aux の STAT 列を眺めるだけでも学びがあります。D（割り込み不可スリープ）が並んでいたら、たいていストレージI/O待ち。「CPUは暇なのに遅い」の典型サインで、ボトルネックがCPUでなくディスクにあると当たりを付けられます。

04
Part II · CPUを配る

スケジューラと並列度

CPU（正確には論理CPU）は、ある瞬間にはたった1つのプロセスしか動かせません。なのに何百ものプロセスが「同時に」動いて見えるのは、スケジューラがごく短い時間（タイムスライス）ずつ、高速に持ち回りで切り替えているから。この切り替えがコンテキストスイッチです。

ここに本書らしい実験があります。同じ計算をプロセス数を増やしながら走らせ、スループットを測る。論理CPUが4個なら、4プロセスまでは台数に比例して伸び、それを超えると頭打ちになります。CPUは4個しかないので、5個目以降は順番待ちが増えるだけ——「測ると一目瞭然」の代表例です。

# 論理CPUの数を確認
$ nproc
4
# CPU使用率を1秒ごとに観測（user/system/iowait/idle の内訳）
$ sar -P ALL 1
CPU   %user  %system  %iowait  %idle
all    98.2     1.3      0.0     0.5    ← 4プロセス時：使い切っている
並列度（プロセス数）論理CPU=4 での挙動読み取れること
1 〜 4ほぼ比例してスループット向上CPUに空きがある。増やす価値あり
= 4全CPUを使い切る（%idle≈0）このマシンの上限に到達
5 以上スループットは伸びず、応答だけ悪化過剰並列。切り替え損が増えるだけ
よくある誤解

「プロセスやスレッドを増やせば増やすほど速くなる」は誤り。物理的な論理CPU数が天井です。それ以上はコンテキストスイッチのオーバーヘッドが積み上がり、かえって遅くなることもある。sar の %idle が0でないなら、まだCPUに余地がある、と読み替えられます。

05
Part III · メモリの魔法

仮想メモリのしくみ

もし各プロセスが物理メモリのアドレスを直接握っていたら、互いのアドレスが衝突して共存できません。そこでカーネルは、各プロセスに「自分専用の広大なメモリがある」という幻を見せます。これが仮想メモリ。プロセスが触るのは仮想アドレスで、カーネルとCPUの MMU がページテーブルを使って物理アドレスへ翻訳します。

この仕組みが、いくつもの便利を生みます。プロセス分離（他人のメモリは見えない）、共有（同じライブラリの実体は1つを共有）、そしてデマンドページング。デマンドページングとは、確保した瞬間には物理メモリを割り当てず、実際にアクセスした時に初めて割り当てること。だから「巨大な配列を確保したのに、実メモリはあまり減らない」が起こります。

メモリ統計を読むと、この遅延割り当てが見えます。VSZ（仮想サイズ＝確保した約束）と RSS（実際に物理メモリへ載っている量）は別物です。

$ ps -o pid,vsz,rss,comm -p 1234
PID    VSZ      RSS     COMMAND
1234   810240   24576   ./app    ← VSZ(約束)は大きいが RSS(実体)は小さい
なぜ「約束」だけ先に渡すのか

プログラムは確保したメモリを全部使うとは限りません。使うかどうか分からない分まで先に物理メモリを割けば、もったいない。「使うと言った時ではなく、本当に触れた時に渡す」ことで、物理メモリを実際の需要に合わせて配れる。仮想メモリは、有限の資源を「あるように見せる」ための堂々たるごまかしです。

06
Part III · メモリの魔法

メモリを使い切ったら

デマンドページングの裏返しとして、Linuxはオーバーコミットを許します。物理メモリより多くを「約束」できてしまう。普段は問題ありませんが、皆が一斉に本気で使い始めると物理メモリが足りなくなる。その時カーネルが取る手段が2つあります。

手段1

スワップ

当面使われていないページをストレージへ退避し、物理メモリを空ける。必要になれば読み戻す。ただしディスクは桁違いに遅いので、頻発すると激重になる。

手段2

OOM Killer

退避でも追いつかず限界に達すると、カーネルがプロセスを強制終了して空きを作る。最後の安全装置。誰が殺されるかは点数（oom_score）で決まる。

特に怖いのが スラッシング。空きが足りずスワップの出し入れに追われ、CPUは暇なのにディスクI/Oだけが張り付いて、システム全体がほとんど進まなくなる状態です。本書はこれも測って見せます——si/so（スワップイン/アウト）が立ち、%iowait が跳ね上がる。

$ sar -B 1        # ページング統計
pgpgin/s  pgpgout/s   fault/s   majflt/s
  9821      4503      12044      980    ← majflt(ディスクから読み直し)が多い＝スラッシング兆候

$ free -h
              total   used    free   available
Mem:           7.6G   7.2G    120M       90M    ← 空きがほぼ無い
Swap:          2.0G   1.9G     90M             ← スワップも満杯寸前

ページフォルトには2種類あります。マイナーフォルト（物理メモリ上にはあるが対応付けがまだ／初回アクセスの割り当て）は安い。メジャーフォルト（ストレージから読み直しが必要）は遅い。majflt の多さは、メモリ不足の確かな指標です。

現場での読み方

「サーバが急に固まった」の多くは、CPU過負荷ではなくメモリ枯渇からのスラッシングか、OOM Killer の発動です。前者は free と %iowait、後者は dmesg や journalctl に残る "Out of memory: Killed process" のログで確認できます。原因の切り分けがしくみから逆算できると、復旧が速くなります。

Part IV · 速さの正体

記憶階層とキャッシュ

記憶装置は「速いほど小さく高価、遅いほど大きく安い」という階層をなします。CPUは速すぎて、毎回メインメモリを待っていられない。そこで間にCPUキャッシュを挟み、よく使うデータを手元に置く。さらにストレージは桁違いに遅いので、一度読んだファイルの内容をメモリに残すページキャッシュで覆い隠します。

ページキャッシュの効果は実験で劇的に見えます。同じファイルを2回読むと、2回目はディスクに触れずメモリから返るので段違いに速い。本書定番のデモです。

$ sync; echo 3 | sudo tee /proc/sys/vm/drop_caches  # キャッシュを捨てる
$ time cat bigfile > /dev/null
real  0m1.82s     ← 1回目：ディスクから読む（遅い）
$ time cat bigfile > /dev/null
real  0m0.07s     ← 2回目：ページキャッシュから（圧倒的に速い）

キャッシュが効くのは、プログラムのアクセスに局所性があるから。直前に使った場所や、その近くを再び使いやすい（時間的・空間的局所性）。CPUキャッシュはキャッシュラインという塊で運ぶので、メモリを連続して順に触るコードはキャッシュに乗りやすく、飛び飛びに触るコードは遅くなります。同じ計算量でもアクセス順で速度が変わるのはこのためです。

設計に効く一言

「メモリは平らで一様」と思って書くと、思わぬ遅さに出会う。データを近くにまとめ、順に触る——配列を素直に走査する、構造体の並びを意識する。それだけでキャッシュが味方になり、体感速度が変わります。

Part V · 永続化

ファイルシステム

ストレージは本来「ただのバイトの並び」です。そこにファイルシステムが「名前・ディレクトリ・サイズ・権限」という秩序を与え、人が扱える形にします。Linuxはあらゆる種類のFS（ext4・XFS・Btrfs…）をVFSという共通の層で抽象化し、アプリは open/read/write という同じシステムコールで、種類を意識せず扱えます。

ファイルシステムが向き合う最大の難問はクラッシュ整合性。書き込みの途中で電源が落ちたら、中身が壊れかねません。多くのFSが採る対策がジャーナリング。本番の更新の前に「これから何をするか」をジャーナル（ログ）に先に書く。途中で落ちてもログを見て「やり直す／なかったことにする」を判断でき、半端な状態を避けられます。

Btrfsのような新しいFSは、もう一歩進んでコピーオンライトを採ります。更新を元の場所に上書きせず別の場所に書いてから差し替えるので、途中状態が原本を壊さない。さらにスナップショット（ある時点の状態を軽量に保存）やチェックサムによる破損検知も自然に手に入ります。fork のCoWと同じ「上書きせず、書いてから切り替える」哲学です。

注意：書いた＝保存された、ではない

write() が返ってきても、データはまだページキャッシュの上にあるだけかもしれません（性能のため遅延書き込みする）。電源断で消える可能性があり、本当に永続化したいなら fsync() でストレージへの書き出しを待つ必要があります。「速さ（遅延書き込み）」と「安全（即時永続化）」は、ここでもトレードオフです。

Part V · 永続化

ストレージデバイス

FSの下にある物理デバイスの性格を知ると、性能の理由が腑に落ちます。HDDは円盤を磁気ヘッドが読む機械です。データの場所が飛ぶたびにヘッドの移動（シーク）が要る。だから連続した領域を一気に読む「シーケンシャルアクセス」は速く、あちこち飛ぶ「ランダムアクセス」は激しく遅い。本書はこれをグラフで突きつけます。

SSDには可動部がなく、電気的に読み書きするためシークがありません。だからランダムアクセスでの落ち込みが小さい。とはいえSSDも「連続のほうが速い」傾向は残ります。fio や dd で自分のディスクのこの差を測れます。

# 連続書き込みの速度をざっくり測る
$ dd if=/dev/zero of=testfile bs=1M count=1024 oflag=direct
1073741824 bytes copied, 3.2 s, 336 MB/s   ← シーケンシャル

# 4KBランダムの実力は fio で（IOPSで効いてくる）
$ fio --name=rr --rw=randread --bs=4k --size=1G --direct=1

カーネルは少しでも有利にしようと、I/Oスケジューラで要求を並べ替えます。近い場所へのアクセスをまとめ、HDDならシークを減らす。バラバラに来た読み書きを賢く整理してデバイスへ渡す、最後の仲介役です。

設計に効く要点

遅いのが「CPU待ち」か「I/O待ち」かで打つ手は正反対。%iowait が高ければ、コードを速くするよりI/Oの仕方——まとめ読み・連続配置・キャッシュ活用——を見直すほうが効きます。デバイスの性格を知ることが、的を外さない最短路です。

Part VI · 隔離して動かす

仮想化とコンテナ

増補改訂版の目玉が、ここまでの土台の上に立つ仮想マシン（VM）とコンテナです。両者は「隔離して動かす」点で似ていますが、隔離の深さが決定的に違います。

観点仮想マシン（VM）コンテナ
隔離の単位ハードウェアごと仮想化（ゲストOSを丸ごと動かす）1つのカーネルを共有し、見える範囲だけ仕切る
カーネルゲストごとに別のカーネルホストのカーネルを共用
起動・サイズ重い・遅い（OS一式が必要）軽い・速い（プロセスに近い）
隔離の強さ強い（別マシン相当）VMよりは緩い（同じカーネルを攻撃面に持つ）

コンテナの正体は、特別な仮想化技術ではなくカーネルの2つの機能の合わせ技です。名前空間（namespace）で「見える範囲」を仕切り、cgroupで「使える量」を制限する。プロセスから見れば、自分専用のプロセス番号・ネットワーク・ファイルツリーがあるように見える——けれど実体は、ホスト上のただのプロセスです。

仕切る

namespace

PID・ネットワーク・マウント・ホスト名などを独立して見せる。コンテナ内の "PID 1" は、ホストでは普通の一プロセス。「見える世界」を区切る機能。

配給する

cgroup

CPU時間・メモリ・I/Oの上限を設定する。1つのコンテナが資源を食い尽くして他を巻き込まないための枠。「使える量」を区切る機能。

だから本書の前半が効いてきます。コンテナのメモリ制限を超えれば、§06のOOMがコンテナ単位で起きる。CPU制限は§04のスケジューラの上に乗る。ps で見ればコンテナの中身はホストの普通のプロセスとして並びます——魔法ではなく、しくみの組み合わせだと分かると、デバッグの目線が一段上がります。

つながった

コンテナが「軽い」のは、ゲストOSを持たずホストのカーネルをそのまま使うから。プロセス・メモリ・FS・スケジューラというこれまでの全要素が、namespace と cgroup で「区切られた」だけ。OSのしくみを知っていれば、コンテナは新しい概念ではなく既知の部品の再構成として理解できます。

一言で

CPUは時分割で、メモリは仮想化と遅延割り当てで、ストレージはキャッシュで——カーネルは3つの制約を隠し続けている。性能問題とは、その手品が破綻して地金が見えた瞬間です。しくみを知れば、数字から原因へ最短でたどり着けます。

用語集

システムコールアプリがカーネルに処理を依頼する正式な窓口。ユーザモード↔カーネルモードを行き来する。
fork / exec分身を作る（fork）→別プログラムに化ける（exec）の2段でプロセスを起動する流儀。
コピーオンライトまず共有し、書き換える時にだけ実コピーする省資源の技。fork や Btrfs で活躍。
コンテキストスイッチCPUが動かすプロセスを切り替える操作。多すぎると切り替え損が積もる。
仮想メモリ各プロセスに専用アドレス空間の幻を見せ、ページテーブルで物理へ翻訳する仕組み。
デマンドページング確保時ではなく実アクセス時に物理メモリを割り当てる遅延戦略。VSZとRSSが食い違う元。
OOM Killerメモリ枯渇の最終手段として、カーネルがプロセスを強制終了する仕組み。
ページキャッシュ一度読んだファイル内容をメモリに残し、遅いストレージへの再アクセスを省く。
