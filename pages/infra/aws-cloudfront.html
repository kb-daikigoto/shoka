Content Delivery Network

Amazon CloudFront 入門
世界の端から、速く届ける

CloudFront は、世界中に分散した「エッジロケーション」から、Web コンテンツを利用者に最も近い拠点で届ける CDN（コンテンツ配信ネットワーク）です。静的ファイルも動的な応答も、AWS の高速バックボーン網を通して低遅延で配信します。

01

一言でいうと

普通にサイトを運営すると、利用者のリクエストは毎回、遠く離れたオリジン（S3 や Web サーバー）まで、いくつものネットワークを越えて届きます。距離が遠いほど遅く、アクセスが集中するとオリジンの負荷も上がります。

CloudFront は、利用者の近くにコンテンツのコピー（キャッシュ）を置きます。リクエストは最寄りのエッジロケーションへ routed され、そこにコピーがあれば即座に返ります。結果として、最初のバイトが届くまでの時間（レイテンシ）が縮み、転送速度が上がり、オリジンの負荷も減って可用性が高まります。

なぜ速いのか

CloudFront は各リクエストを AWS のバックボーン網を通して最適なエッジへ流します。利用者のリクエストが通過するネットワーク数が劇的に減るため、一般のインターネット経路より速くなります。さらに、ファイル（オブジェクト）のコピーが世界中の複数拠点に保持されるので、信頼性も上がります。

02

主要な用語（これだけ押さえる）

TERM 01

エッジロケーション / POP

世界中に散らばる配信拠点。利用者に最も近い拠点が応答する、配信の最前線。

TERM 02

オリジン (Origin)

コンテンツの正本の置き場所。S3 バケット、ALB、API Gateway、任意の HTTP サーバー（カスタムオリジン）など。

TERM 03

ディストリビューション

CloudFront の設定単位。「どのオリジンから・どう配信するか」を定義。作ると dxxxx.cloudfront.net のドメインが割り当てられる。

TERM 04

オブジェクト

配信されるファイルの呼び名。HTML・CSS・JS・画像・動画など、HTTP で配れるものは何でも。

独自ドメイン（例 www.example.com）を割り当てることもできます。その場合 URL は CloudFront のドメインではなく自分のドメインのまま使えます。

03

仕組み（配信の流れ）

セットアップから配信までの流れはこうです。まずオリジンを指定し、ディストリビューションを作成すると CloudFront がドメイン名を割り当て、その設定（コンテンツ本体ではなく）を全エッジへ配布します。コンテンツは、利用者が初めてアクセスしたときにオリジンから取得され、エッジにキャッシュされます。

取り込みは無料

S3・ELB・API Gateway などの AWS オリジンからの取り込み転送は常に無料です。課金されるのは「CloudFront から利用者へ出ていく転送（アウトバウンド）」だけ。AWS 内に origin を置くほどコスト効率が良くなります。

04

キャッシュの寿命と更新

各ファイルがエッジに留まる時間は デフォルトで 24 時間。最小は 0 秒、最大の上限はありません。これはオリジンが付ける Cache-Control などのヘッダや、CloudFront 側の TTL 設定で制御します。

「キャッシュ期間中にコンテンツを更新したい」ときの定石は 2 つ。ひとつはキャッシュの無効化（invalidation）で、指定パスのキャッシュを強制的に破棄します。もうひとつはバージョン付きファイル名（例 app.v2.js）で、URL を変えて常に新しいものとして扱わせる方法です。後者の方がコストも安く、本番では推奨されます。

05

セキュリティ

CloudFront はセキュリティ機能の入口にもなります。代表的なものを押さえておきましょう。

  • HTTPS / TLS：ACM（AWS Certificate Manager）の証明書で暗号化配信。独自ドメインでも HTTPS を提供できます。
  • 署名付き URL・署名付き Cookie：限られた利用者にだけ、期限付きでコンテンツを配る仕組み（有料動画や会員限定ファイルなど）。
  • OAC（Origin Access Control）：S3 を非公開のままにし、CloudFront 経由のアクセスだけを許可する。バケットを公開せずに済みます。
  • AWS WAF / Shield：Web 攻撃のフィルタリングや DDoS 防御をエッジで適用。
06

エッジでコードを動かす

CloudFront は単に配るだけでなく、エッジ上で軽いロジックを実行できます。用途で 2 つを使い分けます。

機能性格向いている処理
CloudFront Functions超軽量・超低遅延・JavaScriptヘッダの書き換え、URL リライト、簡単なリダイレクトや認可
Lambda@Edge高機能・複数言語・やや重いオリジン応答の加工、複雑な認証、A/B 出し分けなど

「リクエストやレスポンスを、利用者の近くで少しいじりたい」場合に強力です。重い処理ほど Lambda@Edge、ごく軽い処理は CloudFront Functions が目安。

07

標準 vs マルチテナント配信

ディストリビューションには 2 種類あります。標準ディストリビューションは 1 つのサイト／アプリごとに固有設定を持つ、最も一般的なタイプ。まずはこれで始めます。

マルチテナント配信（CloudFront SaaS Manager）は、SaaS 事業者が多数の似た構成のサイトをまとめて効率的に管理するためのもの。共有設定を中央で管理しつつ、テナントごとに配信したいケース向けです。

08

料金の考え方

課金の柱は 2 つ。エッジから出ていくデータ転送量と、HTTP / HTTPS リクエスト数です。料金は利用タイプ・地域・使う機能によって変わります。前述のとおり AWS オリジンからの取り込みは無料なので、実質「利用者へ届けた量」に対して支払う形になります。

注意

具体的な単価や無料枠は地域・時期で変わります。見積もりは必ず 公式の料金ページで最新を確認してください。

一言で

CloudFront は「利用者の近くにコンテンツのコピーを置き、AWS の速い網で届ける」サービス。速度・オリジン負荷軽減・セキュリティの入口を、まとめて引き受けてくれます。

本記事は AWS 公式「What is Amazon CloudFront?」をもとに初心者向けに再構成したものです。仕様・料金・機能はバージョンや時期で変わるため、最新は公式ドキュメントを確認してください。
