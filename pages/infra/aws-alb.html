Application Load Balancer 入門
L7 で、賢く振り分ける

ALB は、受け取った HTTP/HTTPS リクエストを、複数のアベイラビリティゾーンにある健全なターゲット（EC2・コンテナ・IP・Lambda）へ振り分けるロードバランサーです。OSI 第 7 層（アプリケーション層）で動くので、URL やヘッダの中身を見て送り先を変えられます。

01

一言でいうと

ロードバランサーは、クライアントにとっての唯一の窓口（単一の接点）になり、後ろにいる複数のサーバーへ負荷を分散します。Elastic Load Balancing（ELB）はこれを自動で行い、ターゲットの健康状態を監視し、健全なターゲットにだけトラフィックを流し、負荷に応じて自動的にスケールします。

ELB には 4 種類あります。ALB（Application）・NLB（Network）・GWLB（Gateway）・CLB（Classic, 旧式）。この記事は HTTP/HTTPS 向けの ALB を扱います。

ALB が解決すること

サーバー 1 台では、落ちたら全停止、混んだらさばけません。ALB を前に置くと、複数台へ分散し、落ちた台は自動的に外れ、台数の増減も無停止で行えます。さらに「/api はこちら、それ以外はあちら」のように中身で振り分けることもできます。

02

「L7 で振り分ける」とは

ALB は OSI 参照モデルの第 7 層＝アプリケーション層で動きます。これは「HTTP リクエストの中身（URL パス、ホスト名、ヘッダ、メソッド、クエリ等）まで読んで判断できる」という意味です。リクエストを受け取ると、ALB はリスナールールを優先度順に評価し、最初に条件に合ったルールのアクションを実行して、送り先を選びます。

対照的に NLB は第 4 層（トランスポート層＝TCP/UDP）で動き、中身は見ずに「接続」を高速にさばきます。中身を見て賢く分けたいなら ALB、ひたすら速く大量にさばきたいなら NLB、という住み分けです（後述）。

03

3 つの構成要素

ALB はこの 3 つの部品でできています。この関係さえ掴めば設定で迷いません。

PART 01

リスナー (Listener)

「どのプロトコル・どのポートで待つか」。例: HTTPS の 443 番。クライアントからの接続を受ける耳。

PART 02

ルール (Rule)

リスナーに紐づく振り分けの規則。優先度・条件・アクションの組。条件に合えばアクション実行。1 つは必ずデフォルトルール。

PART 03

ターゲットグループ (TG)

実際の送り先の束。EC2・IP・Lambda を登録し、ヘルスチェックは TG 単位で設定する。

04

ルーティング条件（中身で分ける）

ALB の真価はここ。ルールの条件に使えるリクエスト要素が豊富で、1 台の ALB で複数サービス・複数ドメインをさばけます。

  • パス条件：/api/* はサービス A、/img/* はサービス B、のように URL で分岐。マイクロサービス化に有効。
  • ホスト条件：HTTP ヘッダのホスト名で分岐。1 台で複数ドメインを受けられる。
  • HTTP ヘッダ・メソッド・クエリ文字列・送信元 IPでも分岐可能。
  • リダイレクト（別 URL へ）や固定レスポンス（定型の HTTP 応答を返す）もアクションとして指定可能。
アクションと条件

各ルールは「優先度・1 つ以上の条件・1 つ以上のアクション」の組。上から評価し、最初に合致したルールが勝ちます。どれにも合わなければデフォルトルールが適用されるので、必ず 1 つは用意します。

05

ターゲットとヘルスチェック

ターゲットグループには EC2 インスタンス・IP アドレス（VPC 外も可）・Lambda 関数を登録できます。1 つのターゲットを複数の TG に登録することもでき、ルーティングは TG ごとに独立して行われます。

ヘルスチェックは TG 単位で設定し、登録ターゲットの健康状態を監視します。これにより ALB は健全なターゲットにだけリクエストを送ります。送り先の選び方（ルーティングアルゴリズム）は既定がラウンドロビン、もう一つ最小未処理リクエスト（least outstanding requests）が選べます。

ターゲットは複数の AZ にまたがって配置でき、Auto Scaling と組み合わせれば、起動した EC2 は自動で TG に登録、終了した EC2 は自動で登録解除されます。需要に応じてサービスごとに動的にスケールできます。

07

ALB と NLB の使い分け

迷ったときの指針です。

観点ALBNLB
動作する層L7（アプリ層・HTTP/HTTPS）L4（TCP/UDP）
判断材料URL・ヘッダなど中身接続情報のみ（中身は見ない）
得意パス／ホストで賢く振り分け、マイクロサービス、認証超低遅延・超大量接続・固定 IP・非 HTTP
ターゲットEC2 / IP / LambdaEC2 / IP / ALB

「Web アプリの前段で、中身を見て分けたい」なら基本 ALB。「ゲーム・IoT・金融など、とにかく速く大量の接続をさばきたい」なら NLB が候補です。

08

料金の考え方

ALB は「使った分だけ」の課金です。大きくは稼働時間と、処理量を表す内部単位（LCU＝新規接続数・アクティブ接続数・処理バイト数・ルール評価数を総合した指標）に基づきます。つまり「立てている時間」と「さばいた量」で決まる、と捉えておけば十分です。

注意

単価や LCU の内訳は時期・地域で変わります。見積もりは 公式の料金ページで最新を確認してください。

一言で

ALB は「リスナーで受け、ルールで中身を見て、健全なターゲットグループへ振り分ける」L7 の交通整理役。1 台で複数サービス・複数ドメインをさばける賢さが持ち味です。
