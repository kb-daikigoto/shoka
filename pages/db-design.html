PART 1

設計の地図を手に入れる

個々のテクニックの前に、「DB設計とは何をやっているのか」という全体像を押さえます。ここを外すと、後の正規化の話が宙に浮きます。

01

そもそも「DB設計」とは何か

DB設計とは「現実世界のデータを、矛盾なく・重複なく・取り出しやすい形でテーブルに割り当てる作業」である。

私たちが扱う多くのデータベースは リレーショナルデータベース（RDB） です。RDBはデータを 2次元の表（テーブル） として持ちます。行が1件のデータ、列がその属性です。SQLはこの表に対して「絞り込む・つなぐ・集計する」を行う言語にすぎません。

そして決定的に重要なのは、SQLの書きやすさも、システムの性能も、データの正しさも、すべて「テーブルの分け方（＝設計）」で先に決まってしまうということです。設計が悪いと、どんなにSQLを工夫してもデータの重複や矛盾は消せません。だから設計を先に学ぶ価値があります。

なぜ重要か

アプリのコードは後から直せますが、稼働後のテーブル構造は直すのが非常に高コストです。すでに大量のデータが入り、たくさんのプログラムがそのテーブルに依存しているからです。設計のミスは最も修正費用が高いバグだと考えてください。

RDBを管理するソフトウェアを DBMS（Database Management System）と呼びます。MySQL・PostgreSQL・Oracle・SQL Server などが代表例です。本ガイドの設計理論はどのDBMSでもほぼ共通して通用します（細かな機能差はあります）。

02

論理設計と物理設計

「どんなデータをどう分けるか」が論理設計。「それを実際の機械にどう載せ、速く動かすか」が物理設計。

DB設計は大きく2段階に分かれます。順番が大事で、論理設計が先、物理設計が後です。論理設計は「ハードウェアやDBMSの都合を一旦忘れて、データの意味だけを考える」フェーズです。

LOGICAL

論理設計

エンティティ（実体）の洗い出し、テーブルと列の決定、正規化、キーの設定、ER図の作成。物理的制約は考えない。

PHYSICAL

物理設計

テーブルの実体化、インデックス設計、データの物理配置（ファイル/ディスク）、性能を出すための調整。

この分離を支えるのが 3層スキーマ という考え方です。ユーザーから見える形（外部）、論理構造（概念）、物理的な保存（内部）の3層に分け、それぞれを独立して変更できるようにします。下図のイメージです。

図: 3層スキーマ ― 中段の概念スキーマが論理設計の成果物
要点

論理設計は「データの正しさ」を、物理設計は「速さ」を担当します。本ガイドの大半（正規化）は論理設計の話です。先に正しさを固め、必要なら後で速さのために崩す、という順序を覚えてください。

PART 2

正規化 ― この本の心臓部

正規化は「重複と矛盾の余地をテーブルの形そのものから取り除く」技術です。ここを理解すれば、設計の8割は身についたと言えます。焦らず一段ずつ登ります。

04

キーと制約 ― 設計の土台

キーとは「行を見分けるための列」。これが決まらないと、テーブルは集合として成立しない。

主キー（PRIMARY KEY）

主キーは、テーブルの中の1行を一意に特定できる列（または列の組）です。主キーが決まれば、残りの全ての列の値は主キーによって一意に定まります。主キーには次の鉄則があります。

一意である

同じ主キー値を持つ行は2つと存在できない。

NULL不可

主キーは必ず値を持つ。「不明」は許されない。

不変が望ましい

後から値が変わると参照関係が壊れる。

複合キーと候補キー・代理キー

複数の列を組み合わせて初めて一意になる主キーを 複合キー と呼びます。また、主キーになりうる候補が複数ある場合、それらを 候補キー と呼び、その中から1つを主キーに選びます。自然な意味を持つ列をキーにする 自然キー に対し、連番などの人工的な列を立てるものを 代理キー（サロゲートキー） と呼びます（17章で詳述）。

外部キー（FOREIGN KEY）と参照整合性

外部キーは、別テーブルの主キーを指す列です。これにより「存在しない会社コードの社員」のような宙に浮いたデータを防げます。これを 参照整合性制約 と言います。

schema.sql ― キーと制約の例
CREATE TABLE 会社 (
  会社コード CHAR(4) PRIMARY KEY,   -- 主キー：一意 & NULL不可
  会社名     VARCHAR(50) NOT NULL
);

CREATE TABLE 社員 (
  社員ID     CHAR(6) PRIMARY KEY,
  社員名     VARCHAR(50) NOT NULL,
  会社コード CHAR(4),
  FOREIGN KEY (会社コード) REFERENCES 会社(会社コード)
  -- 存在しない会社コードは登録できない（参照整合性）
);
tip

「制約」はデータの正しさをDBMS側に守らせる仕組みです。アプリ側のチェックは抜けがありますが、制約はDBが必ず守ります。正しさは可能な限りDB側で担保するのが鉄則です。

05

関数従属性 ― 正規化の言語

「Aが決まればBが1つに決まる」関係を A → B と書く。正規化はこの矢印を整理する作業にすぎない。

正規化を理屈で理解する鍵が 関数従属性（functional dependency）です。難しそうな名前ですが、意味は単純です。ある列Aの値が決まると、別の列Bの値が必ず1つに定まるとき、「BはAに関数従属する」と言い、次のように書きます。

図: {社員ID} → {社員名}（社員IDが決まれば社員名が1つに決まる）
なぜ重要か

主キーとは「テーブルの他の全列が関数従属する列」のことです。つまり正しいテーブルとは『主キー → 全ての列』だけが成り立つテーブル。正規化とは、主キー以外への余計な矢印（部分従属・推移従属など）を見つけて、別テーブルに追い出す作業です。これから出てくる用語はすべてこの「矢印」の種類だと思ってください。

06

正規化とは ― なぜ必要か

正規化とは「1つの事実は1か所にだけ書く」を徹底するためのテーブル分割ルールである。

正規化していないテーブルがなぜ危険か、具体例で見ましょう。次の「社員」テーブルは、社員の情報と会社の情報を1枚に詰め込んでいます。

社員ID(主キー)社員名会社コード会社名
E001佐藤C01アクメ商事
E002鈴木C01アクメ商事
E003田中C02ビーコン物産

「アクメ商事」という会社名が2回書かれています。これが諸悪の根源です。重複があると、次の3つの更新時異常（anomaly）が起きます。

更新異常

会社名が「アクメ株式会社」に変わったら全行を直す必要がある。1行直し忘れると矛盾が生まれる。

挿入異常

社員のいない新会社を登録できない。社員ID(主キー)が空では行を作れないから。

削除異常

最後の社員を消すと、その会社の情報まで道連れで消える。

要点

正規化のゴールは美しさではなく「これらの異常を構造的に起こせなくする」こと。重複を消せば、更新は1か所で済み、矛盾は生まれようがありません。次章から、段階的にこの社員テーブルを直していきます。

tip ― 正規化は可逆

正規化で分割したテーブルは、JOIN（結合）で必ず元に戻せます。情報は1つも失われません。だから安心して分割できます。逆に言えば「戻せない分割」は正規化ではありません。

07

第1正規形（1NF）

1セルには1つの値だけ。繰り返し項目や「複数値の詰め込み」を排除する。

第1正規形のルールは1つだけ。各セルの値はこれ以上分割できない単一の値（スカラ値）であること。次のような設計は1NF違反です。

社員ID社員名子の名前（NG）
E001佐藤太郎, 花子
E002鈴木一郎

「太郎, 花子」のように1セルに複数値を詰め込むと、「花子だけ検索」「子供を1人追加」がSQLでまともに書けません。これを別行に分けて1NFにします。

社員ID社員名子の名前
E001佐藤太郎
E001佐藤花子
E002鈴木一郎
danger ― 配列やJSONの安易な利用

「列にカンマ区切りで詰める」「配列型に放り込む」のは1NF違反の典型です。検索・集計・整合性チェックがすべて困難になります。繰り返す情報は行に展開するのが原則。どうしても1セルに複数を入れたくなったら、それは別テーブルに分けるべきサインです。

なお1NFにして初めて「主キー → 各列」という関数従属が定義できます。次の2NF・3NFは、この主キーへの従属の仕方をさらに精密に整えていく作業です。

08

第2正規形（2NF）

複合主キーの「一部」だけで決まる列を、別テーブルに追い出す（部分関数従属の排除）。

2NFが問題になるのは 複合主キー のときだけです。次の「受注明細」テーブルは {受注ID, 商品ID} が複合主キーです。

受注ID商品ID数量商品名
R01P103ボールペン
R01P201ノート
R02P105ボールペン

ここで関数従属を観察します。数量は {受注ID, 商品ID} の両方がないと決まりません（＝完全関数従属＝正しい）。しかし 商品名は {商品ID} だけで決まります。これは主キーの一部への従属＝部分関数従属で、2NF違反です。

図: 商品名は商品IDだけに従属（部分関数従属）＝分離すべき

解決策は、部分従属している列を別テーブルへ切り出すことです。商品マスタを独立させます。

受注明細：受注ID(PK)商品ID(PK)数量
R01P103
R01P201
商品：商品ID(PK)商品名
P10ボールペン
P20ノート
要点

2NF違反は「主キーが複合のとき、その一部分を見れば値が決まってしまう列」。商品名は今や1か所にしかなく、商品名の変更は商品テーブルの1行を直すだけで済みます。

09

第3正規形（3NF）

「主キー → 非キー列A → 非キー列B」という間接的な従属（推移的関数従属）を断ち切る。

06章の社員テーブルに戻りましょう。これは単一主キーなので2NFは満たしますが、別の問題があります。

社員ID(PK)社員名部署コード部署名
E001佐藤D1営業部
E002鈴木D1営業部
E003田中D2開発部

関数従属を追うと、{社員ID} → {部署コード} → {部署名} という2段の矢印になっています。社員IDから部署名へは、部署コードを経由して間接的に決まる。これが 推移的関数従属 で、3NF違反です。「営業部」が複数行に重複しているのが見て取れますね。

図: 部署名は部署コードを経由して間接的に決まる（3NF違反）

解決は2NFと同じ発想。間接的に従属している部分を別テーブルへ切り出します。

社員：社員ID(PK)社員名部署コード(FK)
E001佐藤D1
E002鈴木D1
部署：部署コード(PK)部署名
D1営業部
D2開発部
query.sql ― 分割しても JOIN で元に戻る
-- 社員名と部署名を一緒に見たいときは結合する
SELECT e.社員名, d.部署名
FROM   社員 e
JOIN   部署 d ON e.部署コード = d.部署コード;
要点 ― 第3正規形までが実務の標準

多くの業務システムは 第3正規形 まで正規化すれば十分です。覚え方は有名な一文 ―「キーに、キー全体に、キー以外の何ものにも従属しない」。非キー列は、主キーだけに・主キー全体に・主キー以外には従属しない、という3条件が、そのまま1NF→2NF→3NFの精神を言い表しています。

10

ボイス・コッド正規形（BCNF）

3NFを満たしても残る「非キー列 → キーの一部」という逆向きの従属をなくす、より厳しい3NF。

BCNFは「すべての関数従属の左側（決定項）が候補キーであること」を要求します。3NFをほぼ満たすが、候補キーが複数あって一部が重なる特殊なケースで違反が起きます。

例: {学生, 科目} が主キーで、各科目を担当する教員が1人だけ決まっているとします。すると {教員} → {科目}（教員が決まれば担当科目が決まる）という従属が生まれます。教員は候補キーではないのに、キーの一部（科目）を決めてしまう ― これがBCNF違反です。

warn ― 実務での立ち位置

BCNF・第4・第5正規形は理論上重要ですが、実務で意識する場面は多くありません。「3NFまできっちりやれば、ほとんどの異常は防げる」と理解した上で、こういう上位の正規形が存在すること、そして稀に必要になることだけ頭の片隅に置いておけば十分です。深入りより、3NFを確実に使えることが大切です。

11

第4・第5正規形

1つの主キーに、互いに無関係な「多対多」が2つ以上ぶら下がっている状態を分割する。

第4正規形は 多値従属 を排除します。例えば「社員」に対して、互いに独立な「扱える言語」と「持っている資格」を同じテーブルに同居させると、無意味な組み合わせ爆発が起きます。これらを別々のテーブルに分けるのが4NFです。第5正規形（結合従属の排除）はさらに稀なケースを扱います。

要点

4NF/5NFの核心は「独立した関係を1つのテーブルに混ぜない」という、3NFと同じ直感の延長です。新しい考え方というより、関数従属の整理を多対多にも徹底したもの、と捉えれば怖くありません。日常設計では「独立な多対多は別テーブル」とだけ覚えておけば実害は出ません。

12

正規化と非正規化のトレードオフ

正規化は「正しさ」を上げるが、しばしば「速さ」を下げる。両者は基本的にトレードオフの関係にある。

正規化すればするほどテーブルは増え、欲しい情報を得るのに JOINの回数が増えます。JOINは計算コストが高く、巨大なテーブルでは性能問題になります。そこで、あえて重複を許して結合を減らす 非正規化（正規化崩し） が選択肢になります。

観点高度に正規化非正規化（崩す）
データの正しさ◎ 矛盾が起きにくい△ 重複で矛盾リスク
検索（SELECT）△ JOINが増え遅くなりがち◎ 結合不要で速い
更新（UPDATE）◎ 1か所直すだけ△ 全コピーを直す必要
保守性◎ 構造が素直△ 整合性維持の手間
danger ― 順番を間違えない

非正規化は「最後の手段」です。最初から崩すのではなく、まず正規化した設計を作り、実測して性能が足りないと判明した箇所だけ、リスクを理解した上で限定的に崩します。「速くなりそうだから」という想像で崩すのは、矛盾だらけのDBへの第一歩です。

なぜか ― 崩した重複は誰が守る？

非正規化で重複させたデータは、DBMSが自動では一致を保証してくれません。一致させ続ける責任はアプリ側のコードや運用に移ります。この「見えないコスト」を必ず勘定に入れてから判断してください。多くの場合、まずインデックス（14章）で十分速くなります。

PART 3

図とパフォーマンスの基礎

設計を人に伝えるための「ER図」と、物理設計の主役「インデックス」。論理設計と物理設計をつなぐ章です。

13

ER図 ― 設計を1枚の地図にする

ER図は「どんなテーブルが、どんな関係（1対多など）でつながっているか」を一目で示す設計図。

正規化で分割が進むと、テーブル同士の関係を把握しづらくなります。それを可視化するのが ER図（Entity-Relationship図）です。重要なのは カーディナリティ（多重度） ― 1対1、1対多、多対多のどれかです。

図: 「部署 1 ―― 多 社員」の関係。FK（外部キー）が関係を表す
tip ― 多対多は分解する

「学生 と 講義」のような多対多は、RDBに直接は持てません。間に 連関テーブル（中間テーブル）（例: 受講テーブル）を挟み、「1対多」＋「多対1」の2本に分解します。正規化を進めると自然にこの形になります。

14

インデックス ― 速さの主役

インデックスは「本の索引」。全ページ読まずに目的の行へ直行できる、最も費用対効果の高い高速化手段。

インデックスがないと、DBMSは目的の行を探すのにテーブルを先頭から全部読み（フルスキャン）ます。100万行なら100万行調べます。インデックスを張ると、ソート済みの索引をたどって一気に目的地へ到達できます。多くのDBMSは内部で B-tree（バランス木） という構造を使います。

図: B-tree。大小で枝を選びながら少ない比較回数で目的の行に到達する
index.sql
-- 「会社コードで社員を検索」が多いならインデックスを張る
CREATE INDEX idx_社員_会社 ON 社員(会社コード);

-- これで下のWHERE検索が高速になる
SELECT * FROM 社員 WHERE 会社コード = 'C01';
warn ― インデックスはタダではない

インデックスは検索を速くしますが、INSERT/UPDATE/DELETEのたびに索引も更新するため書き込みは遅くなり、ディスクも消費します。「とりあえず全列に張る」は逆効果。検索条件(WHERE)・結合(JOIN)・並び替え(ORDER BY)でよく使う列に絞って張るのが基本です。値の種類が極端に少ない列（性別など）は効きにくい点にも注意。

15

統計情報 ― DBMSの「勘」の源

DBMSは統計情報を見て「インデックスを使うか、全部読むか」を自分で決めている。

同じSQLでも、DBMSは状況に応じて最も速そうな実行方法（実行計画）を選びます。その判断材料が 統計情報 ― 「テーブルに何行あるか」「ある列に何種類の値があるか」などのデータの分布です。これを使って最適な計画を立てるDBMS内部の仕組みを オプティマイザ と呼びます。

なぜ重要か ― 急に遅くなる原因No.1

統計情報が古い（実態とずれている）と、オプティマイザが判断を誤り、せっかくのインデックスを使わずフルスキャンする、といった事故が起きます。「昨日まで速かったのに突然遅い」の典型原因です。大量のデータ投入後は統計情報を更新する（多くのDBMSで ANALYZE 等）、と覚えておきましょう。

PART 4

やってはいけない設計

「動くけれど、いずれ必ず後悔する」設計のパターン集です。原則（正規化）を学んだ今なら、なぜダメかが理屈で分かります。

16

論理設計のバッドノウハウ

これらは「絶対に避けるべき」アンチパターン。1つでも見つけたら設計を疑う。

BAD

非スカラ値

1セルに複数値を詰める（1NF違反）。検索も整合性も壊れる。→ 行に展開する。

BAD

ダブルミーニング

1つの列が状況によって違う意味を持つ（「金額」が時に在庫数）。列は1つの意味だけに。

BAD

単一参照テーブル

性質の違う複数のマスタを「コード種別＋値」の1表に無理やり統合。型も制約も効かなくなる。

BAD

不適切なキー

可変長文字列や「意味が変わりうる値」を主キーにする。値が変わると参照が壊れる。

BAD

テーブル分割の乱用

「月別テーブル」など物理都合での水平/垂直分割。検索が複雑化し正規化の利点を失う。

BAD

ダブルマスタ

同じ意味のマスタが2つ存在し、どちらが正か分からなくなる。マスタは1つに統一。

danger ― 共通する病巣

これらの多くは「正規化のサボり」または「物理都合の論理設計への混入」が原因です。論理設計の段階では性能を考えない、という02章の原則を守れば、その大半は自然に避けられます。

17

グレーノウハウ

「常に悪」ではないが、使うなら理由とリスクを言える状態で使うべき技法。

代理キー（サロゲートキー）

自然キーが存在しない、または不安定（値が変わる・重複しうる）なとき、連番などの人工キーを立てます。有用ですが、安易に全テーブルでやると自然キーへの一意制約を忘れて重複データを許す事故が起きがちです。代理キーを使っても、本来の自然キーには UNIQUE 制約を付けるのが安全です。

列持ち と 行持ち

「1月,2月,3月…」を列として並べるか（列持ち）、行として持つか（行持ち）。列持ちは見やすい反面、項目が増えると列追加＝スキーマ変更が必要で硬直的。行持ちは柔軟ですが集計SQLがやや複雑。原則は行持ちが拡張に強くおすすめです。

方式長所短所
列持ち1行で横並び、見やすい項目追加でスキーマ変更が必要
行持ち項目追加が行追加だけで済む横断集計のSQLが複雑になりがち

多段ビュー

ビューの上にビュー、さらにその上にビュー…と重ねると、元のテーブル構造が見えなくなり、性能も予測困難になります。ビューは浅く保つのが原則です。

要点

グレーノウハウの共通テーマは「便利さと引き換えに、整合性や保守性のどこかを担保する責任が自分に移る」こと。使うこと自体は禁止ではなく、トレードオフを言語化できるかどうかが、達人と初心者の分かれ目です。

18

木構造（階層データ）の格納

「組織図」「カテゴリ階層」のような木構造を、平らな表でどう表現するかの定番パターン。

RDBは平らな表が得意で、親子の階層（木構造）は素直には表せません。代表的な格納方法が3つあります。

最も基本

隣接リストモデル

各行に「親のID」列を持たせる。実装が直感的で更新も簡単。反面、「全ての子孫」を取るには再帰が必要で深い階層に弱い。

検索が速い

入れ子集合モデル

各ノードに左右の番号(座標)を振り、範囲で子孫を表現。子孫の一括取得が1クエリで速い。反面、ノード追加時に多くの番号を振り直す必要があり更新が重い。

経路で表現

経路列挙モデル

「/1/4/9/」のように先祖の経路を文字列で持つ。先祖・子孫検索がLIKEで簡単。反面、経路が長くなり移動時の更新に注意。

tip ― まずは隣接リストから

初学者はまず隣接リストモデル（親IDを持つ）を確実に使えるようになれば十分です。多くのDBMSは再帰クエリ（WITH RECURSIVE）に対応しており、深い階層もたどれます。性能で困ってから他のモデルを検討すれば遅くありません。

★

チートシート ― 正規形 早見表

正規形排除するもの一言で
1NF非スカラ値（繰り返し）1セルに値は1つ
2NF部分関数従属主キーの一部で決まる列を分離
3NF推移的関数従属非キー経由で決まる列を分離
BCNF非キー→キー一部の従属決定項は必ず候補キー
4NF多値従属独立な多対多を別表に
5NF結合従属これ以上分割で矛盾が出ない形
最重要の合言葉

非キー列は 「キーに、キー全体に、キー以外の何ものにも従属しない」。これが1NF→2NF→3NFの本質。実務はまず3NFを確実に。

設計の判断フロー

状況すべきこと
1セルに複数値がある行に展開（1NF）
複合主キーの一部で決まる列別テーブルへ分離（2NF）
非キー列が他の非キー列で決まる別テーブルへ分離（3NF）
検索が遅いまずインデックスを検討（14章）
インデックスでも足りない実測の上で限定的に非正規化（12章）
自然キーが不安定代理キー＋自然キーにUNIQUE（17章）
階層データを持ちたいまず隣接リスト（親ID）（18章）
＃

用語集

RDB / DBMS
データを表で持つDB（RDB）と、それを管理するソフト（DBMS）。
主キー
1行を一意に特定する列。一意・NULL不可・不変が原則。
外部キー
他テーブルの主キーを指す列。参照整合性を担保する。
候補キー
主キーになりうる列の組。複数あれば1つを主キーに選ぶ。
代理キー
連番などの人工的な主キー。自然キーが不安定なとき使う。
関数従属（A→B）
Aが決まればBが1つに定まる関係。正規化の基本言語。
部分関数従属
複合主キーの一部だけで決まる従属。2NFで排除。
推移的関数従属
主キー→非キー→非キー、と間接的に決まる従属。3NFで排除。
正規化 / 非正規化
重複を消す分割 / あえて重複を許し結合を減らす崩し。
更新時異常
重複が原因の更新・挿入・削除時の矛盾や不都合。
ER図
テーブルと関係（1対多など）を表す設計図。
カーディナリティ
関係の多重度（1対1 / 1対多 / 多対多）。
連関テーブル
多対多を1対多×2に分解する中間テーブル。
インデックス
検索を高速化する索引。多くはB-tree構造。
実行計画 / オプティマイザ
SQLをどう実行するかの計画 / それを決めるDBMSの頭脳。
統計情報
データ分布の情報。古いと実行計画を誤る。
