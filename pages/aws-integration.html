CloudFront × ALB × Lambda
3 層を 1 つの構成につなぐ

三部作の総まとめ。これまで個別に見てきた 3 つのサービスは、「届ける・振り分ける・処理する」という役割で重なり合い、1 本の Web アプリ構成として組み合わさります。リクエストが端から端までどう流れるのかを通して見ていきましょう。

CloudFront 届ける ALB 振り分ける Lambda 処理する
01

3 層を 1 つにつなぐ

典型的な構成はこうなります。利用者の最前面に CloudFront が立ち、静的ファイルはエッジから配り、動的なリクエストだけを後段へ転送。その動的リクエストを ALB が受けて中身を見て振り分け、実際の処理を Lambda が実行して、データ層へ読み書きします。

02

それぞれの役割分担

3 つは競合せず、責務がきれいに分かれます。「外向き・入口・中身」と覚えると整理しやすいです。

LAYER 01 · 届ける

CloudFront

世界中のエッジから配信。静的をキャッシュして高速化し、オリジン負荷を減らす。TLS・WAF の入口も兼ねる。

LAYER 02 · 振り分ける

ALB

動的リクエストの入口。L7 でパスやホストを見て、適切なターゲット（ここでは Lambda）へ振り分ける。

LAYER 03 · 処理する

Lambda

実際のビジネスロジック。イベントに反応して起動し、データ層を読み書きして応答を返す。

分業の効用

各層が独立して別々にスケール・別々に保守できます。キャッシュは CloudFront、ルーティング変更は ALB、ロジック変更は Lambda、と変更の影響範囲が閉じるのが、この構成の大きな利点です。

03

リクエストの一気通貫フロー

1 本のリクエストが、どの層をどう通るか。静的と動的で道が分かれます。

1

利用者のリクエストが、最寄りの CloudFront エッジに届く。ここで TLS を終端し、WAF を適用。

2

静的（例 /static/app.js）なら、エッジのキャッシュにヒットすれば即返す。無ければ S3 から取得してキャッシュし返す。ALB も Lambda も動かない。

3

動的（例 /api/todos）なら、CloudFront はオリジンである ALB へ転送。ALB はリスナールールで /api/* を判定し、Lambda のターゲットグループを選ぶ。

4

ALB が Lambda を同期呼び出し。Lambda はリクエスト内容（イベント JSON）を受け取り、データ層を読み書きしてロジックを実行。

5

Lambda の応答 → ALB → CloudFront と逆順に戻り、利用者へ。キャッシュ可能な応答なら CloudFront が保持し、次回を高速化する。

要点

静的は CloudFront＋S3 で完結し、後段を一切使わない。動的だけが ALB→Lambda へ降りる。「重い処理はなるべく後段へ流さない」のが、この構成の速さとコスト効率の源泉です。

04

静的と動的の分岐（CloudFront の振る舞い）

CloudFront はパスごとに「振る舞い（ビヘイビア）」を設定でき、行き先（オリジン）とキャッシュ方針を変えられます。これが静的・動的を分ける要です。

パスパターンオリジンキャッシュ方針
/static/*, 画像, JS, CSSS3長め（積極的にキャッシュ）
/api/* など動的ALB原則オフ or 短め。ヘッダ・クッキー・クエリを転送

静的アセットは思い切りキャッシュして高速化、API はキャッシュせず（または短時間）に毎回 ALB→Lambda へ通す、という出し分けです。バージョン付きファイル名（app.v2.js）と組み合わせると、静的の更新反映も安全になります。

05

ALB → Lambda の連携（接着面の正体）

この構成の“接着剤”は、Lambda を ALB のターゲットに登録できること（ターゲットグループの種別を lambda にする）。ALB は Lambda を同期呼び出しし、HTTP リクエストを特定のイベント JSONに変換して渡します。

ALB が Lambda に渡すイベント（抜粋）

{
  "requestContext": { "elb": { "targetGroupArn": "arn:aws:...:targetgroup/..." } },
  "httpMethod": "GET",
  "path": "/api/todos",
  "queryStringParameters": { "limit": "10" },
  "headers": { "host": "example.com" },
  "body": "",
  "isBase64Encoded": false
}

Lambda 側は、ALB が解釈できる形で必ず特定の形の応答を返す必要があります。statusCode と body を含む JSON です。

Lambda が返すべき応答

{
  "statusCode": 200,
  "statusDescription": "200 OK",
  "headers": { "Content-Type": "application/json" },
  "isBase64Encoded": false,
  "body": "{\"items\":[...]}"
}
ポイント

API Gateway の Lambda 統合と似た発想ですが、イベントの形は ALB 専用（requestContext.elb が目印）。この形に沿って受け取り・返すだけで、ALB の後ろに Lambda がぶら下がります。前作「ALB」で触れた “ターゲット＝EC2 / IP / Lambda” の Lambda が、まさにこれです。

06

スケールと耐障害

3 層それぞれが、別の仕組みで伸び縮みし、別の方法で障害に耐えます。

  • CloudFront：エッジでキャッシュヒットした分は後段に届かない。アクセス増の多くをエッジで吸収し、オリジンを守る。
  • ALB：トラフィックに応じて自動スケール。複数 AZ のターゲットへ分散し、ヘルスチェックで不健全なターゲットを自動的に外す。
  • Lambda：同時実行ぶん環境を自動複製。サーバー台数を意識せず急増に追従（同時実行上限と後段 DB の接続数には注意）。
注意 — 後段がボトルネック

Lambda は急にスケールできても、その先の RDS などが同時接続数で詰まることがあります。RDS Proxy で接続をまとめる、DynamoDB を使う、同時実行数を予約する、などの対策を構成段階で検討しましょう。

07

セキュリティの重ね方

各層がそれぞれセキュリティの役割を持ち、多層防御になります。

  • エッジで弾く：CloudFront に WAF / Shield を付け、悪意あるリクエストや DDoS を最前線で遮断。TLS もここで終端。
  • S3 を隠す：OAC で S3 を非公開にし、CloudFront 経由のアクセスだけ許可。バケットを世界に晒さない。
  • ALB でも WAF：必要なら ALB にも Web ACL を付け、認証（Cognito/OIDC）を振り分け前に挟める。
  • 最小権限：Lambda は実行ロールで、触れる AWS リソースを必要最小限に。プライベートサブネット内のリソースには VPC 接続で。
08

構成のバリエーション

同じ「届ける・振り分ける・処理する」でも、要件によって部品の選び方が変わります。

パターン後段向いている場面
CloudFront → ALB → Lambdaサーバーレス断片的な API、アクセスにムラ、運用を最小化したい
CloudFront → ALB → ECS / EC2コンテナ / VM常時負荷が高い、長時間処理、既存アプリの移行
CloudFront → API Gateway → LambdaサーバーレスAPI 管理機能（認可・スロットリング・スキーマ）を厚く使いたい
ALB か API Gateway か

Lambda を HTTP で公開する入口は、ALB でも API Gateway でも可。すでに ALB で他の EC2/ECS も束ねているなら ALB に Lambda も相乗りさせるのが素直。API としての管理機能を重視するなら API Gateway が有力です。

連携の一言まとめ

CloudFront が静的を捌いて動的だけを通し、ALB が /api/* を Lambda ターゲットへ流し、Lambda がイベント JSON を受けて応答を返す。3 つの「型安全ならぬ役割安全」な分業で、速く・安全に・伸縮自在な Web 構成ができあがります。

本記事は AWS 公式ドキュメント（CloudFront / ALB / Lambda）をもとに、連携構成として再構成したものです。イベント／応答の形・仕様・料金は時期で変わるため、実装時は公式と手元の設定を確認してください。
