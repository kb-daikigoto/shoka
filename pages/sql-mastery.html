ループを捨て、
集合で考える。

SQLが伸び悩む最大の原因は、構文ではなく「考え方」です。1行ずつ処理する手続き型の発想を捨て、データを集合としてまるごと変換する発想へ。CASE式・NULL・EXISTS・ウィンドウ関数という武器を通して、その思考の転換をこれ一本で体得します。

PART 1

考え方を変える

テクニックの前に、SQLという言語の「世界観」を入れ替えます。ここが変わると、後半の武器がすべて自然につながります。

01

SQLは「集合」を扱う宣言型言語

SQLでは「どう処理するか(HOW)」ではなく「何が欲しいか(WHAT)」だけを書く。手順はDBMSが決める。

多くの人はC言語やPython、JavaScriptのような手続き型言語から入ります。そこでは「変数を用意し、ループを回し、1件ずつ処理する」のが当たり前です。ところがSQLはまったく別の血統 ―宣言型かつ集合指向の言語です。

DECLARATIVE

宣言型

「欲しい結果」を記述するだけ。実際の処理手順（どのインデックスを使い、どう結合するか）はDBMSが自動で決める。

SET-ORIENTED

集合指向

処理の単位は「1行」ではなく「行の集まり＝集合」。集合をまるごと別の集合へ変換する、と考える。

なぜ重要か

この本が一貫して伝えるのは「SQLが下手なのは構文を知らないからではなく、手続き型の頭で書いているから」という一点です。ループで考える限り、SQLはぎこちなく、遅く、長くなります。集合で考えられるようになると、同じ処理が驚くほど短く・速く書けます。本ガイドの全テクニックは、この発想転換の具体例にすぎません。

02

ループからの脱却

「1行ずつ if で振り分けて数える」という手続きは、GROUP BY と CASE式で一撃に置き換わる。

例として「県ごとに男女の人数を数えたい」という典型業務を考えます。手続き型の頭だと、こうなりがちです ― 全行をループし、県ごとの変数を用意し、性別を if で見て加算する。実装も長く、県が増えるたびに直す必要があります。

図: 同じ集計でも、手続き型は「縦の繰り返し」、集合指向は「集合の一括変換」
crosstab.sql ― ループ不要。GROUP BY + CASE で一発
SELECT
  県名,
  SUM(CASE WHEN 性別 = '男' THEN 人口 ELSE 0 END) AS 男性,
  SUM(CASE WHEN 性別 = '女' THEN 人口 ELSE 0 END) AS 女性
FROM 人口
GROUP BY 県名;
要点

ループ・一時変数・if分岐 ― 手続き型の三種の神器は、SQLでは GROUP BY・集約関数・CASE式 に対応します。「ループで書きたくなったら、それは集合演算で書けないか？」と一度立ち止まる。これが上達の最短ルートです。

PART 2

中核の武器

集合指向を実戦で使うための道具を、効果の大きい順に習得します。CASE式とNULLは全SQLの土台なので、ここは特に丁寧に。

03

CASE式 ― SQLの万能スイッチ

CASE式は「文」ではなく「値を返す式」。値を書ける場所ならどこにでも置ける。

多くの人はCASEを「SELECTで表示を切り替えるもの」と思っていますが、本質は値を返す式であることです。値が書ける場所 ―SELECT・WHERE・GROUP BY・ORDER BY・集約関数の中 ― すべてに書けます。これがCASEを万能たらしめています。

検索CASE と 単純CASE

case_basic.sql
-- 検索CASE：条件式を自由に書ける（こちらを基本に）
CASE WHEN 点数 >= 80 THEN 'A'
     WHEN 点数 >= 60 THEN 'B'
     ELSE 'C' END

-- 単純CASE：1つの値との一致のみ（簡潔だが表現力は低い）
CASE 性別 WHEN '1' THEN '男' WHEN '2' THEN '女' END

真価①：条件分岐の集計（行→列のピボット）

02章で見た SUM(CASE WHEN …) がそれです。集約関数の中にCASEを入れることで、行に縦持ちされたデータを横方向に集計できます。これは集合指向の典型的な威力です。

真価②：並び順を式で操る

case_order.sql ― 特定の値を優先して並べる
SELECT 商品名, 在庫
FROM 商品
ORDER BY
  CASE WHEN 在庫 = 0 THEN 0 ELSE 1 END,  -- 在庫切れを先頭に
  在庫 DESC;
tip ― 終端のENDを忘れない

CASEは END で閉じます（言語によくある END CASE ではない）。またELSEを省くと、どの条件にも当てはまらない行はNULLになります。意図しないNULLを防ぐため、原則 ELSE を明示しましょう。NULLの怖さは次章で。

04

NULLと3値論理 ― 最大の罠

NULLは「値」ではなく「不明」という状態。だからNULLとの比較は真でも偽でもなく「不明(UNKNOWN)」になる。

NULLは 0 でも空文字でもありません。「値が不明・未確定」という状態です。値ではないので、= NULL という比較は永遠に成立しません。結果は真(TRUE)でも偽(FALSE)でもなく、第3の値 UNKNOWN（不明） になります。これがSQLが3値論理と呼ばれる理由です。

AND真不明偽OR真不明偽
真真不明偽真真真真
不明不明不明偽不明真不明不明
偽偽偽偽偽真不明偽

そして重要な原則 ―WHERE句が行を通すのは「真(TRUE)」のときだけ。「不明」の行は、偽と同じように結果から除外されます。ここから有名な罠が生まれます。

null_trap.sql ― NOT IN にNULLが混じると全滅する
-- 危険：対象外IDの集合にNULLが1つでもあると、1件も返らない
SELECT * FROM 社員
WHERE 部署ID NOT IN (SELECT 部署ID FROM 対象外部署);

-- 安全：NOT EXISTS で書き換える（NULLに惑わされない）
SELECT * FROM 社員 e
WHERE NOT EXISTS (
  SELECT * FROM 対象外部署 t WHERE t.部署ID = e.部署ID
);
danger ― なぜ NOT IN は全滅するのか

x NOT IN (a, b, NULL) は x≠a AND x≠b AND x≠NULL に展開されます。最後の x≠NULL は必ず不明。「真 AND 不明」は不明になり（上の表）、WHEREは不明を通さないので、すべての行が消えます。サブクエリにNULLが混じりうるなら NOT IN は使わず NOT EXISTS、と覚えてください。

要点 ― NULLの鉄則

① NULLの判定は IS NULL / IS NOT NULL のみ（= NULL は不可）。② 必要なら COALESCE(列, 既定値) で早めにNULLを潰す。③ そもそも列に NOT NULL 制約を付け、NULLを生まない設計を優先する。

05

自己結合 ― 1つの表を2つに見立てる

同じテーブルに別名を2つ付けると、まるで2枚の表のように結合できる。組み合わせ・行間比較の定番技。

自己結合（self join）は、1つのテーブルに異なる別名（エイリアス）を与え、自分自身と結合する技法です。これにより「同じ集合の要素同士を突き合わせる」処理が書けます。よく使うのは 組み合わせ・順列の列挙 と 行間の比較 です。

非等値結合で「組み合わせ」を作る

商品のペアを重複なく列挙したいとき、結合条件に = ではなく < を使うのがコツです（非等値結合）。これで「自分自身のペア」と「順序違いの重複」を同時に排除できます。

self_join.sql
-- {A,A} や {りんご,みかん}と{みかん,りんご}の重複を < で防ぐ
SELECT p1.商品名 AS 商品A, p2.商品名 AS 商品B
FROM   商品 p1
JOIN   商品 p2 ON p1.商品名 < p2.商品名;
なぜ < なのか

= で結ぶと自分自身（A=A）ばかりになります。<>（不等）だと {A,B} と {B,A} の両方が出て重複します。< なら「名前の小さい方が左」というルールを1つに固定でき、組み合わせ（順序を区別しない）がちょうど1回ずつ得られます。順列（順序を区別する）が欲しければ <> を使います。

06

HAVING句 ― 集合そのものに条件をつける

WHEREは「1行」をふるいにかける。HAVINGは「グループ＝集合」をふるいにかける。

初学者がつまずくのが WHERE と HAVING の違いです。役割は明快で、WHEREは個々の行に対する条件、HAVINGはGROUP BYでまとめた後の集合に対する条件です。集約関数（COUNT, SUM…）を使った条件はHAVINGに書きます。

図: 処理順 ― WHERE(行を絞る) → GROUP BY(集合化) → HAVING(集合を絞る)
having.sql ― 重複データを集合の性質で見つける
-- 同じ氏名が2件以上ある「グループ」だけを残す
SELECT 氏名, COUNT(*) AS 件数
FROM   会員
GROUP BY 氏名
HAVING COUNT(*) >= 2;
要点

HAVINGは「集合に対する条件」を書く場所。データの歯抜けチェック（HAVING COUNT(*) <> MAX(連番) など）や、「条件を満たす要素だけからなる集合か」を問う 特性関数的な使い方もできます。集合をひとつのモノとして扱う感覚が、ここで鍛えられます。

07

EXISTS述語と量化 ― 「すべて」をどう書くか

SQLに「すべて(∀)」を直接書く構文はない。「〜でない行が存在しない」と裏返して NOT EXISTS で表す。

EXISTS は「サブクエリが1行でも返すか？」を判定する述語で、これは論理学でいう 存在量化（∃：〜が存在する） に対応します。一方、SQLには 全称量化（∀：すべての〜について） を直接書く構文がありません。

発想の核心 ― 裏返す

「すべてのxについてP(x)が成り立つ」は、論理的に「P(x)が成り立たないxは1つも存在しない」と同値です。SQLには∀がない代わりに∃（EXISTS）があるので、全称を NOT EXISTS で表現します。これは本書屈指の重要テクニックです。

図: 全称量化は「否定の存在しない」へ裏返して表現する
all_pass.sql ― 全科目に合格した学生を探す
SELECT DISTINCT 学生ID
FROM 成績 s1
WHERE NOT EXISTS (        -- 次のような行が「存在しない」
  SELECT * FROM 成績 s2
  WHERE s2.学生ID = s1.学生ID
    AND s2.点数 < 60      -- 不合格の科目
);
tip ― EXISTS は速くて安全

EXISTSは「1行見つかった時点で打ち切る」ため大規模データで効率的なことが多く、04章で見たNULLの罠も避けられます。「IN / NOT IN で詰まったら EXISTS / NOT EXISTS」は実務でも有効な定石です。

08

ウィンドウ関数 ― 行と行を比較する

GROUP BYは行を畳んでしまう。ウィンドウ関数は行を残したまま、各行に「集計値」や「順位」を添える。

「順位をつけたい」「累計を出したい」「前月と比べたい」 ― こうした行間比較は、かつて自己結合や相関サブクエリで苦労して書いていました。ウィンドウ関数はこれを宣言的に、しかも行を減らさずに実現します。構文の核は OVER (PARTITION BY … ORDER BY …) です。

図: GROUP BYは行を集約して減らす／ウィンドウ関数は行を保ち列を足す
window.sql ― 部署ごとの順位と合計を同時に出す
SELECT
  社員名, 部署, 給与,
  RANK() OVER (PARTITION BY 部署 ORDER BY 給与 DESC) AS 部署内順位,
  SUM(給与) OVER (PARTITION BY 部署)               AS 部署合計
FROM 社員;

PARTITION BY は集計の「区切り」（GROUP BYに似るが行は減らない）、ORDER BY は窓の中での「順序」です。代表的な関数を押さえましょう。

関数用途
ROW_NUMBER()単純な連番（同値でも別番号）
RANK()順位（同値は同順位・次を飛ばす：1,1,3）
DENSE_RANK()順位（同値は同順位・飛ばさない：1,1,2）
SUM/AVG() OVER累計・移動平均などの集計
LAG()/LEAD()前の行／次の行の値を参照（前月比など）
lag.sql ― 前月との差を出す（行間比較の宣言的な書き方）
SELECT 月, 売上,
  売上 - LAG(売上) OVER (ORDER BY 月) AS 前月差
FROM 月次売上;
要点

「明細も見たいが集計も添えたい」「順位や前後比較がしたい」ときはウィンドウ関数。自己結合や相関サブクエリで書いていた行間比較の多くが、これで簡潔かつ高速になります。集合指向の発想が最も鮮やかに効く道具です。

09

相関サブクエリ ― 行ごとの小さな問い合わせ

外側の各行を参照しながら、その行ごとにサブクエリを評価する。「行と集合」を結びつける橋。

通常のサブクエリは1回だけ評価され、外側とは独立です。相関サブクエリは内側から外側の行を参照するため、外側の行ごとに評価されます。「各行を、それが属するグループの集計と比べる」といった処理の基礎になります。

correlated.sql ― 自部署の平均より給与が高い社員
SELECT 社員名, 部署, 給与
FROM   社員 e1
WHERE  給与 > (
  SELECT AVG(給与) FROM 社員 e2
  WHERE  e2.部署 = e1.部署   -- 外側の行(e1)を参照＝相関
);
tip ― 今ならウィンドウ関数も候補

上の例はウィンドウ関数 AVG(給与) OVER (PARTITION BY 部署) でも書けます。相関サブクエリは行ごとに繰り返し評価され重くなりがちなので、同じことができるならウィンドウ関数の方が読みやすく速いことが多いです。とはいえ相関サブクエリは思考の土台なので、両方書けるようにしておくと理解が深まります。

10

集合演算 ― UNION / INTERSECT / EXCEPT

テーブルを「集合」として、和・積・差で組み合わせる。集合指向の最も直接的な表現。

SQLは集合の言語なので、数学の集合演算をそのまま持っています。和（UNION）・積（INTERSECT）・差（EXCEPT）です。2つのSELECT結果を1つの集合として束ねたり、比べたりできます。

図: 2つのSELECT結果を集合として 和・積・差 で組み合わせる
setops.sql ― 2つの表の差分を取る
-- 旧名簿にいて新名簿にいない人（＝退会者）を抽出
SELECT 会員ID FROM 旧名簿
EXCEPT
SELECT 会員ID FROM 新名簿;
warn ― UNION と UNION ALL / DBMS差

UNION は重複を除くため内部でソートが走り遅くなります。重複を気にしない（または起きない）なら、ソートしない UNION ALL の方が速いです。また INTERSECT / EXCEPT は対応しないDBMSやバージョンがあり、名称も異なる場合があります（差を MINUS と呼ぶ製品など）。使う前にお使いのDBMSの最新ドキュメントで確認してください。

なお「ある集合の要素をすべて含む行を探す」関係除算（division）も集合演算の一種で、07章の NOT EXISTS の二重否定で表現できます。「全称量化＝NOT EXISTS」がここでも効いてきます。

PART 3

仕上げ ― 速く、読みやすく

正しく書けるようになったSQLを、現場で通用する品質に引き上げます。性能の勘所と、読みやすさの作法です。

11

SQLを速くする ― 性能の勘所

速いSQLの基本は「無駄なソートを減らす」「インデックスを活かす」「中間結果を小さく保つ」。

無駄なソートを削る

DISTINCT・UNION・ORDER BY・GROUP BY は暗黙のソートを伴う。重複除去が不要なら UNION ALL を選ぶ。

インデックスを殺さない

WHEREでインデックス列を加工しない（WHERE 関数(列)=…は索引が効きにくい）。LIKE '%語' の前方ワイルドカードも索引が効かない。

EXISTSを活かす

存在確認は IN より EXISTS が有利なことが多く、見つかり次第打ち切れる。NULLの罠も回避できる。

早く絞る

結合や集計の前に、WHEREで対象行をできるだけ減らす。中間結果が小さいほど後段が軽くなる。

なぜ ― 最後は実測

性能の最終判断は推測ではなく実行計画の確認と実測です（多くのDBMSの EXPLAIN 等で確認できます）。DBMSやデータ量で最適な書き方は変わるため、「この書き方が常に速い」と決めつけないこと。前作のDB設計編で触れたインデックス・統計情報の話と、ここはひと続きです。

12

SQLプログラミング作法

SQLは「あとで他人（と未来の自分）が読む文書」。集合指向で短く、構造を見せて書く。

動くだけのSQLと、保守できるSQLは違います。最後に、達人が共有する作法をまとめます。

段階はCTEで名付ける

入れ子サブクエリを深く重ねるより、WITH句（CTE）で各段に名前を付けると、処理の流れが上から読める。

ループ的発想を疑う

カーソルで1行ずつ回す前に、CASE式・ウィンドウ関数・集合演算で書けないか考える（01〜10章の総まとめ）。

NULLを明示的に扱う

COALESCE で既定値を与える、NOT NULL 制約で源を断つ。NULLを放置しない。

標準SQLを意識

製品独自の構文に頼りすぎない。移植性と可読性のため、まずは標準的な書き方を選ぶ。

cte.sql ― WITH句で処理を段階的に名付ける
WITH 部署別平均 AS (
  SELECT 部署, AVG(給与) AS 平均給与
  FROM 社員 GROUP BY 部署
)
SELECT e.社員名, e.給与, a.平均給与
FROM 社員 e
JOIN 部署別平均 a ON e.部署 = a.部署
WHERE e.給与 > a.平均給与;
本書を貫く一文

結局すべては01章に戻ります ―「ループで考えず、集合で考える」。CASE式もEXISTSもウィンドウ関数も、この一つの発想を実現するための道具でした。構文を増やすより、この見方を体に入れることが、達人への最短距離です。

ウィンドウ関数で集計を区切る単位（GROUP BY似だが行は減らない）。
ランキング関数
ROW_NUMBER / RANK / DENSE_RANK など順位付けの関数。
集合演算
UNION(和)・INTERSECT(積)・EXCEPT/MINUS(差)。表を集合として組み合わせる。
関係除算
「指定集合の要素をすべて含む」行を求める演算。NOT EXISTSの二重否定で表現可。
CTE (WITH句)
クエリ内で名前付きの一時結果を定義する仕組み。可読性を高める。
