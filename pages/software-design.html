ちょうぜつソフトウェア設計入門 ― 完全理解ガイド

動くコードの先へ。
変更に強い設計を。

コードは「動けば終わり」ではありません。要求は必ず変わり、その変更を安全に・楽にできるかどうかが設計の良し悪しを決めます。クリーンアーキテクチャを目的の絵として掲げ、オブジェクト指向・SOLID・DI・TDD・デザインパターンを、すべて「依存をどう制御するか」という一本の糸で貫いて理解します。例はPHPです。

PART 1

目的の絵を先に見る

本書は珍しく「クリーンアーキテクチャ」という完成形から話を始めます。ゴールの絵を先に持つと、後の原則がすべて「そこへ至る手段」として腑に落ちます。

01

良い設計とは何か

良い設計とは「正しく動くこと」ではなく「変更を、安全に・楽にできること」である。

初学者がまず越えるべき壁は、「動くコード＝良いコード」という思い込みです。動くだけなら設計はいりません。問題は、ソフトウェアが soft（やわらかい＝変わる前提のもの） だということです。仕様は追加され、要求は覆り、外部サービスは差し替わります。

つまり変更は例外ではなく前提。だから設計の目的はただ一つ ― その避けられない変更を、こわごわではなく安心して行える状態をつくることです。本書のすべての技法は、この一点に奉仕します。

悪い兆候

変更が怖い

一箇所直すと、関係なさそうな別の場所が壊れる。どこに影響するか読めない。＝結合が強すぎる。

良い兆候

変更が局所的

仕様変更が、関係する1〜2クラスの修正で収まる。影響範囲が予測できる。＝結合が弱く凝集が高い。

なぜ重要か ― 結合度と凝集度

設計の良し悪しは究極的に2語で測れます。結合度（モジュール同士の依存の強さ）は低く、凝集度（1モジュール内のまとまりの強さ）は高く。以降に出てくるSOLIDもDIもパターンも、要は「結合を弱め、凝集を高める」ための具体策です。この物差しを常に手元に置いてください。

02

クリーンアーキテクチャ ― 依存は内側へ

UIもDBもフレームワークも「詳細」にすぎない。中心の業務ルールを、詳細から独立させる。

クリーンアーキテクチャ（ロバート・C・マーチンが整理）は、システムを同心円で描きます。中心ほど「アプリの本質（業務ルール）」、外側ほど「技術的な詳細（画面・DB・Web）」です。そして唯一にして絶対のルールがあります。

図: 依存性のルール ― 矢印（依存）は常に外側から内側へ。内側は外側を知らない
依存性のルール（The Dependency Rule）

ソースコードの依存は、必ず外側から内側へ向ける。内側の円は、外側の円のことを一切知ってはならない。 中心の業務ルールは「どのDBか」「どの画面か」「どのフレームワークか」を知りません。だからそれらを差し替えても、中心は1行も壊れません。

ここが初学者の常識をひっくり返すポイントです。多くの人はDBやフレームワークを設計の「主役」だと思っています。本書は逆だと言います ―DBもWebも「詳細」であり、後から決めて差し替えてよい脇役。主役はアプリ固有の業務ルールです。

要点 ― 形より向き

同心円の絵を丸暗記する必要はありません。本質は「依存の矢印の向きを内側へ制御する」こと。クリーンアーキテクチャは目的ではなく、依存をきちんと制御した結果として現れる形です。ではどうやって矢印の向きを操るのか? その道具がPart 2のオブジェクト指向と原則です。

PART 2

部品 ― オブジェクト指向と原則

「依存の向きを制御する」を実現する道具箱。オブジェクト指向の本当の役割と、それを言語化したSOLID原則を学びます。

03

オブジェクト指向の本当の意味

オブジェクト指向は「現実を写し取る道具」ではなく「依存を制御し、変更に強くする道具」である。

「OOPとは現実世界をモデル化すること」「クラスとは“もの”」 ― この説明だけだと、なぜ便利なのかが分かりません。本書はもっと実用的に捉えます。OOPの価値は変更に強い構造をつくれること。その手段が三大要素です。

隠す

カプセル化

データと振る舞いを1つにまとめ、内部実装を外から隠す。外部は「使い方」だけを知り、内部の都合に依存しない。

差し替える

多態性（ポリモーフィズム）

同じ呼び出し方で、相手によって違う振る舞いをさせる。本書が最重視する要素。

注意して使う

継承

共通部分を親に括り出す。便利だが結合を強めやすく、濫用は禁物。多くの場合コンポジションの方が安全。

warn ― 継承よりコンポジション

初学者はつい「コードの共有＝継承」と考えますが、安易な継承は親子をきつく縛り、変更に弱くします。「AはBの一種か?（is-a）」が本当に成り立つ時だけ継承を使い、それ以外は必要な相手を持たせて使う（コンポジション）方が、結合が弱く柔軟です。

04

多態性こそ主役 ― 依存を逆転させる鍵

呼び出す側を「抽象（インターフェース）」だけに依存させれば、具体的な相手を自由に差し替えられる。

クリーンアーキテクチャの「依存を内向きにする」を実際に可能にするのが、多態性です。仕組みはこうです ― 呼び出す側はインターフェース（抽象）だけを知り、その背後にどんな具体実装が来るかを知りません。だから具体は後から差し替え自由です。

Notifier.php ― 抽象に依存し、具体を差し替える
// 抽象：通知の「使い方」だけを定義
interface Notifier {
    public function send(string $message): void;
}

// 具体①
class EmailNotifier implements Notifier {
    public function send(string $message): void { /* メール送信 */ }
}
// 具体②（後から追加。呼び出し側は変えなくてよい）
class SlackNotifier implements Notifier {
    public function send(string $message): void { /* Slack送信 */ }
}

// 呼び出し側は Notifier(抽象) しか知らない
function notifyUser(Notifier $notifier): void {
    $notifier->send('ようこそ！');
}
図: 呼び出し側は抽象だけに依存。具体実装は抽象を実装する側＝矢印が逆向きになる
要点 ― これが「依存性の逆転」

普通なら「使う側 → 具体クラス」と依存しますが、間に抽象を挟むと、具体クラスの方が「抽象 ← 具体」と逆向きに依存します。この向きの反転こそ、02章で見た“依存を内向きに制御する”の正体です。次章のSOLIDの「D」で正式に名前が付きます。

05

SOLID原則 ― 設計の5つの指針

SOLIDは「結合を弱め、変更に強くする」ための5つの合言葉。バラバラの規則ではなく、互いに支え合う。

頭文字原則一言でいうと
S単一責任クラスが変更される理由は1つだけにする
O開放閉鎖拡張には開き、修正には閉じる
Lリスコフ置換派生型は基底型と入れ替えても壊れない
Iインターフェース分離使わないメソッドに依存させない
D依存性逆転具体ではなく抽象に依存する

S ― 単一責任原則（SRP）

「責任」とは変更理由のこと。1つのクラスが「請求書の計算」と「画面表示」と「DB保存」を抱えると、3つの別々の理由で書き換えられ、互いに巻き添えになります。理由ごとにクラスを分ければ、変更が局所化します。

O ― 開放閉鎖原則（OCP）

新機能を足すとき、既存コードを書き換えず、新しいコードを追加するだけで済むのが理想。04章の通知の例がまさにそれ ― Slack対応は新クラスを足すだけで、呼び出し側は無修正でした。多態性がOCPを実現します。

L ― リスコフの置換原則（LSP）

派生クラスは、基底クラスを期待している場所にそのまま入れても正しく動くべき、という約束。継承で親の「契約」を破る（例外を投げる、戻り値の意味を変える）と、これが崩れてバグの温床になります。03章の「継承は慎重に」の理論的裏付けです。

I ― インターフェース分離原則（ISP）

巨大なインターフェースを1つ持つより、役割ごとに小さなインターフェースに分ける。使わないメソッドにまで依存させられると、関係ない変更の影響を受けてしまうからです。

D ― 依存性逆転原則（DIP）

上位（業務ロジック）も下位（DBアクセス等）も、互いにではなく「抽象」に依存せよ。これがクリーンアーキテクチャの矢印を内向きにする理論的核心であり、次章のDIで実装手段を得ます。

要点

5つは独立した暗記項目ではありません。SRPで責任を分け、ISPで窓口を小さく保ち、DIPで抽象に向け、OCPで拡張可能にし、LSPでその安全を保証する ― すべて「結合を弱め変更に強くする」一つの目的の別の顔です。

06

パッケージ原則 ― クラスより大きな単位

SOLIDがクラスの話なら、パッケージ原則は「クラスの集まり（モジュール）」をどう束ね、どう依存させるかの話。

規模が大きくなると、クラス単位の整頓だけでは足りません。パッケージ（モジュール）同士の関係を整える原則があります。初学者はまず結合に関する2点だけ押さえれば十分です。

ADP

循環依存を作らない

AがBに、BがAに依存する循環は禁物。どちらも単独でビルド・テスト・理解できなくなる。依存の矢印は一方通行に。

SDP

安定したものに依存する

変わりにくい（多くから依存される）安定パッケージに向けて依存する。コロコロ変わるものに依存しない。

なぜ抽象は安定なのか（SAP）

「安定したものに依存せよ」と「変化に対応せよ」は一見矛盾します。これを解くのが「安定したものほど抽象的にする」という考え方です。抽象（インターフェース）は実装の詳細を含まないので滅多に変わらない＝安定。だから安心して依存できる。DIPで抽象に依存したのは、パッケージ視点でも正解だった、というわけです。

PART 3

実践 ― 作りながら設計する

原則を現場で動かす具体的な技。DIで抽象依存を実装し、TDDで設計を駆動し、パターンで語彙を増やし、アジャイルで設計を育てます。

07

依存性注入（DI）

必要な部品を自分で new せず、外から「渡してもらう」。これだけでDIPが実装できる。

DIP（抽象に依存せよ）を実際のコードにする最も簡単な方法がDI（依存性注入）です。コツはたった一つ ―クラスの中で具体クラスを new しない。代わりに、必要なものをコンストラクタで受け取ります。

UserService.php ― new するか、注入されるか
// ✘ 悪い例：内部で具体クラスを new → EmailNotifierに固定され差し替え不能
class UserService {
    public function register(): void {
        $notifier = new EmailNotifier();   // 具体に依存…
        $notifier->send('登録完了');
    }
}

// ✔ 良い例：抽象を外から注入 → どの通知手段でも差し替え自由＆テスト容易
class UserService {
    public function __construct(private Notifier $notifier) {}

    public function register(): void {
        $this->notifier->send('登録完了');
    }
}
tip ― DIコンテナは「組み立て係」

「では誰が具体クラスを new するの?」という疑問の答えがアプリ最外周の組み立て役（しばしばDIコンテナ）です。どの抽象にどの具体を結びつけるかを一箇所で決め、必要な場所へ注入します。業務ロジックの内側は最後まで具体を知らずに済み、矢印は内向きのまま保たれます。

08

テスト駆動開発（TDD）

テストを先に書く。テストは品質保証だけでなく、「使いやすい設計」を引き出す設計ツールでもある。

TDDは「失敗するテストを先に書く → 通す最小限の実装をする → 整える」を小さく繰り返す進め方です。Red → Green → Refactor の三拍子で回します。

図: Red（失敗するテスト）→ Green（通す）→ Refactor（整える）を小さく回す
なぜ設計が良くなるのか

テストを先に書くと、まず「このクラスはどう使われるか（外から見たインターフェース）」を考えることになります。さらにテストしやすくするには、依存を外から差し替えられる必要があり、自然とDI・疎結合・小さな責任へ向かいます。つまりTDDは、これまで学んだ原則を否応なく実践させる仕組みなのです。Refactorの安全網があるから、設計を後から育てられます。

09

デザインパターン ― 定番解と共通語彙

パターンは「よくある設計問題への定番の解き方」。暗記対象ではなく、原則が形になった実例。

デザインパターン（GoF＝Gang of Fourの23種が有名）は、先人が繰り返し出会った問題への定石集です。価値は2つ ― 車輪の再発明を避けられること、そして「ここはStrategyで」と一言で意図を伝えられる共通語彙になること。

振る舞い

Strategy

アルゴリズムをインターフェースで抜き差しする。04章の通知の例がまさにこれ。OCPの典型。

生成

Factory

オブジェクトの生成手順を専用の場所に隔離し、利用側を生成の詳細から切り離す。

構造

Adapter

既存クラスの「形が合わない」インターフェースを、欲しい形に変換して繋ぐ。

振る舞い

Observer

状態変化を、依存する相手へ自動で通知する。結合を弱く保ったまま連動させる。

要点 ― パターンは結果であって目的ではない

本書のスタンスは明快です。「パターンを当てはめよう」と先に考えるのは順序が逆。SOLIDに従って素直に設計すると、結果として既知のパターンの形になっていることが多い。だから名前を覚えるより、「なぜその形が良いのか（どの原則の現れか）」を理解することが大切です。

10

アジャイルと設計 ― 設計は育てるもの

良い設計は一度では完成しない。変更に強い設計と、反復して育てる進め方は、表裏一体である。

最後のピースは「進め方」です。要求が変わり続ける以上、最初に完璧な設計を描き切ることはできません。だから少しずつ作り、動かし、学び、また直す ― この反復（アジャイル）が前提になります。そしてここまでの全技法が、この反復を支えています。

なぜ全部つながるのか

変更に強い設計（クリーンアーキテクチャ・SOLID）があるから、安心してリファクタリングできる。TDDのテストが安全網になるから、こわごわせず直せる。DIで疎結合だから、部品を差し替えられる。つまり「変更に強い設計」と「反復して育てる開発」は、互いを可能にし合う関係です。本書が最初にゴールの絵（02章）を見せたのは、このためでした。

warn ― 過剰設計を避ける（YAGNI）

「将来きっと必要になる」と先回りして抽象や仕組みを盛りすぎると、かえって複雑で変更しにくくなります。YAGNI（You Aren't Gonna Need It：たぶん要らない）。今必要な分だけ素直に作り、必要になった時に育てる。これも立派な設計判断です。
