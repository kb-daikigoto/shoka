sqlc 完全入門
— そして gqlgen との連携

「SQL を書いたら、型安全な Go のデータアクセスコードが自動で生えてくる」。
ORM の魔法に頼らず、生の SQL の力を保ったまま型安全になる sqlc を、完全初心者でも なぜそうなるのか を理解しながら手を動かして学びます。後半では、前回作った gqlgen の todo API と どう連携させるかまで踏み込みます。

所要時間 約 50 分 前提 Go ＋ sqlc DB PostgreSQL 仕上げ gqlgen 連携
01

sqlc とは何か

Go でデータベースを扱うとき、大きく 2 つの道があります。ひとつは ORM（GORM など）。SQL を隠してメソッド呼び出しに置き換えてくれますが、何が実行されるか見えにくく、複雑なクエリで詰まりがちです。もうひとつは 生の SQL を手で書く道。自由ですが、結果を Go の構造体へ手作業で詰め替える退屈な定型コードがついて回ります。

sqlc はそのちょうど中間です。あなたは普通の SQL を書く。sqlc がそれを読んで、型安全な Go コードを生成する。SQL は隠れず、定型コードは消える。いいとこ取りです。

gqlgen と同じ匂い

前回の gqlgen を覚えていれば、もう本質は掴めています。gqlgen が「GraphQL スキーマ → Go コード生成」だったのに対し、sqlc は「SQL → Go コード生成」。どちらもあなたが宣言を書き、機械が定型コードを吐く「コード生成」系の道具です。学ぶリズムは同じです。

02

sqlc を貫く 3 つの考え方

IDEA 01

SQL ファースト

先に「テーブル定義（schema.sql）」と「クエリ（query.sql）」を書く。アプリのコードはその後。設計図が先、実装が後 ―― gqlgen と同じ順番。

IDEA 02

コード生成

sqlc generate を走らせると、SQL を解析して構造体とメソッドを吐き出す。あなたは生成された関数を呼ぶだけ。

IDEA 03

クエリ注釈

各 SQL の上に -- name: ... :one のような注釈を付ける。これが「関数名」と「返り方（1件 / 複数 / 実行のみ）」を決める。

03

全体像 — 一枚の地図

sqlc の流れはとてもシンプルです。2 つの入力（テーブル定義とクエリ）から、2 つの出力（構造体と Queries 型）が生まれ、あなたはその Queries のメソッドを呼ぶだけ。

gqlgen との対比が効きます。gqlgen はあなたに穴（リゾルバ）を埋めさせますが、sqlc はメソッドを完成形で渡してくるので、あなたは呼ぶだけ。だから後半で見るように、gqlgen のリゾルバの中で sqlc のメソッドを呼ぶ、という連携がきれいにハマります。

04

準備するもの

必要なのは Go と sqlc 本体です。sqlc は依存ゼロの単体バイナリなので、お好きな方法で入れてください。

# macOS (Homebrew)
$ brew install sqlc

# Go でインストール（Go 1.17+）
$ go install github.com/sqlc-dev/sqlc/cmd/sqlc@latest

# Docker で使う場合
$ docker run --rm -v $(pwd):/src -w /src sqlc/sqlc generate

入ったか確認します。

$ sqlc version
ここが嬉しいポイント

sqlc generate はデータベースに接続しません。あなたが書いた schema.sql を読んで型を理解するだけなので、DB が起動していなくてもコード生成できます。実際に「実行」するときだけ DB が要ります（PostgreSQL が手元にあると後半が動かせます）。

05

手を動かす① sqlc 単体

公式チュートリアルの「authors（著者）」テーブルを題材に、生成の一周を体験します。上から順にコピペしてください。

1

プロジェクトと設定ファイルを作る

$ mkdir sqlc-tutorial
$ cd sqlc-tutorial
$ go mod init tutorial.sqlc.dev/app

つづいて、生成の設定 sqlc.yaml を作ります。「どの DB か」「SQL ファイルの場所」「出力先パッケージ」を宣言するだけです。

sqlc.yamlyaml
version: "2"
sql:
  - engine: "postgresql"
    queries: "query.sql"
    schema: "schema.sql"
    gen:
      go:
        package: "tutorial"
        out: "tutorial"
        sql_package: "pgx/v5"
2

テーブル定義（schema.sql）を書く

sqlc に「どんな列があるか」を教えるためのファイルです。これが型の出どころになります。

schema.sqlsql
CREATE TABLE authors (
  id   BIGSERIAL PRIMARY KEY,
  name text      NOT NULL,
  bio  text
);
3

クエリ（query.sql）を書く

各 SQL の頭に -- name: 注釈を付けます。注釈が生成される関数の名前と形を決めます。

query.sqlsql
-- name: GetAuthor :one
SELECT * FROM authors
WHERE id = $1 LIMIT 1;

-- name: ListAuthors :many
SELECT * FROM authors
ORDER BY name;

-- name: CreateAuthor :one
INSERT INTO authors (
  name, bio
) VALUES (
  $1, $2
)
RETURNING *;

-- name: UpdateAuthor :exec
UPDATE authors
  set name = $2,
  bio = $3
WHERE id = $1;

-- name: DeleteAuthor :exec
DELETE FROM authors
WHERE id = $1;
注釈の読み方

:one＝1 行を構造体で返す、:many＝複数行をスライスで返す、:exec＝行を返さず実行だけ。$1 $2 は SQL のプレースホルダ（引数）で、これが Go 関数の引数になります。

4

生成する

$ sqlc generate
こう見えるはず
何も出力されなければ成功です（エラー時だけメッセージが出ます）。tutorial/ フォルダに 3 つの Go ファイルが生まれます。
├── go.mod
├── query.sql
├── schema.sql
├── sqlc.yaml
└── tutorial
    ├── db.go          ← 接続をラップする土台
    ├── models.go      ← Author 構造体（schema.sql 由来）
    └── query.sql.go   ← Queries 型と各メソッド（query.sql 由来）
5

生成物をのぞいて、呼んでみる

tutorial/query.sql.go を開くと、ListAuthors や CreateAuthor といったメソッドが、あなたの SQL から作られているのが分かります。これを呼ぶプログラムを書きます。

tutorial.go（新規作成）go
package main

import (
	"context"
	"log"

	"github.com/jackc/pgx/v5"
	"github.com/jackc/pgx/v5/pgtype"

	"tutorial.sqlc.dev/app/tutorial"
)

func run() error {
	ctx := context.Background()
	conn, err := pgx.Connect(ctx, "postgres://user:pass@localhost:5432/mydb")
	if err != nil {
		return err
	}
	defer conn.Close(ctx)

	queries := tutorial.New(conn)

	// 1 件作る（:one なので作った行が返る）
	author, err := queries.CreateAuthor(ctx, tutorial.CreateAuthorParams{
		Name: "Brian Kernighan",
		Bio:  pgtype.Text{String: "Go の共著者", Valid: true},
	})
	if err != nil {
		return err
	}
	log.Println(author)

	// 一覧を取る（:many なのでスライスが返る）
	authors, err := queries.ListAuthors(ctx)
	if err != nil {
		return err
	}
	log.Println(authors)
	return nil
}

func main() {
	if err := run(); err != nil {
		log.Fatal(err)
	}
}
$ go get github.com/jackc/pgx/v5
$ go build ./...
こう見えるはず
エラーなくビルドが通れば成功です。実際に動かすには、稼働中の PostgreSQL と authors テーブル、正しい接続文字列が必要です。コンパイルが通る＝ SQL と Go の型が一致していること自体が、sqlc が守ってくれている安全網です。
チェックポイント

SQL を 1 つ書き換える（例: ListAuthors に WHERE bio IS NOT NULL を足す）→ sqlc generate → ビルド、をやってみてください。SQL の typo はビルド時に捕まる——実行して初めて落ちる ORM との決定的な違いがここです。

06

理屈 — 注釈の意味と「なぜ DB なしで生成できるのか」

よく使う注釈を押さえておきましょう。返り方を決める大事な部分です。

注釈意味 / 返り値
:one1 行を構造体で返す（見つからなければエラー）
:many複数行をスライスで返す
:exec行を返さず実行のみ（INSERT/UPDATE/DELETE）
:execrows実行して「影響した行数」を返す
:batchexec などまとめて実行するバッチ系（応用）
理屈 — gqlgen の「型解析」とそっくり

なぜ DB に繋がず生成できるのか? sqlc は schema.sql を読んで各列の型を知り、query.sql の SQL を構文解析して「この SELECT は何の列を、どんな形で返すか」を静的に割り出します。つまり SQL を実行せず、文章として解析しているのです。

これは前回の gqlgen の工程③「Go コードを実際に型解析する」と同じ発想。どちらも「実行せずソースを解析して、型の辻褄を生成時に保証する」——コード生成系ツールに共通する強さです。

07

gqlgen と組む理由

ここからが本題の連携です。なぜこの 2 つを組み合わせるのか。理由はシンプルで、担当する「層」が隣り合っていて、性格がそっくりだからです。

  • 層が補完的 ── gqlgen は外向きの API 層（GraphQL）。sqlc は内向きの データ層（DB アクセス）。間にぽっかり空く「リゾルバの中身」を、sqlc のメソッド呼び出しで埋められます。
  • 思想が同じ ── どちらも「宣言を書く → 生成 → 使う」。ORM のような隠蔽（マジック）がなく、何が動くか見える。学習コストも二重にならない。
  • 端から端まで型安全 ── DB のカラム型 →（sqlc）→ Go 構造体 →（あなたのマッピング）→ GraphQL モデル →（gqlgen）→ API レスポンス。一直線に型でつながり、どこかがズレればコンパイルで止まります。
08

連携の全体像

イメージは「2 段重ね」です。上に gqlgen、下に sqlc、一番下に PostgreSQL。クライアントのクエリは上から下へ降り、行データが同じ道を上って返ります。

つまり連携の正体は、gqlgen のリゾルバ関数の中で、sqlc の Queries のメソッドを呼ぶこと。前回「穴を埋める」と言ったあの中身が、今度は本物の DB アクセス（sqlc）になるわけです。

09

手を動かす② gqlgen × sqlc

前回の続き

このパートは前回作った gqlgen-todos プロジェクト（モジュール名 example.com/gqlgen-todos）の上に乗せます。あの todo API の「メモリ保存」を、sqlc 経由の PostgreSQL 保存に差し替えるイメージです。

1

テーブルとクエリを SQL で定義

プロジェクト内に db/ フォルダを作り、2 つの SQL を置きます。

db/schema.sqlsql
CREATE TABLE users (
  id   text PRIMARY KEY,
  name text NOT NULL
);

CREATE TABLE todos (
  id      text    PRIMARY KEY,
  text    text    NOT NULL,
  done    boolean NOT NULL DEFAULT false,
  user_id text    NOT NULL REFERENCES users(id)
);
db/query.sqlsql
-- name: ListTodos :many
SELECT * FROM todos ORDER BY id;

-- name: CreateTodo :one
INSERT INTO todos (id, text, user_id)
VALUES ($1, $2, $3)
RETURNING *;

-- name: GetUser :one
SELECT * FROM users WHERE id = $1;
2

sqlc.yaml を置いて生成

出力先を gqlgen のパッケージ内（graph/db）にして、両者を同じツリーに同居させます。

sqlc.yaml（プロジェクト直下）yaml
version: "2"
sql:
  - engine: "postgresql"
    queries: "db/query.sql"
    schema: "db/schema.sql"
    gen:
      go:
        package: "db"
        out: "graph/db"
        sql_package: "pgx/v5"
$ sqlc generate
$ go get github.com/jackc/pgx/v5
こう見えるはず
graph/db/ に db.go / models.go / query.sql.go が生成され、db.Queries 型や db.CreateTodoParams、db.Todo、db.User が使えるようになります。
3

Resolver に「DB の入口」を持たせる

gqlgen の graph/resolver.go を、メモリの todos ではなく sqlc の *db.Queries を持つ形に変えます。これがアプリ全体の依存（DB）の置き場所です。

graph/resolver.gogo
package graph

import "example.com/gqlgen-todos/graph/db"

type Resolver struct {
	Queries *db.Queries
}
4

server.go で DB プールを作って注入

生成済みの server.go の中で、Resolver を作っている &graph.Resolver{} の箇所を、DB を渡した形に置き換えます。接続文字列は環境変数から読むのが定番です。

server.go（要点だけ抜粋）go
// import に追加
//   "os"
//   "github.com/jackc/pgx/v5/pgxpool"
//   "example.com/gqlgen-todos/graph/db"

pool, err := pgxpool.New(context.Background(), os.Getenv("DATABASE_URL"))
if err != nil {
	log.Fatal(err)
}
defer pool.Close()

// もとの &graph.Resolver{} をこれに置き換える
resolver := &graph.Resolver{Queries: db.New(pool)}

// あとは生成コードのまま resolver を Resolvers に渡す
//   ...Config{Resolvers: resolver}...
起動前に

環境変数 DATABASE_URL（例: postgres://user:pass@localhost:5432/todos）を設定し、db/schema.sql の 2 テーブルを実 DB に作っておきます（マイグレーションツール golang-migrate などを使うのが定番）。

5

リゾルバを「sqlc 呼び出し ＋ マッピング」で実装

いよいよ連携の心臓部です。各リゾルバが sqlc のメソッドを呼び、返ってきた sqlc の行（db.Todo）を GraphQL モデル（model.Todo）に詰め替えます。

graph/schema.resolvers.gogo
func (r *queryResolver) Todos(ctx context.Context) ([]*model.Todo, error) {
	rows, err := r.Queries.ListTodos(ctx)        // ← sqlc を呼ぶ
	if err != nil {
		return nil, err
	}
	todos := make([]*model.Todo, len(rows))
	for i, row := range rows {                   // ← db.Todo を model.Todo へ詰め替え
		todos[i] = &model.Todo{
			ID:     row.ID,
			Text:   row.Text,
			Done:   row.Done,
			UserID: row.UserID,
		}
	}
	return todos, nil
}

func (r *mutationResolver) CreateTodo(ctx context.Context, input model.NewTodo) (*model.Todo, error) {
	n, _ := rand.Int(rand.Reader, big.NewInt(100000))
	row, err := r.Queries.CreateTodo(ctx, db.CreateTodoParams{
		ID:     fmt.Sprintf("T%d", n),
		Text:   input.Text,
		UserID: input.UserID,
	})
	if err != nil {
		return nil, err
	}
	return &model.Todo{ID: row.ID, Text: row.Text, Done: row.Done, UserID: row.UserID}, nil
}

// クライアントが user を要求したときだけ呼ばれるフィールドリゾルバ
func (r *todoResolver) User(ctx context.Context, obj *model.Todo) (*model.User, error) {
	row, err := r.Queries.GetUser(ctx, obj.UserID)   // ← ここで本物の DB アクセス
	if err != nil {
		return nil, err
	}
	return &model.User{ID: row.ID, Name: row.Name}, nil
}
理屈 — なぜ「詰め替え」をするのか

sqlc は DB スキーマから db.Todo を、gqlgen は GraphQL スキーマから model.Todo を、別々に生成します。2 つの型は似ていても別物。だからリゾルバで小さなマッピング関数を挟むのが定石です。

列が NULL 可だと sqlc 側が pgtype.Text などになり、GraphQL の *string へ変換が要る点だけ注意。今回は全列 NOT NULL なので素直に string/bool で受け取れます。

（応用: gqlgen の autobind を sqlc の構造体に向ければ詰め替えを省ける場合もありますが、型のズレを吸収できる手書きマッピングのほうが初心者には安全です。）

チェックポイント

go run server.go で起動し、前回と同じ Playground のクエリを投げてみてください。見た目は前回と同じでも、裏では gqlgen のリゾルバが sqlc 経由で PostgreSQL を読み書きしています。これで「API 層 ⇄ データ層」が型安全につながりました。

10

理屈 — ここで N+1 が「本物」になる

前回 gqlgen の章で触れた N+1 問題を覚えていますか。連携した今、それが現実の刃になります。todo を 100 件返すクエリで user { name } を要求すると、Todo.user リゾルバが todo ごとに GetUser を呼ぶ ―― つまり 1 回の一覧取得 ＋ 100 回の user 取得＝ 101 回の SQL が飛びます。

理屈 — 解決の方向

答えは gqlgen の dataloader。「同じ実行サイクル内で発生した GetUser 呼び出しを溜めておき、まとめて 1 回の SELECT ... WHERE id = ANY($1) で取る」仕組みです。

ここでも連携が効きます。sqlc 側に「複数 ID をまとめて取る」クエリ（例: -- name: ListUsersByIDs :many で WHERE id = ANY($1)）を 1 本足し、dataloader からそれを呼ぶ。gqlgen が「いつ束ねるか」を、sqlc が「束ねた SQL」を担当する、きれいな分業です。これが実戦の到達点です。

最後に — 2 つの道具の関係を一言で

gqlgen は「外（API）」、sqlc は「内（DB）」。どちらも SQL / スキーマという宣言を書けば型安全な Go が生える、同じ性格の道具。あなたの仕事は、スキーマと SQL を書き、gqlgen の穴の中で sqlc のメソッドを呼ぶこと。これでカラムから API フィールドまで型が一直線につながります。
