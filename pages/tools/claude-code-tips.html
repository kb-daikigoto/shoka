Claude Code 超実用Tips

ターミナルで動く AI コーディング相棒 Claude Code を、「なんとなく」から「使いこなす」へ。モード切り替え・コンテキスト管理・CLAUDE.md・自作コマンド・サブエージェント・フック・MCP・ヘッドレス実行まで、実務でそのまま効く小技だけを一本にまとめました。

対象 使い始め〜実務 前提 Claude Code 導入済み 形式 Tips 集 読了 約 40 分

Part 1 — まず手に馴染ませる

01

この記事の歩き方

Claude Code は、エディタやチャット窓ではなくターミナルに常駐する開発エージェントです。ファイルを読み、コードを書き換え、テストを走らせ、git を操作する——その一連を会話で任せられます。ただ「お願いする」だけでも動きますが、少しの操作と設定を知っているだけで速さと安全性が段違いになります。この記事はそこに絞ったTips集です。

各節は独立して読めます。気になる番号へ飛んで、コマンドをそのまま試してください。記事中の表記は次のルールです。

  • /clear のように / で始まるものは Claude Code の中で打つスラッシュコマンド。
  • claude -p ... のように claude で始まるものは シェルで打つ CLI コマンド。
  • コードブロックの $ はシェル、› は Claude Code の入力欄を表します。
なぜ操作を覚えるのか

AIに任せる時代こそ「どこまで・何を・どう渡すか」を握る人が速い。モード・コンテキスト・権限の3つを自分でコントロールできれば、暴走を防ぎつつ手戻りを減らせます。本記事の背骨はこの3つです。

02

3つのモードを Shift+Tab で回す

一言でいうとShift+Tab を押すたびに「都度確認 → 編集を自動承認 → プラン（読むだけ）」と動作モードが切り替わる。

Claude Code には作業の「攻め／守り」を切り替えるモードがあります。入力欄で Shift+Tab を押すたびに循環し、いまどのモードかは入力欄の下に表示されます。

モード自動で通すこと向く場面
通常ファイル読み取り慎重に進めたい・重要な変更
編集を自動承認
(accept edits)		ファイル編集を都度確認なしで適用方針が固まり、まとめて回したい
プラン読み取りと調査のみ（書き込まない）まず調べさせて計画を立てたい
使い分けのコツ

「まずプランで全体像 → 通常で1つずつ承認 → 流れが見えたら自動承認で一気に」が定番。自動承認中でも危険なシェルコマンドなどは確認が入ります。怖いと感じたら Shift+Tab でいつでも通常へ戻せます。

03

プランモードで段取りしてから動く

一言でいうといきなり書かせず、まず「読む・調べる・計画する」だけやらせて、計画に納得してから実行に移す。

大きめの変更・触ったことのないコード・「壊したくない」場面では、まずプランモードに入れましょう。Claude はファイルを読み、読み取り専用のコマンドを走らせ、編集はせずに作業計画を提示します。あなたが承認して初めて手を動かし始めます。

入り方は Shift+Tab でプランへ循環するか、最初からプランで起動します。

$ claude --permission-mode plan       # プランモードで起動

計画が出たら、そのまま実行 / 1つずつ承認しながら実行 / もっと練り直すを選べます。計画文面に注文をつけて練り直させるのも有効です。

なぜ効くのか

AIの失敗の多くは「前提のズレ」。プランで方針を文字にして突き合わせると、実装が始まる前に勘違いを潰せます。手戻りはコードを書く前に消すのが一番安い。

Part 2 — コンテキストを制する

04

/clear と /compact でコンテキストを軽く保つ

一言でいうと話が長くなって精度が落ちてきたら、別件は /clear で全消去、続きものは /compact で要約圧縮。

Claude Code は会話の履歴（コンテキスト）を見て応答します。履歴が膨らむと、関係ない過去に引きずられて精度が落ち、コストも上がります。そこで2つの掃除コマンドを使い分けます。

/clear/compact
やること履歴を空にして仕切り直し履歴を要約に畳んで継続
残るものほぼ何も残らない（過去は別途再開可能）要点の要約が残る
使う場面別のタスクに移るとき同じ作業の続きを軽くしたいとき

/compact は要約の力点を指定できます。長い調査のあと、結論だけ残して雑談を捨てたいときに便利です。

› /compact データベース周りの決定事項を中心に残して
いちばんよくある改善

「タスクが終わったら /clear」を癖にするだけで体感がかなり変わります。コンテキストが一杯になると自動で圧縮（auto-compact）も走りますが、節目で自分から区切るほうが結果が安定します。

05

/context で「重さ」を見る

「なんだか遅い・反応が鈍い」と感じたら /context。いまコンテキストの何がトークンを食っているか——システムプロンプト、CLAUDE.md、会話履歴、ツール定義など——を内訳で見せてくれます。重い犯人が分かれば、/clear すべきか、CLAUDE.md を削るか、不要な MCP を切るか、打ち手が決まります。

合わせ技

会話量は /clear・/compact で、常駐分（CLAUDE.md・MCP・スキル）は設定で減らす。/context はどちらを削るべきかの地図です。

06

CLAUDE.md でプロジェクトの作法を覚えさせる

一言でいうと毎回言っている前提（使うコマンド・命名規則・やってほしくないこと）は CLAUDE.md に書いておけば、起動時に自動で読み込まれる。

「テストは npm test」「コミットメッセージは日本語」「main に直接 push しない」——こうした繰り返しの指示は、ファイルに書いておけば毎回伝えずに済みます。Claude Code はセッション開始時に 場所ごとの CLAUDE.md を自動で読み込みます。

場所役割共有
./CLAUDE.mdそのプロジェクトの作法git でチーム共有
./CLAUDE.local.md自分だけのローカルメモ.gitignore 推奨
~/.claude/CLAUDE.md全プロジェクト共通の好み自分の端末のみ
サブディレクトリの
CLAUDE.md		その配下を触るとき読み込むモノレポで便利

書きっぱなしにしない3つの道具

  • 編集する: /memory で現在読み込まれているメモリ系ファイルを一覧・編集できます。
  • 会話から足す: その場で「これを CLAUDE.md に覚えておいて」と頼めば追記してくれます。
  • 他ファイルを取り込む: @パス 記法で別の Markdown を読み込めます（初回は読み込み許可の確認が出ます）。
CLAUDE.md の例（抜粋）
# プロジェクトの約束ごと
- テスト: `npm test` / Lint: `npm run lint`
- コミットメッセージは日本語、要点を1行で
- main へ直接 push しない。必ずブランチを切る
- 仕様は @docs/spec.md を参照
太らせすぎ注意

CLAUDE.md は毎回コンテキストを消費します。「本当に毎回効かせたい数行」だけに絞り、詳細は @import で必要時に引くのがコツ。/context で重さを確認しましょう。

07

/init で叩き台を一発生成

CLAUDE.md をゼロから書くのは面倒——そんなときは /init。Claude がリポジトリを解析し、ビルド・テストの手順、ディレクトリ構成、規約などを盛り込んだ CLAUDE.md の下書きを作ってくれます。出てきた叩き台から不要部分を削り、約束ごとを足すのが一番速い始め方です。

最初の30秒

新しいリポジトリで Claude Code を使い始めたら、まず /init。これだけで以後の指示がぐっと減ります。

Part 3 — 入力と巻き戻し

08

@ファイル・!シェル・/コマンド

一言でいうと入力欄の先頭文字で意味が変わる。@＝ファイルを指す、!＝シェルを直接実行、/＝コマンド。

@ファイルを指す

打ち始めると候補が出る。@src/auth.ts を直して のように、探させずに正確に対象を渡せる。曖昧な説明より速く確実。

!シェルを実行

!git status のように、その場のコマンドを自分で叩いて結果を会話に取り込める。Claude に頼むまでもない確認に。

/コマンド

/ でコマンド・スキルのメニューが開く。組み込みも自作も、ここから呼ぶ。

普通の文

何も付けなければ通常の依頼。@ で対象を、! で材料を添えると精度が上がる。

地味に効く

@ でファイルを名指しするだけで、探索の往復が消えて速くなります。「どのファイルか」を人間が決められる場面は、決めてあげるのが効率的。

09

画像とスクショを渡す

エラー画面、UIのデザイン、ホワイトボードの図——言葉にしづらいものは画像を貼るのが速い。クリップボードの画像を入力欄に貼り付けると [Image #1] のように取り込まれ、「この画面の崩れを直して」と指示できます。スクリーンショットからのUI再現や、グラフの読み取りにも使えます。

貼り付けキーは環境差あり

多くの環境では通常のペーストで貼れますが、ターミナルによってはショートカットが違います（macOS の iTerm では Cmd+V など）。大きすぎる画像は縮小され細部が落ちることがあるので、見てほしい箇所を切り抜いて渡すと確実です。

10

Esc で止め、Esc Esc で巻き戻す

一言でいうと方向がズレたら Esc で即停止。やり直したい地点があれば Esc を2回押して巻き戻す。

「あ、違う」と思ったら待つ必要はありません。Esc で生成を中断し、すぐ指示し直せます。さらに、入力欄が空の状態で Esc を2回押すと巻き戻し（rewind）メニューが開き、過去の地点まで会話やコードの状態を戻せます（/rewind でも開けます）。実験的な変更を試して、ダメなら戻す、が気軽にできます。

戻せるのは Claude の編集

巻き戻しが対象にするのは主に Claude が行ったファイル編集と会話です。あなたが手で実行したシェル操作や外部の副作用までは戻りません。git のコミットという本物のセーフティネットと併用しましょう。

11

キーボードと複数行入力

知っておくと指が速くなる操作たち。

操作できること
Esc生成を中断する
Esc Esc（入力欄が空のとき）巻き戻しメニュー
Ctrl+R過去の入力を逆引き検索
Ctrl+C入力をクリア／（連打で）終了
\ + Enter改行して複数行入力（環境問わず使える）
↑ / ↓入力履歴をたどる

長い指示やコードを貼りたいときは、行末に \ を置いて Enter で改行を続けられます（多くのモダンなターミナルでは Shift+Enter も使えます）。Vim 派は /vim で入力欄を Vim キーバインドにできます。

Part 4 — 自分用に仕込む

12

自作スラッシュコマンドを作る

一言でいうとよく出す定型指示は .claude/commands/名前.md に書いておけば /名前 で呼び出せる。

「変更点をレビューして」「リリースノートを書いて」のような定番作業は、Markdown ファイルにして登録すると毎回打たずに済みます。配置場所は2つ。

  • .claude/commands/*.md — そのプロジェクト用（git で共有できる）
  • ~/.claude/commands/*.md — 自分の全プロジェクト用

本文には引数や、その場で実行するシェル、参照ファイルを埋め込めます。

.claude/commands/review.md
---
description: 変更点をレビューして指摘する
argument-hint: [観点（省略可）]
allowed-tools: Bash(git diff:*), Read
---
いまの変更をレビューしてください。観点: $ARGUMENTS

現在の差分:
!`git diff`

コーディング規約: @docs/style.md
記法意味
$ARGUMENTS呼び出し時に渡した引数すべて
$1 $2位置ごとの引数
!`コマンド`その場でシェル実行し結果を埋め込む
@パスファイルの内容を参照させる

フロントマターの allowed-tools で使えるツールを絞れば安全に、model で実行モデルも指定できます。サブディレクトリに置くと名前空間付き（例 /git:commit）になります。

最初の一本

「/review（差分レビュー）」と「/commit（規約どおりのコミットメッセージ生成）」あたりは費用対効果が高い定番。チームの .claude/commands/ に置けば全員の手癖が揃います。

13

サブエージェントに任せる

一言でいうと調査・レビューなどの下請け作業を、専用の人格と権限を持つ「サブエージェント」に切り出して並行で走らせる。

サブエージェントは、独立したコンテキスト・専用のシステムプロンプト・絞ったツール権限を持つ小さな担当者です。「コードを広く探す」「変更をレビューする」といった、本筋とは別腹の作業を任せると、メインの会話を汚さずに済みます。広い検索や複数ファイルの調査は、結論だけ受け取れるサブエージェント向きです。

セッション内では /agents で管理でき、自作するなら .claude/agents/*.md に定義します（説明・使えるツール・モデルなどをフロントマターで指定）。タスク内容に合えば Claude が自動で呼ぶこともあります。

なぜ分けるのか

1つの会話に何でも詰めると、関係ない情報がノイズになり精度が落ちる。「探す係」「直す係」を分けると、各々が自分の文脈に集中でき、結果がクリーンになります。

14

フックで「必ず」を自動化する

一言でいうと「編集したら必ずフォーマッタ」「危険コマンドはブロック」など、特定のタイミングで自動実行する仕掛けを settings.json に書ける。

フック（hooks）は、ツール実行の前後やプロンプト送信時などのライフサイクルの節目で、決めた処理を必ず走らせる仕組みです。「Claude にお願いする」のではなく、ハーネス側が機械的に実行するので、お願い忘れがありません。

イベント使い道の例
PostToolUse編集後に自動フォーマット／Lint
PreToolUse危険なコマンドを実行前にブロック
UserPromptSubmit送信時に文脈を自動で足す
Stop / SessionStart応答終了・開始時の通知や準備
.claude/settings.json（編集後に Prettier をかける例）
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{ "type": "command",
        "command": "npx prettier --write \"$CLAUDE_FILE_PATHS\"" }]
    }]
  }
}

設定は /hooks から確認・編集でき、ユーザー単位／プロジェクト単位で持てます。

フックは強力＝慎重に

フックはあなたの権限で任意のコマンドを実行します。内容を理解したものだけを登録してください。配布されたフック設定を鵜呑みにしないこと。

15

MCP で外部ツールにつなぐ

一言でいうとMCP サーバーを足すと、GitHub・課題管理・DB・デザインツールなど外部の道具を Claude が直接使えるようになる。

MCP（Model Context Protocol）は、Claude Code に外部サービスのツールを生やす共通規格です。つなぐと、それらのツールが一覧に現れ、Claude が必要に応じて呼びます。追加はシェルから行います。

$ claude mcp add --transport http github https://example.com/mcp/ \
      --header "Authorization: Bearer $TOKEN"
$ claude mcp list                  # つないでいるサーバー一覧

スコープ（適用範囲）は3種類。チームで共有したいならプロジェクトスコープ（リポジトリ直下の .mcp.json にコミット）が便利です。

スコープ範囲
localそのセッション／プロジェクトの自分だけ
project.mcp.json で git 共有（チーム全員）
user自分の全プロジェクト

セッション内では /mcp で接続状態の確認・有効化／無効化・認証（OAuth）ができます。

入れすぎ注意

MCP のツール定義もコンテキストを消費します。常用しないサーバーは /mcp で切る、必要なときだけ有効化する、で軽さを保ちましょう。

16

権限を整える

一言でいうと毎回出る許可ダイアログは、安全なものを allow に、危険なものを deny に登録して減らす。

確認プロンプトが多くて煩わしいときは、/permissions でルールを整えます。よく使う安全なコマンドを許可リストに入れれば聞かれなくなり、逆に危険な操作は拒否リストで明確に止められます。設定は settings.json に保存され、プロジェクト共有も可能です。

.claude/settings.json（権限ルールの例）
{
  "permissions": {
    "allow": ["Bash(npm run test:*)", "Read(./src/**)"],
    "deny":  ["Bash(rm -rf *)", "Read(./.env)"]
  }
}
--dangerously-skip-permissions は最終手段

すべての確認を飛ばす --dangerously-skip-permissions は強力ですが、名前のとおり危険です。使い捨てのコンテナや隔離環境に限り、本番に触れる端末では避けましょう。普段は allow/deny の積み上げで十分快適になります。

Part 5 — ターミナルを超えて

17

ヘッドレス実行と CI（claude -p）

一言でいうとclaude -p "指示" で対話せず一発実行。パイプや JSON 出力と組み合わせて、スクリプトや CI に組み込める。

Claude Code は対話だけでなく、非対話（ヘッドレス）でも使えます。コマンドに -p（print）を付けると、結果だけ返して終了します。ログを流し込んだり、出力を JSON で受け取って後段に渡したり——スクリプト部品として働きます。

$ claude -p "このプロジェクトを要約して"

$ cat build.log | claude -p "このエラーの原因は？"   # 標準入力を渡す

$ claude -p "認証処理を説明して" --output-format json \
      | jq -r '.result'                       # JSONで受けて加工
使いどころ

「PR の差分要約を自動コメント」「ログから障害サマリ生成」「定型レビューを CI で」など、人が毎回やっている定型を自動化するのに向きます。出力を --output-format json にすると後続処理が安定します。

18

セッションを再開する

うっかり閉じても、会話は失われていません。直前の続きは --continue、過去のどれかを選ぶなら --resume です。

$ claude --continue          # 直近のセッションを再開（-c でも可）
$ claude --resume            # 一覧から選んで再開

セッションの中からは /resume で別の会話に乗り換えられます。「昨日の作業の続き」を一瞬で呼び戻せるので、タスクを跨いでも文脈が切れません。

合わせ技

続きものは --continue で再開 → /compact で軽くしてから再開、が快適。別件なら新しいセッションで /clear 相当のまっさらから。

19

git worktree で並行作業

一言でいうと別ブランチを別ディレクトリに展開する git worktree を使えば、複数の Claude Code を衝突させずに同時に走らせられる。

「機能Aを作らせつつ、別の窓でバグBを直させる」——同じ作業ツリーで2つ動かすとファイルが競合します。そこで git worktree。ブランチごとに独立したディレクトリを作れるので、各ディレクトリで別々の Claude Code を回しても干渉しません。

$ git worktree add ../feat-auth feature/auth   # 別ブランチを別フォルダに
$ cd ../feat-auth && claude                     # そこで起動
なぜ速いか

レビュー待ちや長いテスト中に手が空く——その間に別タスクを並行で進められる。人間が複数の作業を同時に監督できるのが、ターミナル常駐エージェントの強みです。

さらに効く小技

/modelモデルを切り替え

難所は強いモデル、軽作業は速いモデル、と会話の途中で切り替えられる。コストと速度のバランス取りに。

/cost使った量を見る

そのセッションのトークン使用量・概算コストを表示。重い作業の前後で確認する癖を。

/config設定を整える

テーマ・エディタモード（Vim 等）・各種挙動をまとめて調整できる入口。

/add-dir外のフォルダも見せる

作業ツリーの外にある別リポジトリや設定ディレクトリを文脈に追加。横断作業に。

拡張思考じっくり考えさせる

難しい問題では拡張思考（extended thinking）をオンにすると、より時間をかけて推論する。トグルで切り替える。

/help困ったら一覧

使えるコマンドの一覧と説明。新機能の発見にも。まず叩いて損なし。

機能はアップデートで変わる

Claude Code は更新が速く、コマンド名・ショートカット・既定の挙動が変わることがあります。手元の最新は /help と公式ドキュメントで確認してください。本記事は2026年6月時点の挙動を簡略化してまとめたものです。

用語集

コンテキストClaude が今回答に使う「読んでいる材料」の総量。会話履歴・CLAUDE.md・ツール定義などを含む。膨らむと精度とコストに響く。
プランモード編集せず、調査と計画提示だけを行う読み取り専用モード。承認してから実行に移る。
CLAUDE.mdプロジェクトや個人の作法を書いておくメモリファイル。起動時に自動で読み込まれる。
スラッシュコマンド/ で始まる短縮コマンド。組み込みのほか、Markdown で自作できる。
サブエージェント独立した文脈・権限を持つ下請けの担当者。調査やレビューを切り出して任せる。
フック (hooks)ツール実行の前後などの節目に、決めた処理を機械的に走らせる仕組み。
MCPModel Context Protocol。外部サービスのツールを Claude に生やす共通規格。
ヘッドレス / -p対話せず一回で結果を返す実行方法。スクリプトや CI への組み込みに使う。
worktreegit の機能。別ブランチを別ディレクトリに展開し、並行作業を可能にする。
